动态工作流：治什么病、怎么治、还缺哪一环

底座是 opencode serve × MiniMax M2.5（192k）——一个在上下文管理、记忆、技能调用上都明显弱于 Claude Code / Codex 的引擎。外壳的使命是把弱模型抬到真能独立干活的水位。 动态工作流是这个使命里最重的一块承重墙：它治两大问题——智能体不清楚背景知识，和需要大规模探索的任务。 本版是教学向的展开：每一节都讲清原理（为什么会这样）→ 治法（怎么做）→ 失效模式（照做也会踩的坑）。
TL;DR
  1. 两大问题同根不同枝：知识为零逼着每次任务都去探索；探索面超过单会话容量又让探索本身成为弱模型最容易崩的阶段。
  2. 治法的本质：把「长程连贯性」从模型收编到壳层——小批 replan 管计划、fan-out 子代理管隔离、grounding 管真伪、交接协议管传递。上下文成本从 O(探索面) 摊薄为 O(结论数)。
  3. 还缺一环：沉淀回路。探索结果若随 session 蒸发，每次任务都在全额交探索税——动态工作流会退化成一台西西弗斯机器。
  4. 补上回路后经济学反转：探索税一次性缴清，此后带地图进场。弱模型 + 饱和领域知识，在根因高度重复的内网系统上能赢过强模型 + 零知识——而且这份资产不随模型换代贬值。
§ 00

导读

读完这篇你应该能回答五个问题：

  1. 为什么 192k 上下文用不满——衰减的机理是什么，红线该划在哪（§01）；
  2. 为什么「找不到根因」多半不是模型笨，而是知识饥饿——以及知识分几类、各自怎么腐烂（§02）；
  3. 三种编排模式（ReAct / Plan-and-Execute / 小批 replan）各自的适用边界，为什么弱模型 + 探索型任务只有第三种能活（§04）；
  4. 一次动态编排从进场到交付，每一步谁烧 token、谁持有状态、数据长什么样（§04 泳道图）；
  5. 知识怎么沉淀、怎么注入、怎么防腐，账怎么算（§05–§06）。

阅读路径建议：赶时间看 §01 的公理框 + §04 泳道图 + §08 清单；系统学习按顺序读完。

§ 01

理论底座：上下文为什么会塌，弱模型靠什么翻盘

1.1衰减的机理：三个叠加效应

「长会话后期胡说」不是玄学，是三个机制叠加的必然结果：

  • 注意力稀释 —— 注意力是零和预算：序列越长，softmax 摊到每个 token 上的注意力越薄。关键约定和海量 grep 输出争抢同一份预算，输的总是前者。
  • 位置偏置（Lost in the Middle） —— 模型对上下文的记忆呈 U 形：开头和结尾记得牢，中段最容易丢。你在第 20 轮塞进的关键事实，到第 60 轮正好落在最容易被无视的中部。（Liu et al., 2023《Lost in the Middle》；后续 NoLiMa 等评测进一步表明：needle 一旦不能靠字面匹配找到，长上下文性能塌得更早。）
  • 干扰累积（context rot） —— 无关的工具输出不只是占地方，它是主动噪声：为模型的注意力检索提供大量错误目标。旧的 grep 结果会被「检索」出来拼进当前推理——这就是「自信地编造前文」的来源。

另外一层：许多模型的标称长度是靠位置编码插值/外推撑出来的，标称窗口的尾段本来就是质量次品区。 M2.5 的 192k 大概率同样如此。三个效应对弱模型全部更凶——拐点更靠前、塌得更陡。

1.2有效上下文：怎么估、怎么用

  • 怎么测 —— 拿一个固定任务，垫入不同量的无关前缀（真实的 grep 输出最好，比随机文本更接近实战噪声），看成功率从哪里开始跳水。那个拐点就是这台模型在你的负载下的有效上下文。
  • 经验值 —— 中档模型通常落在标称的 40–60%。对 192k 的 M2.5，按 55% 算约 106k。
  • 工程用法 —— 把拐点当红线写进壳层：水位过线就触发 session 旋转（蒸馏交接单 → 新会话）。关键转变是心态上的：192k 不是预算，106k 才是；而且越接近红线，每个 token 的边际质量越差。

设计公理

外层能补的是「流程智能」，补不了「单步智能」。

推论一：凡是要求模型保持长程一致性的设计，都是 bug 温床——连贯性必须由壳层持有。
推论二：凡是能用确定性代码接管的步骤，都不该烧 token——模型只出现在真需要判断的地方。

历史印证：GPT-3.5 时代的 agent 框架（AutoGPT、早期 LangChain）全靠重脚手架把弱模型抬到能用；模型变强后脚手架变薄。 今天 Claude Code 的能力跃迁主要来自模型升级，harness 是乘数。但当模型固定时，harness 是唯一的杠杆——这正是本项目的处境。

1.4弱模型翻盘的数学：生成-验证差

很多任务验证比生成容易一个复杂度等级：写对一段代码难，跑测试看红绿容易；找出正确锚点难，核验锚点存在容易。 这个不对称叫生成-验证差（generation-verification gap），它是弱模型翻盘的全部数学基础：

设弱模型单次做对的概率是 p，独立采样 n 次、由硬验证器挑出对的那次，至少一次成功的概率是 1 − (1 − p)ⁿ。p = 0.3 时，n = 8 → 94%。 也就是说：只要判据够硬、token 够便宜，0.3 的模型能交出 0.94 的活。

  • 你恰好两样都有 —— 私有化推理 token 近乎免费（API 计价下没人敢这么挥霍）；判据体系已经建了一半：回放断言、V8 覆盖率、文件:行 锚点核验、契约对账。
  • 唯一的坑：软判据 —— 让模型自评「你觉得对吗」不算验证器，弱模型会骗过它自己（Goodhart 定律：把测量当目标，测量就失效）。判据必须是壳层能机械执行的：文件存在吗、断言过了吗、覆盖率击中了吗。
§ 02 · 病

问题一：背景知识为零

2.1现状与症状链

M2.5 冷启动时拿到的系统背景知识几乎为零。知识来源只有三种——自己 grep 探索、人工现场敲给它、或者干脆没有。 每一个新会话，它都是第一天入职的空降新人。由此产生一条完整的症状链：

  • 只能字面读码 → 不知道模块沿革、历史坑位，诊断停在表面症状，修补是打补丁不是除根；
  • 不知道隐性约定 → 改出风格违和的代码，踩三年前就踩过的坑；
  • 不知道「谁跟谁共享一张表」 → 影响面判断漏项，改 A 崩 B；
  • 上下文不传承 → 同一需求，每个接手的会话都要重新打捞一遍来龙去脉。

2.2知识分类学：四种类型，四种腐烂速度

「背景知识」不是一团浆糊，它可以精确分类——分类决定了每种知识从哪蒸馏、何时注入、多快腐烂。 借用认知科学的记忆分型（语义记忆=事实、程序记忆=怎么做、情景记忆=发生过什么）：

2.3关键判断：根因盲是先验缺失

关键判断

「找不到根因、只看到表面」多半是知识问题伪装成推理问题。

用贝叶斯的话说：高级工程师快，是因为他的先验分布对——看到「授信页白屏」，他脑内的候选根因已经按历史频率排好序，前三个命中率 80%。 空降新人（和 M2.5）的先验是均匀分布，只能地毯式排查，排查又受限于上下文容量，于是停在第一个「说得通」的表面解释上。 修法因此明确：不是换更聪明的模型，是把先验（=知识资产）递到模型手里。

内网信贷场景把问题和解药同时放大：领域知识密度高、文档长期缺位、tribal knowledge 全在老员工脑子里——所以病重； 但内网系统的根因分布高度重复，同类事故翻来覆去发生——所以知识复利的回报率也比任何开源项目都高。

§ 03 · 病

问题二：需要大规模探索的任务

3.1一个真实任务的 token 账本

定义：探索面超过单会话有效上下文的任务。拿一个双仓团队的日常任务——「授信额度字段贯穿改，先定位影响面」——把单会话直探的账逐步记下来：

步骤动作典型产出累计水位
1前端仓 grep 字段名（含驼峰/下划线变体）≈3k3k
2读 6 个前端命中文件（组件 + api 层）≈30k33k
3后端仓 grep 方法/字段引用≈5k38k
4读 8 个后端文件（service + mapper + dto）≈52k90k
5git log / blame 3 处关键改动≈14k104k
6读 DB 映射与 SQL 片段 ← 越过 106k 红线（55%）≈16k120k
7汇总分析——恰恰是最需要「全部细节同时在场」的一步—已在劣化区作业

注意第 7 步的残酷之处：探索的终点才是对上下文质量要求最高的时刻（要把 120k 的素材拼成一张影响面地图）， 而这个时刻恰好落在质量最差的水位上。这不是偶然，是单会话直探的结构性缺陷。

3.2恶性耦合与随机游走

  • 恶性耦合 —— 探索期是工具输出涌入最猛的阶段，病根一的三个症状全在此时集中发作。越需要探索的任务，越先把弱模型搞疯。
  • 随机游走 —— 没有地图的探索，模型 grep 到哪算哪：不知道该先看哪、哪里有坑、哪个方向是死路。探索路径长度直接决定 token 消耗——先验越差，路径越长，污染越重，结论越糊。探索深度和会话质量成反比，这是问题一和问题二咬合在一起的死结。
§ 04 · 治

治法：动态工作流

一句话概括本质：把「长程连贯性」这个弱模型最不擅长的东西，从模型收编到壳层；模型只做单步判断。 计划状态、探索边界、结论真伪、阶段传递——全部由壳层持有和裁决。

4.1先看谱系：三种编排模式，为什么只有一种能活

  • 水位随步数线性涨，红线必破
  • 目标存在上下文里，必被稀释

  • 结构清楚，可并行执行

  • 探索型任务计划必过时——计划赶不上发现

  • 计划质量要求降到「单批」粒度
  • 新发现即时反哺下一批
  • 编排开销 + 需要壳层账本（值得）

选型逻辑一句话：任务的不确定性决定计划的保质期。探索型任务里每一批的发现都会改写「接下来该干什么」， 所以计划保质期只有一批——那就只做一批的计划。这同时把「长计划生成」这个弱模型做不好的活，替换成了「短计划生成 × N 次」这个它做得好的活。

4.2解剖一次编排：泳道图

下图是一次完整编排的时序。看两条主线：谁烧 token（描边=模型会话，实心=确定性代码），状态在哪（永远在壳层账本，从不在模型上下文里）。

壳层编排器确定性代码 · 0 token
Planner模型 · 常驻低水位
探索子代理 ×N模型 · 一次性
Grounding 校验器确定性代码 · 0 token
1
接任务 → 难度分诊。简单任务直答结束；复杂任务开编排账本（总目标 · 已知事实 · 批历史 · 水位表）→ Planner：目标卡 + 知识索引（§05）
2
产出下一小批：3–5 个窄问题，每题带预算与停止条件→ 壳层：批计划 JSON{问题, 范围, 预算, 停止条件}
3
机械校验批计划格式（JSON 修复在这层做，不问模型）；fan-out 启动 N 个子代理→ 子代理 ×N：各发一张窄任务卡
4
并行探索：grep / 读码 / git blame / CDP……各烧各的上下文（40–80k），互相不可见、也看不到主线程
5
各自收敛，输出蒸馏结论（≤500 token），随后整个子上下文被丢弃→ 校验器：{事实, 文件:行锚点, 置信度}
6
机械核验：锚点文件/行存在？行内容与结论相关（字符串级抽查）？✗ 无锚点或核验失败 → 打回该子代理重采样 ×n（token 便宜）
7
结论入账本；两项确定性检查：空转判死（本批有无新增事实？无进展 → 杀分支）· 水位检查（Planner 会话接近红线 → 旋转）
8
收敛判定：账本里的未决问题清空了吗？未收敛 → 回步骤 2，把账本摘要（非历史转录）喂给 Planner ↺
9
汇聚：账本蒸馏成结构化交接单（事实 / 决策 / 未决 / 锚点）→ 交付给人 · 传给下一阶段 · 沉淀入库（§05）
图例：实心壳层确定性代码，不烧 token 描边模型会话，常驻但控制在低水位 虚线描边一次性模型会话，用完即弃。 注意步骤 8 的细节：Planner 每轮拿到的是账本摘要，不是自己的历史转录——这就是它水位恒低的原因。

① Agent 自主分诊 编排开销不滥用 模型判断而非写死规则；简单任务直答 错杀（简单任务被编排 → 慢）/ 漏诊（复杂任务单跑 → 崩）。解法：允许中途升级——单会话跑到水位红线自动转编排
② 小批 replan 长计划质量差 + 计划过时 每轮只出 3–5 个窄问题；账本喂摘要 批太大 → 退化成 Plan-and-Execute；批太小 → 编排开销占比过高。批大小是要调的旋钮
③ fan-out 隔离 探索污染主线程 子代理独立上下文，只回传 ≤500 token 结论 任务卡太宽 → 子代理内部自己变成大规模探索（递归病）。任务卡必须窄到「一个问题一个答案」
④ grounding 弱模型结论不可信 结论必须带 文件:行；壳层机械核验 锚点真实但不相关（模型报了个存在的行敷衍了事）。解法：核验升级为「行内容与结论关键词匹配」的廉价二审
⑤ 交接 + 判死 阶段间信息衰减 + 僵尸任务 结构化交接单强制字段；判死看进展不看时长 交接单模板太松 → 关键事实漏传、下游返工。字段必须强制：事实 / 决策 / 未决 / 锚点——「未决」最容易被省略也最致命

§ 05 · 缺

缺的一环：沉淀回路

西西弗斯问题

只要探索结果随 session 蒸发，每个任务都在全额交探索税。

动态工作流让每次探索死得更体面——有组织、不崩会话、结论可信；但第二天同类任务进场，一切从零再来。 巨石推上山顶，滚回原点。缺的不是新子系统，是给现有工作流加一个收尾动作：探索完，写回去。

5.2冷启动自举：把「读懂这套系统」交给动态工作流自己

知识库不必手工建——「读懂这套信贷系统」本身就是一个标准的大规模探索任务，交给动态工作流跑，还顺便当它的压力测试。 《信贷系统 Wiki 生成手册》就是现成的自举 prompt。分五个阶段，每阶段的交接单喂给下一阶段：

  1. 骨架 —— 目录树、构建配置、入口文件、路由表 → 产出「系统长什么样」的 10 条左右架构条目；
  2. 模块职责 —— 每个顶层模块派一个子代理，回答「这个模块管什么、暴露什么、依赖谁」；
  3. 关键链路 —— 登录 / 授信 / 放款等 3–5 条核心业务流端到端走读（前端入口 → 接口 → service → 表）；
  4. 数据层 —— 表 ↔ 服务映射，重点标注「多服务共享一张表」这类高危结构（坑位候选）；
  5. 隐性知识 —— git log 高频改动点 + blame 热点 = 历史坑位候选；提交信息里的「fix/回滚/紧急」聚类。

预期产出：30–80 条条目。数量刻意控制——知识库的价值密度比条数重要，垃圾条目会污染注入。

5.3增量蒸馏：什么值得存？三问判据

工作流收尾固定加一步蒸馏。判什么该入库，三问全过才收：

  1. 这是关于系统还是关于本次任务？ ——「t_quota 被两个服务共享」是系统知识；「本次改了 3 个文件」是任务日志，进台账不进知识库；
  2. 三个月后还成立吗？ —— 结构性事实（表结构、约定）入库；临时状态（「XX 分支还没合」）不入；
  3. 下次谁会用到、在什么场景？ —— 写不出注入场景的条目就是没用的条目，宁缺毋滥。

两个补充规则：事实与猜测分级——探索中「疑似竞态但未证实」也值得存，但必须标 confidence: suspected，下次命中时优先安排核实； 验证 PASS 的修复自动蒸成事故指纹——旗舰⑤（事故指纹库）就在这一步孵化，不需要单独立项。 交接协议练出来的蒸馏能力直接复用：交接单是传给下一棒，知识条目是传给三个月后——同一块肌肉，两个持久化尺度。

5.4索引制注入：两级加载，三种命中

session 起步只注入索引——每条一行（约 30 token），50 条也才 1.5k；命中相关的才展开全文（每条 200–400 token）。 这是「渐进披露」：把背景知识递到模型手里，又不吃掉宝贵的有效水位。命中方式按可靠性排序：

  • 确定性匹配（首选） —— 任务卡里的文件路径 / 模块名 / 表名 / 接口名与条目锚点做字符串匹配。零 token、可解释、弱模型免疫；
  • Agent 自选（兜底） —— Planner 起步时看一眼索引，点名要展开哪几条。花少量 token，覆盖确定性匹配漏掉的语义关联；
  • Embedding 相似（后补） —— 语义检索需要向量库，收益在条目上千后才明显。P2 之后再考虑，别为 50 条知识先建一套检索系统。

--- 知识条目 · knowledge/limit-service-shares-quota-table.md ---
name: limit-service-shares-quota-table

5.6防腐管道：注入前的四步廉价校验

必须防的坑 · 知识会腐烂

代码天天在变，知识条目不校验就会过期。过时的地图比没有地图更危险——弱模型拿着它会错得更自信。 防腐必须是确定性代码（不烧 token），在注入前与夜间巡检两个时机跑：

C1锚点文件还存在吗？fs.existsSync —— 失败直接判红
C2锚点符号还存在吗？grep 类名/方法名 —— 失败判红
C3行漂移了吗？比对锚点行内容 hash；漂移则在附近重定位（grep 符号），成功就回写新行号，失败判红
C4verified 之后相关文件改了几次？git log --since —— churn 超阈值判黄（内容可能已变，锚点还没变）
绿
直接注入
黄
注入但附带「可能过时」标注，命中时提示核实
红

6.3反转发生在三个维度

  • 成本 —— 探索税从「每次全额」变为「一次性缴清」。探索步骤从「摸清这是什么」降级为「核实地图还准不准」，token 消耗差一个量级。
  • 质量 —— 带地图进场等于把先验分布递给了模型（§2.3）：知道该先看哪、哪里有坑、谁跟谁共享状态。根因命中率的提升幅度远大于任何提示词技巧。
  • 会话稳定 —— 探索变短 → 水位变低 → 病根一的三个症状同步下降。三个病根，一条回路牵动其二。
战略论断

在根因高度重复的内网系统上，弱模型 + 饱和领域知识 > 强模型 + 零知识。

这也是本产品面对 Claude Code / Codex 的真正护城河：模型会换代，提示词会过时， 但知识资产随任务复利增值，且不随模型换代贬值——模型越强，同一份地图只会被用得越好。

防腐校验的伪代码（确定性，注入前跑）：

function verifyEntry(entry) {
  for (const a of entry.anchors) {
    if (!fs.existsSync(a.file))        return RED;   // C1 文件没了
    if (!grepSymbol(a.file, a.symbol)) return RED;   // C2 符号没了
    if (hash(readLine(a.file, a.line)) !== a.lineHash) {  // C3 行漂移
      const hit = relocateNear(a.file, a.symbol);
      if (!hit) return RED;
      a.line = hit.line; a.lineHash = hash(hit.text);       // 回写新锚点
    }
  }
  const churn = gitChurnSince(entry.verified, entry.anchors);
  return churn > THRESHOLD ? YELLOW : GREEN;              // C4 改动太多降级
}

7.2待决设计决策：知识放哪

维度跟着目标仓库走（docs/）壳层私有存储
团队共享天然共享，随 clone 分发纯个人资产
版本同步与代码同 commit 演进，锚点天然对齐需自行维护 verified 元数据
侵入性写入他人仓库，需团队认可零侵入
实现成本中（提交流程、冲突处理）低
路线图对齐对齐 SDLC 团队资产定位对齐个人智能体定位

可行的折中：壳层起步，格式可平移——纯 markdown + 相对路径锚点，不绑任何壳层私有结构； P2 阶段再依据团队接受度决定是否落进仓库。决策可以推迟，格式兼容性必须现在锁定。

旋钮原则

壳层里所有「弱模型工伤补偿」——多路采样、对抗评审、凶狠的水位阈值、确定性技能路由——都要做成可拧松的旋钮而非焊死的结构： M2.5 换代变强后一档一档退掉（§4.6 的对照表就是退档路线图：往 Claude Code 那一列退）。 而知识资产、验证回路、确定性快路径是永久资产，模型越强，它们的杠杆越长。

§ 08 · 律

方法论提炼：可迁移的十条

把整篇收拢成一张 checklist——它不只适用于 BocomHermes，适用于任何「固定弱模型 + 外壳增强」的智能体：

量出有效上下文，把红线写进壳层。标称不是预算；拐点（通常 40–60%）才是。越线自动旋转。
长程一致性一律收编到壳层账本。模型只见「当前批 + 账本摘要」，永不见历史转录。
探索烧在一次性子上下文里。主线程只收蒸馏结论——把 O(探索面) 摊薄为 O(结论数)。
一切结论必须可机械核验。锚点 / 断言 / 覆盖率；无判据的输出按未完成处理。软判据（模型自评）不算数。
弱模型多采样 + 硬验证 ≈ 强模型单次。1−(1−p)ⁿ；token 便宜时这笔账永远划算。
每个工作流以蒸馏收尾。三问判据（系统级？三月仍真？有注入场景？）；知识不落盘等于探索没发生。
注入走索引制两级加载。确定性命中优先，Agent 自选兜底，embedding 等条目上千再说。
知识必须带锚点与 verified 元数据。注入前跑防腐管道（C1–C4），腐烂即隔离——过时地图比没地图危险。
判死看进展不看时长，打回看证据不看语气。慢任务不误杀，嘴硬的幻觉不放行。
所有弱模型补偿做成旋钮。模型升级能一档档退掉；知识资产、验证回路、确定性快路径是永久资产。
