# 信贷管理系统 · 「需求分析 → 改动点 → 需求问题 → 详细设计文档」完备解决方案

> 研发流程第一道大关(需求 → 设计)的端到端方案,针对**信贷管理系统**,重点攻破**专业业务知识 / 数据库 / 外系统调用**三块难点。
> 
> 产出方式:8 个 facet 并行专家级设计 → 信贷业务完备性 / 技术可行性 / 防幻觉 三道对抗评审 → 整合(已吸收评审的 blocker/major)。
> 本文档是**方案与落地蓝图**,后续按路线分期实现。

---

## 一、方案总览

这道关被攻破的核心思路:把研发第一道关做成一条"有状态机(INTAKE→APPROVED)、有 REQ-ID 贯穿主键、有四道人审闸口(G0-G3)、每条结论挂可机读证据(Claim Ledger)"的流水线;但两轮对抗评审把它从"七大旗舰一次性上线"砸成了诚实的现实——(1)信贷完备性不是叙事而是清单:必须先建"领域大脑"(域对象/流程枚举/口径词典/制度库),且枚举里缺了贷后全场景(展期重组/核销/担保生命周期/额度冻结/不良处置)、跨产品差异(对公/零售/信用卡)、制度文件类规则(不在代码里),否则 Coverage Matrix 会"自信地报告 covered"却系统性漏掉半个信贷业务;(2)底座契合度被高估一个数量级:git_grep/行级 scan_impact/SQL-PROC 解析/向量库/契约对账引擎/docx 全是要新建的承重墙,真正现成的只有 orchestrate+onBeforeBatch、git blame/log/show、outbox、mail。因此整合方案的骨架是:用"人工业务定性兜底机器漏扫"替代"纯机器 flag"、把 reg(制度)证据提升为一等公民、按产品线先分诊、把领域大脑降级为人工维护的 glossary.json 起步、并显式分期(v1 只交付状态机+需求问题生成+claim 账本这个不依赖未建底座的最小闭环,影响雷达/DB血缘/契约对账作为 v2/v3 的半自动强人审降级版)。

## 二、端到端管线(状态机 + 四道人审闸口)

REQ-ID 贯穿主键;每条结论挂可机读证据(Claim Ledger);状态 INTAKE → STRUCTURED → IMPACTED → CLARIFYING → DESIGNED → IN_REVIEW → APPROVED。

### S0 来源分诊与立卡 (INTAKE → G0)

- **输入**:邮件(mail_get_full 原文+messageId+附件文本)/工单/本地文档;以及两类信贷特殊来源:监管整改(带时限+条款)、生产事故/数据修复(改存量数据而非代码)
- **输出**:需求卡(Intake Card):REQ-ID(REQ-{产品线}-{YYYYMM}-{seq})+ 标题+一句话诉求+提出方+【产品线分诊:对公/零售/小微/信用卡】+ 需求类型(新功能/缺陷/监管整改/参数调整/报表/数据修复)+ 原文锚点(messageId 钉死);写入 req-ledger.json
- **怎么做**:analyst 子 Agent 抽要素。关键修正(吸收评审 blocker):产品线分诊是第一维度——同一'额度/审批/计息'在对公与信用卡是完全不同的模型,必须先定线再映射域对象,否则后续 lens 大面积错配。监管整改/数据修复走特殊子流程标记(前者强制挂制度条款+合规人参与,后者走数据订正审批+账务平衡校验)。
- **人审闸口**:G0 立卡确认(轻闸口):人确认'这是一条需求、归我处理、产品线判对、REQ-ID 成立'。低风险可 auto,但监管整改/数据修复类不可 auto。

### S1 需求结构化与现状采集 (STRUCTURED)

- **输入**:需求原文 + REQ-ID 上下文 + 产品线 + 领域大脑(v1=人工维护 glossary.json/码值表/域对象枚举,v2=ontology+RAG)
- **输出**:StructuredReq 对象(域对象/触发流程节点/触账?/调外系统?/影响存量?/影响跑批?/监管口径/验收点/约束)+ 现状证据 EvidenceNote(code/db/external/reg 四类,confirmed/maybe)
- **怎么做**:analyst+credit_domain_lens 把自然语言映射到标准枚举;同时 fan-out 只读 worker 采集代码/DB/外系统/制度现状。关键修正:对'新产品/新功能'(代码现状几乎全空)不退化成'全转问题轰炸'——此时以制度库(reg)和领域大脑为主证据源,代码空缺标注'待实现'而非'已覆盖';reg 类证据与 code 同等地位,有独立条款级索引+生效版本。无独立闸口,结果并入 G1 人审。
- **人审闸口**:无(自动流转);结构化结果在 G1 一并人审

### S2 改动点与影响分析 (IMPACTED → G1)

- **输入**:StructuredReq + 双仓代码(前端/后端各 ensureServe)+ DB 离线索引(DDL/码表;v2 含 PROC)
- **输出**:ChangeImpactList:[{cpId,repo,file,line,symbol,kind,risk:P0-P3,confidence,owner(blame),flags{触账,调外系统,影响存量,影响跑批,跨双仓,审批流,担保,监管报送}}] + 影响半径矩阵 + DBImpactReport + 盲区清单(PROC/动态SQL/规则引擎配置)
- **怎么做**:v1:单仓+跨双仓 git grep 文件级命中 + git_blame 责任人(用现成能力,不假装有行级 BFS)。v2:新建行级 scan_impact(带 kind 过滤/分层/片段)+ 有界 BFS 闭包 + DB 字段反查。账务/担保/审批流/跑批/分片键命中=P0。关键修正(吸收评审):账务一致性不窄化为'借贷平衡'——额度勾稽/总分核对/表内外核对这类'不记账但破坏一致性'的场景单列;PROC 业务逻辑机器读不可靠,一律进盲区清单逼人工,不自动判定。
- **人审闸口**:G1 范围与影响确认(最贵闸口):人勾误报/补漏报/定风险分级。强制不可 auto 条件=触账/外系统/存量/监管/担保。新增硬兜底(吸收评审 major):人必须独立回答三个业务定性问题'是否触及账务/监管/客户权益',而非只审机器扫出的清单——防止机器漏扫则人审跟着漏。

### S3 需求问题生成 (CLARIFYING → G2)

- **输入**:StructuredReq + G1 确认后的改动点 + 现状证据(含 maybe/缺口)
- **输出**:ClarificationSet:[{qId,category(歧义/缺口/冲突/边界/异常分支/数据口径/合规/性能/兼容),blockingScore(P0-P3),question,why(挂cpId/证据),evidenceRefs,askWho,askDept,status}] 分必答区/备注区
- **怎么做**:九类检测器 lens 在'已定位证据'上生成问题(无证据丢弃,压琐碎);规则+模型混合评分。关键修正(吸收评审 minor):澄清对端是分散的(口径问财务会计/分类问风险部/监管问合规/账务问财务)——askWho 升级为带部门路由,复杂口径标'需 A+B 部门会签';单 thread 邮件模型不足以表达会签,需多对端拆分 + 不因'收到一封回复'就置已澄清。无源/UNKNOWN/UNCOVERED 自动熔成问题卡。
- **人审闸口**:G2 问题外发确认:人审后经 outbox 15 秒延迟闸门 mail_reply 挂回原 thread(可改可撤)。数据口径/合规类高风险问题成稿前强制人审一眼。

### S4 详细设计文档生成 (DESIGNED → IN_REVIEW)

- **输入**:StructuredReq + 改动点 + 澄清答复/假设 + 领域大脑 + Claim Ledger
- **输出**:ddd.json(机器稿,每 claim 带 layer:fact|suggest|todo + evidence + reqIds)→ 渲染 Markdown(源)+ Word/Confluence(人读);traceability.matrix;open-issues.list
- **怎么做**:按 16 节信贷模板逐节 fan-out(architect 主笔 + 各域 reviewer),分节落盘再组装(避免一次 reduce 超 128k token——评审指出的真实约束)。三层标签:fact 必挂证据、suggest 需人采纳、todo 必挂 Q-id。关键修正:文件:行 锁 commit hash 而非浮动行号;account/计息/状态机规则只能 fact 引现有实现或 todo 挂业务,禁止 Agent 自编账务规则;未决项不清零则文档头水印'草稿/待定 N 项'阻断定稿。
- **人审闸口**:闸2(未决清单):onBeforeBatch 把 [待确认] 推给人逐条判定;闸3(争议设计):DB变更/状态机/账务冲正/外系统降级策略的 suggest 必须人采纳才能转 fact

### S5 评审确认与签发 (IN_REVIEW → APPROVED → G3)

- **输入**:设计文档 + 全链路追溯 + Completeness Report(覆盖率/置信度分布/合规检查单)
- **输出**:评审记录(必改/建议/可忽略分级)+ 定版设计文档 + 闸口落痕(by/at/decision/comment 挂 REQ-ID)
- **怎么做**:多 Agent 对抗评审(security/账务一致/biz/arch)先做机器自审产出评审材料包,再交人逐条批注。ddd_validate(纯 JS 不走 LLM)校验:事实类 claim 缺证→降级、REQ 未覆盖→TODO、mermaid 语法、commit hash 反查行是否漂移。
- **人审闸口**:G3 设计评审签发(总闸):账务/计息/还款计划/审批权限/监管报送类改动无条件进人审,即便机器置信度 C5。签发即 APPROVED、定版、回写工单/邮件(outbox)、REQ-ID 进入下游编码。

## 三、三大难点专项设计

### 专业业务知识(领域大脑)——评审点名的真正承重墙

**设计**

分三步落地,不一次性建重型 RAG。【v1 起步:人工维护的结构化知识,无向量】glossary.json(口径词典:敞口含不含表外/余额时点vs平均/利率ACT360vsACT365/逾期自然日vs营业日+宽限期/本息费冲销顺序)+ domain-objects.json(域对象与流程枚举)+ code-tables.json(五级分类01-05/还款方式/担保类型/合同放款状态机)。这些是纯 JSON、人工沉淀、可 grep,不依赖向量库/embedding(底座零向量能力,评审已证)。【枚举必须补全——评审 blocker】流程枚举不能只有'申请受理/审批/放款/还款/计息/分类'这个高频子集,必须补:展期/重组/借新还旧/置换、核销('账销案存'+冲减拨备+表内转表外)、不良处置/以物抵债/债权转让、额度冻结/解冻/调减/失效/恢复、合同要素变更(LPR换锚/还款方式变更/担保人变更)。【产品线维度——评审 blocker】域对象/状态机/口径按对公/零售/小微/信用卡分线维护:信用卡的循环授信/账单日还款日/最低还款/分期手续费/免息期在对公模型里根本不存在;零售/小微大量是自动审批/评分卡而非人工多级审批。【reg 制度库提升为一等公民——评审 blocker】信贷大量规则(展期次数/核销条件/分类迁徙矩阵/集中度限额/利率定价权限)不在代码里而在制度文件,必须像代码一样建条款级索引+生效版本管理;否则'口径以哪个监管文为准'这个硬问题模型只能现编,比不答更危险。【v2/v3 增强】ontology.json 本体图(节点带 grounding evidence,无凭据标 unverified)+ 本地 RAG(依赖内网 embedding 端,中文术语召回差需 glossary 同义词+符号倒排双路兜底)。

**信贷特有考量**

领域库覆盖范围必须有验收标准(评审指出方案缺这个):每个产品线 × 每个流程节点 × 是否有制度依据,做成 onboarding checklist 由信贷业务专家逐项确认置 high 可用;业务专家在 G1/G3 闸口有固定席位(不是可选)。不良/贷后整条线(展期重组/核销/担保代偿追偿/不良批转)与担保独立生命周期(抵押登记解押/价值重估/担保变更释放)必须进枚举,否则一条'展期'需求只会落到'还款计划/计息'漏掉合同关闭/风险重分类/担保重估三条主线。客户维度规则(关联方合并授信/黑名单准入/多头授信/集团集中度)纳入授信测算前置校验。消费者权益保护(年化利率明示/息费透明/催收合规/个人信息授权链)作为独立合规维度,不并入数据合规。

### 数据库维度——DB 变更影响(诚实砍到可自动化的子集)

**设计**

【v1 可落地子集——评审 major:PROC 自动血缘投入产出不对称,砍掉】只做:(a)DDL/码表字段级反查(列名/表名 git grep 命中 Mapper.xml/Entity/动态SQL/报表SQL);(b)大表在线 DDL 锁风险提示(ADD COLUMN 带默认值全表重写、加索引需 gh-ost/pt-osc/DBMS_REDEFINITION、改类型锁表),按表量级给锁时长'量级估计'并明标'需 DBA 用生产统计复核';(c)分片键判定(改 shardingKey=灾难标红默认拦截、新查询不带分片键=全分片扫描);(d)灰区清单(动态SQL/EXECUTE IMMEDIATE/PROC体/拼列名)强制人工确认,绝不静默判'无影响'。【摄取通道——评审 major:来源悬空】DB 索引走离线快照(不连生产):数据字典Excel/CSV、DDL/Flyway脚本、PROC源码、Mapper映射,落 userData/db_index 带快照版本号+来源git ref;catalog_status 暴露覆盖率与解析失败率,过旧索引报告顶部告警。【账务一致性扩维——评审 major】对账不窄化为'本系统流水 vs 外系统文件':补总分核对/表内外核对/额度勾稽(已用+可用=授信)/担保授信勾稽/新老系统并账;'改额度模型不直接记账但破坏额度勾稽'这类必须能被识别。【v2】轻量 SQL 方言分词器(Oracle/DB2/MySQL/达梦/GaussDB)做简单 INSERT/UPDATE 表名抽取与 PROC 读写矩阵——但仍把 PROC 业务逻辑判定留给人。

**信贷特有考量**

信贷表分三类分开设计:台账/余额表(双写幂等+乐观锁版本号)、流水表(只追加+冲正标识+原流水号)、参数码表(生效日期)。金额强制 DECIMAL(18,2) 禁 FLOAT(会计铁律,误改直接标红)。账务日≠系统时间(受日终影响)。计提模型变更触发存量重算/追溯补提(IFRS9 减值)是最大跑批回算,必须和'计提模型'场景显式挂钩。跑批改动必须出断点重跑/补跑设计。EAST/1104 报送视图依赖稳定字段口径,任一台账字段语义变更可能污染报送,历史回填不能改变已报送口径。

### 外系统调用——发现/契约/时序(静态推断+待联调确认)

**设计**

【识别】ext_scan_integrations 复用 git grep 多信号打分:客户端类名(*Client/*Adapter/*Gateway/*Feign)、配置键(url/endpoint/wsdl/appId/cert)、契约物料(wsdl/xsd/proto/定长报文/ESB服务码/ISO8583域)、MQ topic、@Scheduled/xxl-job 外呼、入向回调控制器(notify/callback)。关键修正:把'接口表/中间表写入+dblink+异库同义词'也识别为外系统边界——信贷大量'外系统调用'其实是写接口表+核心跑批捞,这是最易漏的一类。用可达性收口只呈现本次改动真正触达的子集。【契约 diff——评审 major:旗舰②不存在】诚实定位:底座无契约对账引擎、无 wsdl/xsd 解析、httpcap 是浏览器抓包不能抓ESB报文,且'代码不出网+不联调'阶段拿不到真实报文。所以 v1 退化为:解析仓库里的契约物料(常缺失/过期)与代码字段做文本比对,结论一律标'待联调确认';金额单位(分/元)/日期格式/字符集(GBK vs UTF-8)/码值映射/鉴权字段(渠道号/机构号/MAC验签/证书)任何变更专项标红;破坏性变更(删字段/改类型/必填收紧/码值含义变)升级为强制人审问题。【非功能契约】对每个触达外系统按幂等/重试/超时/降级/对账/时序六维体检,从代码取现状证据并标缺口。【时序】mermaid sequenceDiagram 改动前后对比图,标幂等键/超时/补偿冲正/对账回路。

**信贷特有考量**

写类外呼(放款/代扣/记账/额度占用释放)必须有幂等键且外系统按键去重,否则重试重复记账/重复扣款;半成功(我方超时但核心实际已记账)必须有冲正/红冲+对账兜底,不能简单重试——账务一致策略强制人工拍板,Agent 只汇证不定策。征信分'查询'(授权留痕/频次合规)与'报送'(向人行征信中心报借贷记录,评审指出方案把征信只当查询漏了报送方向,报错是监管事件)。反洗钱命中是阻断还是转人工需业务定。决策/规则引擎入参变更静默改审批结论,需关注规则版本/灰度。对账口径(金额精度/流水号/状态码映射)变更影响 T+1 文件对账,需求阶段必须显性问。担保涉及抵押登记中心/中登网这一被漏掉的外部登记机构交互。

## 四、详细设计文档章节模板(信贷详设)

1. 1. 背景与目标(源:需求卡 mail messageId;区分业务诉求 vs 合规硬约束,标产品线)
2. 2. 需求与验收点(源:REQ卡;每条验收点=一个 REQ-ID,文末生成 REQ-ID↔章节↔代码落点↔测试用例追溯矩阵)
3. 3. 术语与缩略语(源:glossary.json;信贷强制消歧:授信/敞口/额度(集团/客户/产品/单笔四层)/五级分类/计提拨备/本金利息罚息复利/用信支用/展期借新还旧)
4. 4. 业务流程 As-Is/To-Be(mermaid;审批流画全多级/退回/并签会签/转授权/有条件通过/否决/贷审会票决,每节点标权限角色与流转触发条件)
5. 5. 领域模型(mermaid ER/class;客户-授信-合同-担保-放款-还款计划-账务的聚合边界与基数;含担保独立生命周期与额度冻结解冻状态边)

## 五、完备性保证与防幻觉机制

- 证据强制 Evidence-or-UNKNOWN:每条业务/影响/外系统/账务断言落 Claim Ledger(append-only),必带 source(code:file:line@gitsha / db:schema.object / api:operationId / mail:messageId / reg:监管条款号 / memory)+ confidence;reduce 成文档前 evidenceGate 拦截无源 claim,自动降级 UNKNOWN 并熔成澄清问题——宁可问不可编
- reg 制度证据与 code 同等地位(吸收评审 blocker):信贷大量规则不在代码里,必须建条款级制度库+生效版本;无制度库时模型现编监管条款比不答更危险,故 reg 缺失维度在 Coverage Matrix 显式标'制度库覆盖不足'而非假装完备
- 覆盖度 Coverage Matrix(完备性的分母):先机器枚举受影响实体全集(代码/库表/接口/审批分支/外系统/监管),再逐行标 covered/partial/UNCOVERED/NA;判据=无 UNCOVERED 或每个已转问题/签字豁免。关键:分母由域大脑枚举驱动,域枚举不全=分母缺失=机器'自信报告 covered',故域枚举完整性本身是 Coverage 有效性前提,需专家验收
- 人工业务定性兜底机器 flag(吸收评审 major):不把'是否高风险'唯一交给自动识别器;G1 强制业务/风险人员独立回答'是否触及账务/监管/客户权益'三问,机器漏扫时人审不跟着漏。flag 是关键词匹配易误标/漏标,人工定性是不可绕过的兜底
- 置信度五级 C1-C5:机器按 source 类型自动定级,人可下调不可上调;memory 类封顶 C2(防上次会话猜测被当事实);账务/计息/监管/审批权限 claim 未达 C4 强制进人审
- 合规检查单(详设产出前自动跑):审计留痕/数据安全数据不出网/账务一致(事务边界+对账+冲正)/监管报送(EAST-1104+征信报送+集中度校验)/变更可回溯+消费者权益保护,每项 pass/缺口/NA+证据 ref,缺口全进人审
- 文件:行 锁 commit hash 而非浮动行号,ddd_validate(纯 JS)定期反查行漂移标 STALE;LLM judge 做引文核对(claim.text vs source 实际内容)防转述幻觉,不一致标 MISCITED 降级——但关键账务/监管 claim 不能只靠 judge,必须落人审 G3

## 八、仍存风险

- 领域大脑覆盖范围无量化验收=系统性漏掉半个信贷业务:Coverage Matrix 有效性 100% 依赖域枚举完整性,而枚举易被高频子集(放款前+计息)定义,缺贷后/资产处置/跨产品/制度类规则时机器会'自信报告 covered'。缓解:专家验收 checklist + 人工业务定性兜底 + reg 缺失显式标'覆盖不足',但根治需持续的业务投入,这是项目最大不可控变量
- 代码现状≠业务现状的 proxy 在'新产品/监管整改'场景失效:代码几乎全空时,以代码为主证据底座的管线会么把全部规则转问题轰炸、要么 Coverage 因枚举不出实体而误判已覆盖。缓解:reg 制度库+域大脑为主证据源,但制度库本身是冷启动短板
- 底座承重墙被低估一个数量级:git_grep/行级scan/SQL-PROC解析/向量库/契约引擎/docx 全是新建,按现有 1 人份工程密度,v2/v3 多个库级子系统(SQL方言解析器/中文embedding/wsdl解析)落地周期与资源高度不确定,PROC 自动血缘可能永远停在半自动
- PROC/动态SQL/规则引擎配置静态分析天然不全(漏报比误报致命):信贷核心账务逻辑藏在 PL/SQL,EXECUTE IMMEDIATE/拼列名/规则文件 grep 抓不到。缓解=灰区清单强制人工+默认保守判'可能受影响',但'扫不到'与'没影响'的边界永远靠人兜底
- 契约影响在不出网不联调阶段只能静态推断且物料常缺失/过期:外系统多为线下 Word/PDF 接口文档无结构化物料,契约 diff 退化为文本比对,兼容性结论须标'待联调确认',真伪要等联调 httpcap——需求设计阶段给不出确定性兼容判断
- 并发与回灌正确性依赖尚未实现的事务层:req-ledger 多卡/多子Agent 并发 req_advance 会丢更新(现 JSON 全量覆盖无锁);答复回灌按 messageId 匹配依赖 In-Reply-To,在 Outlook/Exchange 不总可靠串联,错挂污染追溯链——需文件锁+版本校验+人工确认兜底,不默认自动回灌
- auto 模式滥用绕过风险闸口 + 闸口疲劳:关键词式 flag 误标导致 auto 全面失效(用户疲劳后绕过)或为降噪放宽匹配导致漏标。缓解=高风险硬编码禁 auto+人工定性兜底,但人审负载(尤其 G1 补漏报、G2 多对端会签澄清、灰区逐条确认)可能成为吞吐瓶颈
- MCP 数量膨胀拖慢每次会话:七 facet 需新增 10+ MCP,全注册进全局共享 opencode.json 被所有会话加载,工具表膨胀让模型选工具变慢、启动开销增大;需先解决按需加载/多实例/per-repo cwd 注入的底座问题

## 附录 A · 对抗评审发现的缺口(已在方案中吸收)

### 🔴 Blocker(6)

- **[信贷业务完备性 / 合规视角(对抗评审)] 信贷业务流程枚举严重不全:贷后/资产处置/合同变更整条线缺失,域对象表把'信贷'窄化成了'放款前+计息'** — 方案的'触发流程节点'枚举反复出现的是:申请受理/审批/放款/还款/计息/分类。但信贷业务的下半场几乎全缺,而这恰恰是规则最复杂、最易出事故、最常提需求的部分:(1)展期/重组/借新还旧/置换——涉及原合同关闭与新合同生成、未收利息结转、担保链重新评估、风险分类是否自动下调(重组类按监管要求通常应至少归为关注甚至次级)、是否进重组观察期,这是一整套独立业务流程,方案只在 creditSpecifics 里一句'展期重组'带过,没有进域对象/流程枚举,credit_domain_lens 自然不会识别。(2)核销——核销是有独立审批权限矩阵、独立账务处理(冲减拨备、表内转表外、核销后仍需继续催收'账销案存')、独立监管报送口径的重大业务,方案完全没提。(3)不良处置/以物抵债/资产转让/批量转让——抵债资产入账、债权转让定价与账务、不良批转,完全缺失。(4)额度冻结/解冻/调减/失效/恢复——额度生命周期管理(风险预警触发冻结、客户主动申请解冻、额度到期失效、年审恢复)是高频需求,方案只在'额度占用/释放'层面提及,没有冻结解冻这条状态边。(5)合同要素变更——利率重定价(LPR 换锚)、还款方式变更、期限调整、担保人变更,这些是合同存续期最常见的变更类需求。一条号称覆盖'信贷需求→详设'的管线,如果域对象/流程枚举里没有这些节点,credit_domain_lens 把一条'某客户申请展期'的需求映射时就只会落到'还款计划/计息',漏掉合同关闭、风险重分类、担保重估三条主线,而管线的'完备性自检'(Coverage Matrix)只会基于它枚举出的分母去判断——分母本身缺失,机器会'自信地'报告 covered。这是方案最大的结构性漏洞:完备性保证机制的有效性完全取决于域枚举的完整性,而域枚举被一个明显偏窄的高频子集定义了。
- **[信贷业务完备性 / 合规视角(对抗评审)] 把'业务规则'等同于'代码/DB 现状'——信贷大量规则在制度文件和监管口径里,不在代码里,这套证据底座对它们系统性失明** — 整套'防幻觉/完备性'设计的证据类型是:code(file:line@sha)、db(DDL/存储过程)、api(契约)、mail(messageId)、reg(条款号)、memory。前四类是代码/数据现状,占绝对主导;reg 类虽然列了,但没有任何采集、索引、版本管理的设计(L0 语料采集里'监管/制度源'只是'只读、带条款号'一句话,没有像代码那样建索引/RAG)。问题在于:信贷的核心业务规则相当一部分根本不在代码里——展期能展几次/最长多久、核销的审批层级与条件、风险分类的迁徙规则(逾期天数 vs 本金偿还能力 vs 担保覆盖的综合判定)、利率定价权限、单户/集团授信集中度限额、关联交易认定——这些是写在信贷管理办法、授信尽职指引、风险分类指引、监管规章里的。方案的逻辑是'代码里看到 X 所以问 Y / 断言 Y',但对'代码里根本没实现、而业务上必须有'的规则(典型:本次新系统/新产品需求,代码里压根没有对应实现),它的'现状采集'会返回空,然后呢?方案说'无源即降级为 UNKNOWN 转问题'——这看似安全,但会导致一种灾难性退化:对一个全新信贷产品的需求(代码现状几乎全空),管线会把几乎所有业务规则都转成'待澄清问题'轰炸业务,或者更糟,Coverage Matrix 因为'代码分母'枚举不出东西而显示'无受影响实体=已覆盖'。本质矛盾:用代码现状当业务现状的 proxy,在'增量改造存量系统'场景下尚可,在'新产品/新功能/监管整改新增规则'场景下(这恰恰是信贷需求的大头)直接失效。reg 类证据必须被提升为和 code 同等地位的一等公民,有独立的制度库建设、条款级索引、版本生效日期管理,否则'合规口径以哪个监管文为准'这个被方案自己列为'自动生成的硬问题'根本无法回答(没有制度库,模型只能现编监管条款,这比不答更危险)。
- **[信贷业务完备性 / 合规视角(对抗评审)] 跨产品差异(对公/对私/小微/零售/信用卡)被完全抹平,域对象/流程/规则全部按单一模型设计,这在信贷系统里是致命简化** — 方案通篇用'信贷'作为单一领域,域对象图、状态机、口径词典都是一套。但'信贷管理系统'在真实银行里,对公(公司授信)、零售/个贷、小微、信用卡/分期 的业务模型差异巨大,且常常是不同子系统或同系统不同分支:(1)授信主体——对公是法人客户+集团关联(涉及集团授信、统一授信、关联交易合并计算),零售是自然人+共借人+担保人,模型完全不同。(2)额度结构——对公有集团/单一客户/品种/单笔多级限额和敞口/或有的区分,零售多是单一产品额度或循环额度(信用卡),方案的'额度'域对象只画了'集团/客户/产品/单笔四层',这是对公视角,零售/信用卡的循环授信、分期、最低还款、账单日/还款日模型完全没有。(3)审批模式——对公是人工多级审批(方案重点描述的并签会签退回),零售/小微大量是自动审批/评分卡/规则引擎决策(申请评分、行为评分、额度模型),信用卡是实时授权决策,这套以'审批工作流状态机'为核心的设计对'自动化决策授信'场景几乎不适用。(4)风险分类——对公多为逐笔人工分类,零售是组合/资产池方式按逾期天数自动迁徙,口径不同。(5)计息还款——零售有等额本息/等额本金/先息后本/随借随还,信用卡有循环利息/分期手续费/免息期/最低还款,对公有按季结息/到期一次还本等,'还款冲销顺序'在不同产品下监管和合同约定都不同。方案的口径词典 creditSpecifics 把这些塞进同一个'还款方式'枚举,但没有任何'产品维度'的切分。后果:credit_domain_lens 对一条信用卡分期需求和一条对公流贷需求会用同一套域对象去映射,大面积错配。完备性的前提是先回答'这是哪条线/哪个产品的需求',这个最基本的分诊维度在整个七facet里没有出现。
- **[技术可行性与底座契合（站在 BocomHermes-shell 实际代码上找漏）] git_grep 这个被七个 facet 反复依赖的能力，在 git-mcp.mjs 里根本不存在** — facet3/4/5/7 全都把 scan_impact 描述成'复用 git-mcp 的 git grep 引擎'、'扩展 git-mcp 增 git_grep'。我读了 /Users/zhangang/.../mcp/git-mcp.mjs：它只暴露 git_status/git_log/git_diff/git_blame/git_show/git_branch 六个工具，TOOLS 表里没有任何 grep。唯一的 grep 在 repro-mcp.mjs 的 scan_impact 里，是 `execFileSync('git',['grep','-l',...])` —— 注意 `-l` 只返回文件名列表，落盘的 scans/<bundleId>.json 里 files 字段也只有文件路径，没有行号、没有片段（行号片段只在返回给 agent 的临时 preview 里，不持久化）。而方案通篇承诺的是'文件:行 可点跳 + 命中代码片段 + 所在层'的 HitMatrix。结论：行级、分层、带 kind 过滤的符号扫描器是从零新建，不是'扩展'。这是 facet3 整个'影响面雷达'的地基，地基不存在。
- **[技术可行性与底座契合（站在 BocomHermes-shell 实际代码上找漏）] 信贷领域知识底座（facet2）需要本地向量库+embedding+本体+RAG，底座里一行都没有，且与'数据不出网+内网推理端'叠加后工程量巨大** — facet2 要 sqlite-vss/本地 embedding、ontology.json 本体图、kref 寻址、glossary、PROC 规则抽取。我在仓库里搜 sqlite/vector/embedding/向量，零命中（grep 出的几个文件是 opencode.js 等的误匹配，无实际依赖）。这意味着：(a) 要在内网信创环境跑本地 embedding 模型——而 memory 的 project_intranet_mode 说底座是'连内网 opencode serve'，embedding 端是否存在、向量化谁来跑、中文信贷术语召回（facet2 自己列的 risk）如何解决，全是未定的外部依赖；(b) 本体半自动生成+人逐项确认，是一个需要业务深度投入数周的 onboarding 工程，方案 risks 里也承认'业务不配合则大量 unverified'。把这个 facet 当成 facet3/4/5/6 的'领域大脑'前置依赖，等于给整条管线压了一个最重、最不可控、最依赖外部资源的关键路径。建议显式承认：v1 用'人工维护的 glossary.json + 码值表（纯 JSON，无向量）'起步，RAG/本体延后，否则首期不可能交付。
- **[技术可行性与底座契合（站在 BocomHermes-shell 实际代码上找漏）] 分期切分缺失：方案把七个 facet 当成一个整体管线交付，但它们的底座依赖度差异巨大，不分期=不可能尽早可用** — 用户明确问'分期怎么切才能尽早可用'，但方案没有给任何 MVP/分期。从底座契合度排序：能尽早落地的是 facet1（状态机+REQ 账本 req-ledger.json，纯 JSON+orchestrate+onBeforeBatch，全部现成）、facet5/需求问题生成（基于现有 fan-out + outbox + mail_reply 闭环，问题质量靠 prompt 和人审兜底，不强依赖向量库）、facet7 的 claim 账本最小版（append-only JSON，先只做'无源即降级转问题'，不做向量回放）。这三个组合起来就是一个真能用的'需求→结构化→人审→澄清问题外发→闭环登记'v1。而 facet2（领域 RAG/本体）、facet3（行级影响雷达+缓存）、facet4（DB 血缘+SQL 解析）、facet6（契约对账+报文解析）都需要先建重型子系统，应作为 v2/v3 且每个先交付'半自动+强人审'降级版（如 facet3 v1=只做单仓 git grep 文件级命中+blame 责任人，不做 BFS 闭包和跨双仓行级矩阵）。不给这个切分，项目会卡在'领域大脑没建好→全管线都现编→完备性崩塌'（facet2 自己点名的头号 risk）。

### 🟠 Major(14)

- **[信贷业务完备性 / 合规视角(对抗评审)] 审批流只覆盖了'退回/并签/会签/超时升级'的清单名词,但漏掉信贷审批的真正难点:授权权限矩阵、降级审批、转授权额度、面签双录、贷审会、否决与有条件通过** — 方案多处把'并签/会签/退回/超时升级/代办授权'当作要补的清单项,这是对的方向,但停留在流程引擎节点层面,漏掉了信贷审批特有的业务规则维度:(1)审批权限/授信审批权限分级——不同金额/不同风险等级/不同产品对应不同审批层级(支行/分行/总行/贷审会),且有'转授权'(上级把部分权限授予下级)和'降级/越级审批'规则,这不是流程引擎能表达的,是权限金额矩阵。方案说'对接统一身份权限系统',但统一身份权限通常只管功能权限(谁能点这个按钮),管不了'这笔 5000 万的对公授信必须上分行贷审会'这种业务审批权限,二者常被混为一谈。(2)有条件通过/审批意见落实——审批可以'同意但附加条件'(如要求追加担保、降低额度、限制用途),这些条件要在放款前核查落实,是一条独立子流程。(3)否决与重新申报的冷却期/再营销规则。(4)面签、双录(录音录像)、合同面谈——这些合规动作是放款的前置硬卡点,尤其零售和监管要求,代码里可能是一个状态位,但业务上是合规留痕要件,方案的'审批流'里完全没有。(5)贷审会(信贷审查委员会)的会议制审批——多人独立表决、票决规则、回避制度,和普通会签不同。把这些都归约为'退回路径别漏掉',是把信贷审批的合规深度做平了。
- **[信贷业务完备性 / 合规视角(对抗评审)] 计息/计提的'调整类'需求被当成单纯账务变更,漏掉信贷计息的业务复杂度:欠息/表外利息、利息减免、罚息复利、重算补提、税务(增值税价税分离)** — 方案对'计息计提'的处理是:命中触账 flag → 强制走账务时序/借贷方向/冲正方案 + 问'是否调整科目/记账规则/计提模型'。这是把它当成一个会计记账问题,但信贷计息有大量纯业务规则维度被漏:(1)欠息处理——利息未能按期收回时的表内应收利息转表外欠息(进入'表外'核算)、停止表内计息(non-accrual)的触发条件(通常逾期 90 天),这是信贷会计与监管的硬规则,任何动计息的需求都要问'是否影响表内外利息划转口径',方案没有这个维度。(2)利息减免/欠息豁免——有独立审批权限和账务,核销利息和减免利息不同。(3)罚息与复利的计算基数、计收方式(对逾期本金计罚息、对应收未收利息计复利)、上限(罚息利率不得超过监管/合同上限),LPR 改革后的定价规则。(4)计提模型变更的存量重算/追溯补提——预期信用损失(IFRS9/CECL,国内对应监管口径)的减值计提模型一旦变更,要对存量资产重算,这是巨大的跑批回算,方案的'存量重算'只在 DB facet 笼统提到,没有和'计提模型'这个最易触发回算的场景挂钩。(5)税务——贷款利息收入涉及增值税价税分离,计息口径变更可能联动税务计算,这是常被漏的跨域影响。把计息计提仅当'借贷方向+冲正'处理,是只看了账务的'记账'面,漏了它的'业务规则'面和'税务/监管口径'面。
- **[信贷业务完备性 / 合规视角(对抗评审)] 担保/抵押/质押被当成一个 flag,漏掉担保业务的独立生命周期:抵押登记/解押、价值重估、担保变更/置换/释放、担保代偿/追偿、共同担保与担保链** — 方案只在一处提到'担保抵押质押链→关联决策规则引擎、担保物价值重估',把担保当成风险分类的一个输入。但担保在信贷里是一条有独立状态机和独立业务流程的主线,需求频繁:(1)抵押/质押登记与解押——抵押权设立要去不动产登记中心/动产登记(中登网)办登记,有'已抵押/登记中/已解押'状态,放款常以登记完成为前置条件,结清要触发解押,这是和外部登记机构的交互(又一个被漏的外系统)。(2)担保变更/置换/追加/释放——存续期内换担保人、换抵押物、追加保证金、部分释放(如分笔放款分次释放抵押),涉及额度与担保的重新匹配。(3)担保物价值重估周期与减值——抵押物定期重估、价值下跌触发追加担保或风险预警,价值数据可能来自外部评估机构。(4)担保代偿与追偿——保证人代偿、担保公司代偿后的代偿挂账、追偿台账,这是不良处置阶段的核心。(5)共同担保/担保链/循环担保的关联识别(关联交易、互保联保风险)。一条'担保变更'需求进来,方案的 credit_domain_lens 因为没有担保域对象的完整模型,只会把它当成'改个担保字段'走改动点扫描,完全漏掉登记机构交互、额度联动、风险重估三条主线。
- **[信贷业务完备性 / 合规视角(对抗评审)] 监管报送只反复点名 EAST/1104,但信贷监管报送远不止这两个,且报送之外的'监管业务规则'(集中度/拨备覆盖率/资本计量/大额风险暴露/LPR报送/征信报送)整体缺失** — 方案把监管口径几乎等同于'EAST/1104 报送字段映射',这是把监管完备性窄化了:(1)报送清单不全——信贷相关监管报送至少还有:人民银行金融统计(如信贷收支表)、征信报送(向人行征信中心报送借贷记录,这是每笔贷款全生命周期都要报的,且报错是监管事件,方案把征信只当'查询'外系统,漏了'报送'方向)、大额风险暴露报送、贷款基础数据/利率报送(LPR 相关)、绿色信贷/普惠/涉农等专项统计、反洗钱大额可疑交易报送、监管 EAST 之外的检查数据报送。(2)监管业务规则不止报送——授信集中度限额(单一客户/集团客户/最大十家)、拨备覆盖率与贷款拨备率监管红线、资本充足率相关的风险加权资产计量(信贷资产的风险权重)、流动性相关、不良率监控。这些是信贷系统里有强制业务校验逻辑的(如新增一笔授信要校验是否突破集中度限额),不是报送层的事,而是审批/放款环节的硬卡控。方案没有把'监管限额校验'作为一类业务规则纳入域枚举。(3)征信报送方向的合规——报送数据质量、异议处理、不良记录报送的时点与展示期限,都是合规高敏感点。把监管完备性做成'命中 EAST/1104 抽数 SQL 就标 P0',只覆盖了报送的'数据出口'一面,漏掉了监管规则作为'业务前置校验'的一面。
- **[信贷业务完备性 / 合规视角(对抗评审)] 账务对账被设计成'本系统流水 vs 外系统对账文件'的技术对账,漏掉信贷特有的多维核对:总分核对/内外账核对/表内外核对/额度与用信勾稽/担保与授信勾稽/新老系统并账** — 方案的'对账'集中在外系统交互 facet:代扣/放款/记账的 T+1 文件对账、长短款、勾对口径。这是支付清算式对账。但信贷系统的账务对账维度更广,且很多是系统内部的勾稽校验,需求经常涉及:(1)总账与明细账核对(总分核对)——业务台账汇总数必须等于会计总账数,任何动账务的改动都要保证总分平,这是比借贷平衡更高一层的一致性。(2)表内表外核对——已核销资产、表外欠息、未使用额度(或有)的表外科目与台账的勾稽。(3)额度与用信的勾稽——已用额度+可用额度=授信额度,放款/还款实时影响,改动易破坏。(4)担保与授信的勾稽——担保覆盖率、足值担保的统计。(5)新老系统并行期的双账核对(系统迁移/上线常见需求)。(6)计提/拨备与资产质量的勾稽。方案的对账设计基本只覆盖了第(1)条中的'与外部对'那一半,系统内部的多维勾稽校验完全没有作为一类'账务一致性'纳入。结果:一条改额度模型的需求,管线会查'触账了吗'(可能没有直接记账)从而放过它,但它实际破坏了额度勾稽,这种'不记账但破坏一致性'的场景是方案的盲区——它把账务一致性窄化为'记账分录的借贷平衡'。
- **[信贷业务完备性 / 合规视角(对抗评审)] '高风险标记禁用 auto'听起来安全,但风险标记的判定完全依赖域库的触账/外系统识别,域库不全=漏标=auto 放行高危需求,这是把金融级管控建在了知识库完整性这个未经验收的假设上** — 方案的核心管控逻辑是:命中触账/外系统/存量/监管 flag → 强制不可 auto + 强制走专项 checklist。这个逻辑的安全性 100% 依赖于'flag 识别正确'。但 flag 由 credit_domain_lens / scan_impact 自动判定,而方案自己在 risks 里承认'领域大脑缺失会退化''漏报比误报更危险'。这构成一个危险的闭环:(1)如果域库没把'核销'识别为触账业务(前面已论证它根本没在枚举里),那么一条核销需求不会被标高风险,G1 可被 auto 放行,详设不会强制账务时序——而核销恰恰是重大账务+监管事件。(2)flag 是布尔标记,但信贷风险是分级的,'改个利率展示文案'和'改利率计算引擎'都可能命中'计息'关键词,前者 P3 后者 P0,方案的关键词匹配式 flag(creditSpecifics 里写的是命中'计息/计提/放款/还款/账务/总账'就标)会大量误标导致 auto 全面失效(用户疲劳后绕过),或为降噪而放宽匹配导致漏标。真正的金融级管控不能把'是否高风险'交给一个承认自己可能不全的自动识别器作为唯一闸门,必须有'人工业务定性'这一不可绕过的兜底:任何信贷需求在 G1 必须由业务/风险人员显式回答'这是否触及账务/监管/客户权益'三个定性问题,而不是等机器扫出 flag。方案把人审闸口设计成'审机器扫出来的清单',而不是'独立做业务定性',一旦机器漏扫,人审也跟着漏(人只看到清单上没有,不会凭空想到机器没列的)。
- **[信贷业务完备性 / 合规视角(对抗评审)] 客户/主体维度的业务规则被忽略:统一授信与关联方合并、黑名单/受限客户、客户准入与行业政策、共债与多头授信、KYC/受益所有人** — 方案的域对象提到'客户/对手方',但客户维度的信贷业务规则几乎没有展开,而这些是审批和放款的硬约束:(1)关联方/集团客户合并授信——对公授信必须识别关联关系、集团成员合并计算授信和集中度,关联交易要单独认定和审批,这是对公信贷的核心规则,前面跨产品差异已提,这里强调它是'客户准入/授信测算'的前置。(2)黑名单/受限客户/行业限额——客户准入要查内部黑名单、行业信贷政策(限制类/禁止类行业)、区域政策、名单制管理,这些是放款前的硬卡控,涉及独立的名单管理子系统。(3)多头授信/共债识别——通过征信识别客户在他行的负债,控制过度授信,这是监管和风控要求。(4)受益所有人/实际控制人识别(穿透)、KYC、客户风险等级——和反洗钱联动。(5)客户主数据变更——客户合并、客户信息变更(更名、证件换发)对存量授信合同的影响。方案把客户当成一个静态域对象,没有把'客户准入校验''关联合并''名单管控'作为一类业务规则纳入,一条涉及客户准入或关联认定的需求会被严重低估影响面。
- **[技术可行性与底座契合（站在 BocomHermes-shell 实际代码上找漏）] scan_impact 与 bundle_note 强绑 bundleId（调试复现流），与需求阶段语义不兼容，不能直接拿来当需求侧引擎** — repro-mcp.mjs 里 scan_impact 的入参 required:['bundleId','symbol','cwd']，bundle_note 的 required:['bundleId','key','status','evidence']，且 SUSPICIOUS 机制是围绕'一个浏览器复现包 b_xxx'设计的——bundleId 来自前端 bug 的录制取证（DOM/网络/recording）。facet1/3/4/7 想把这套'confirmed/excluded/maybe + evidence'平移到 REQ-ID 驱动的需求分析，但现有实现里没有任何'以 REQ-ID 为主键'的证据空间。要么改 repro-mcp 让它接受 reqId 作为另一种 bundle 命名空间（侵入式改造，且会污染调试流），要么新建一套 claim-ledger-mcp（facet7 自己也承认要新建）。方案多处写'复用 bundle_note 的强制 evidence 范式'，实际是'照着它的思路另写一个'，工作量和兼容风险被一句'复用'抹掉了。
- **[技术可行性与底座契合（站在 BocomHermes-shell 实际代码上找漏）] REQ-ID 上下文注入声称'复用 firstMsgCtx'，但编排路径（orch.js / window.js 的 run）根本不注入任何上下文** — facet1 说'每张挂该需求的会话卡，首条消息静默注入 REQ-ID 前缀，复用现有 firstMsgCtx 机制'。但 firstMsgCtx 只在 src/session.js 的 card-send 里被消费（行 177-179），且只对交互式对话卡生效。而所有 fan-out 子任务走的是 orchestrator.js 的 openCodeRunner 或 orch.js 的 run（行 32-38）——它们直接 createSession+sendMessage(prompt)，prompt 由 buildWorkPrompt 拼，全程不碰 firstMsgCtx、不读 memory、不注入项目背景。也就是说子 Agent 默认是'裸上下文'。要让每个子 Agent 都知道 REQ-ID/信贷领域知识/口径词典，必须改 buildWorkPrompt 或在每个 run 里手动前缀拼接，并自行做长度截断（facet1 risks 里提到的'上下文膨胀'是真问题，但解决它的 loadProjectContext 截断逻辑在编排路径里压根没接上）。这是'看起来复用、实际未接线'的典型。

- **[技术可行性与底座契合（站在 BocomHermes-shell 实际代码上找漏）] 新增 MCP 全部走全局共享 opencode.json，且 git-mcp 的 cwd 在启动时固定，多 repo/多 REQ 并发会串味** — window.js 的 mcpEntries()（行 1990-1996）把 MCP 以静态命令写进一个共享的 opencode.json（mcp-register 写 %APPDATA%/opencode/opencode.json）。git-mcp.mjs 的 cwd 取 process.cwd() 或固定环境变量 BOCOMHERMES_GIT_CWD（行 10），一个 git-mcp 进程只认一个仓。而方案要 facet1 的 req_ledger、facet2 的 knowledge/ontology/dict、facet4 的 db-catalog、facet5 的 db_lens/contract_lens、facet6 的 ext_scan/ext_contract、facet7 的 claim-ledger/coverage/compliance —— 七个 facet 至少要新增 10+ 个 MCP server。问题：(a) 它们都注册进同一个全局 opencode.json，被所有会话（包括无关的日常对话卡）加载，启动开销和工具表膨胀（模型要在几十个工具里选）会拖慢每一次 run；(b) 双仓场景下 git-mcp 只能绑一个 cwd，要同时 blame 前后端必须起两个 git-mcp 实例并分别注入 BOCOMHERMES_GIT_CWD，现有注册机制没有这个 per-serve 注入路径。MCP 的多实例/按需加载/per-repo cwd 是一个需要先解决的底座问题，否则新增 facet 越多，每个会话越重。

- **[信贷业务完备性 / 合规视角(对抗评审)] 时间维度与跑批时序被当成技术问题,漏掉信贷的'会计日/营业日/月末年末'业务语义和跨日交易的业务后果** — 方案在 DB/外系统 facet 提到'跑批时点/日终切日/账务日≠系统时间',方向对,但停留在技术一致性层面,漏掉信贷特有的时间业务语义:(1)会计日/营业日/记账日的区分对计息天数、逾期天数认定的业务影响——逾期'第几天'的认定(自然日 vs 营业日、跨节假日)直接决定罚息和分类,方案的'边界 lens'提到了>30 还是≥30,但没有'营业日 vs 自然日''宽限期'这种信贷特有的天数规则维度作为必问项。(2)月末/季末/年末的特殊处理——结息日(按季 21 日结息)、月末计提、年末结转、跨年计息,这些时点的特殊业务逻辑。(3)同一笔业务跨日的状态后果——比如放款指令当日未处理跨日后利率/额度可能变化。(4)回溯/倒起息——业务上允许的起息日早于交易日的场景。(5)节假日顺延规则(还款日遇节假日顺延/提前)对还款计划的影响。(6)系统切日期间的业务冻结窗口。这些都是'看似只是时间字段、实则承载信贷业务规则'的点,方案把时间统一当技术跑批问题,会漏掉大量由时间语义驱动的业务分支。
- **[信贷业务完备性 / 合规视角(对抗评审)] 需求来源被窄化为'邮件/工单/文档'的功能性变更,漏掉信贷需求的两大特殊来源:监管整改/政策落地、生产事故/数据修复,二者的业务流程与普通需求不同** — 方案的 INTAKE 设计了邮件/工单/文档三种来源,需求类型枚举了'新功能/缺陷/监管整改/参数调整/报表'。但对信贷而言有两类来源的处理流程显著不同,值得单列:(1)监管整改/政策落地类——这类需求往往带强制时限(监管要求 X 月底前完成)、需求来源是监管文件而非业务方、合规口径是硬约束不可商量、且常涉及存量数据合规性追溯整改。这类需求的'澄清问题'不是问业务方而是要查监管原文+可能要向监管报备方案,管线的'问业务/产品'外发模型不适用。方案虽在类型里列了'监管整改'但全流程没有为它做特殊处理(比如强制挂监管条款、合规人员强制参与、时限驱动)。(2)生产事故应急/数据修复类——信贷生产事故(如计息错误、账务不平、批量数据错误)的修复需求,本质是'改存量数据'而非'改代码逻辑',涉及数据修复脚本审批、修复前后账务平衡校验、是否需要重报监管、是否需要客户补偿/退费,这是一条以'数据订正'为核心的特殊流程,且高度合规敏感(动客户的钱/记录)。方案的整条管线是为'代码改动'设计的(改动点=文件:行),对'纯数据修复/批量订正'类需求(不改代码或主要改数据)适配很弱。
- **[信贷业务完备性 / 合规视角(对抗评审)] 客户权益/消费者保护维度完全缺席:利率展示与综合融资成本、不当催收、息费透明、个人信息授权链路、争议处理** — 方案的合规检查单聚焦在'数据不出网/审计留痕/账务一致/监管报送/可回溯'——这是科技合规和监管报送合规,但漏掉了信贷业务合规中越来越重的'金融消费者权益保护'一整类,而这类需求正在变多:(1)综合融资成本/年化利率明示——监管要求向借款人明示年化利率(APR),息费的展示口径,这是产品和合同环节的硬合规。(2)息费收取合规——不得违规收费、捆绑搭售、变相提高利率。(3)催收合规——贷后催收的时间/频率/对象限制、不得暴力催收、委外催收管理,这是贷后系统的合规热点。(4)个人信息保护——征信查询授权(方案提到了)之外,还有个人信息收集使用授权链路、数据共享授权、授权撤回、信息删除,需求频繁。(5)客户争议/投诉处理与差错更正。这些既是合规也是高频需求来源,方案的'合规 lens'命中条件是'涉及征信/反洗钱/客户信息/留痕'就触发,口径偏窄(偏数据合规),没有把'消费者权益保护'作为独立合规维度,会漏掉一类合规风险代价很高的需求。
- **[信贷业务完备性 / 合规视角(对抗评审)] '问题外发给业务/产品'的澄清闭环假设了一个统一的业务对端,但信贷澄清的对象是分散的(业务/风险/财务会计/合规/科技/监管),且很多口径需要跨部门会签,单线邮件 thread 模型过简** — 方案的 G2/澄清闭环是:生成问题→outbox→mail_reply 挂回原 thread→答复回灌。这隐含假设'提需求的人'就能回答所有澄清问题。但信贷需求的澄清对象是高度分散的:数据口径问财务会计、风险分类规则问风险管理部、监管口径问合规、账务处理问财务、产品规则问产品部、技术实现问开发。一个'敞口含不含表外'的口径问题,可能需要财务会计和风险部门会签才能定,不是回封邮件就行。方案的单 thread 模型(挂回原邮件、In-Reply-To 串)会把多个不同对端的问题混在一封回信里,且无法表达'这个口径需要 A 部门和 B 部门共同确认'的会签式澄清。askWho 字段虽然存在,但下游的发送和回灌机制是单 thread 的,askWho 只是个标签没有路由能力。后果:复杂口径问题的澄清责任无法落地,容易出现'问了但回答的人无权定口径'的伪闭环,而管线会因为'收到回复'就把问题置为已澄清,污染追溯链的可信度。
- **[技术可行性与底座契合（站在 BocomHermes-shell 实际代码上找漏）] docx 导出'复用已有 docx skill'——这个 skill 在本仓库不存在，是宿主环境（Claude Code）的能力，不等于 BocomHermes 运行时具备** — facet1/6 多次写'可导出 .docx（已有 docx skill）'、'复用 anthropic docx 技能'。docx 确实在我当前这个 Claude Code 环境的 skill 列表里，但那是评审/开发态的宿主能力；BocomHermes 是一个 Electron + 内网 opencode serve 的独立运行时，里面没有这个 skill（仓库内 grep 不到 docx 实现）。详设文档导出 Word/Confluence 在生产运行时要么自己实现（docx 库 + 三层标签转批注/水印 + 脱敏），要么放弃。这是一个小但典型的'把评审环境能力误当产品底座能力'的混淆，按方案口径会在交付时落空。
- **[技术可行性与底座契合（站在 BocomHermes-shell 实际代码上找漏）] 状态机并发/乐观锁、答复回灌按 messageId 精确匹配——这两处的正确性依赖尚未存在的账本层事务保证** — facet1 risks 自己点了'账本层乐观锁（updatedAt 校验）'和'答复回灌按 messageId/thread 精确匹配，错挂会污染追溯链'。但 req-ledger.json 是个本地 JSON 文件（对齐 todos.json）。todos.js（75 行）是单进程串行写，没有并发控制。多卡/多子 Agent 通过 req_ledger MCP 并发 req_advance/req_attach_artifact 时，JSON 文件读改写竞争会丢更新。outbox.js 已经踩过类似坑（用 save() 全量覆盖 + 重启清理 sending→unknown）。这不是不能做，而是方案把它当 risk 一笔带过，实际需要在 req_ledger MCP 里实现文件锁/版本校验/append-only 事件溯源，属于必须设计但被低估的正确性工程。messageId 回灌匹配则依赖 mail_list 下一轮命中同 thread——而 In-Reply-To/References 在 Outlook/Exchange 环境下并不总是可靠串联，错挂风险真实存在，需要兜底人工确认而非默认自动回灌。

## 附录 B · 各 facet 原始设计(完整)

### 端到端流程与人机闸口:需求录入→理解结构化→改动点&影响分析→需求问题生成→详细设计文档→评审确认 的完整管线(REQ-ID 贯穿,信贷管理系统)

**目标**:把研发第一道大关做成一条"有状态机、有闸口、有证据、可追溯"的流水线:任何来源(邮件/工单/文档)的需求一键熔成带唯一 REQ-ID 的需求卡,经结构化→跨双仓改动点&影响分析→需求问题生成(向业务/产品反问)→详细设计文档→评审确认六个阶段流转;每个阶段有明确输入/输出/状态、有人审闸口落点、在 BocomHermes 里以特定卡片/窗口形态承载;信贷领域的业务规则、账务强一致、外系统契约这三类"看不见的坑"被显式建模成可勾选清单而不是埋在散文里。整条管线复用现有底座(orchestrate 的 onBeforeBatch 闸口、双仓 serve、mail/git MCP、context-matched diff、outbox 闸门、memory 注入),不另起炉灶。

## 0. REQ-ID 作为贯穿主键(第一性设计)

REQ-ID 是整条管线的"账户号"。格式 `REQ-{系统}-{YYYYMM}-{seq}`(如 `REQ-CLMS-202606-017`)。在"需求录入"瞬间生成,之后所有产物(结构化需求、改动点清单、问题集、设计文档、评审记录、后续的会话 diff / 提交 / 回归)都挂在它下面。落地为一个本地 JSON 账本 `req-ledger.json`(与 todos.json/memory.md 同级,在 app.getPath('userData')),单条记录形如:
```
{ reqId, title, stage, status, sources:[{type:'email', messageId, subject, date, from}], 
  artifacts:{ structured, changeImpact, questions, designDoc }, gates:[{gate, by, decision, at, comment}],
  sessions:[sid...], threadMsgIds:[...], createdAt, updatedAt }
```
REQ-ID 注入复用现有 `firstMsgCtx` 机制:每张挂在该需求下的会话卡,首条消息静默注入 `<需求REQ-CLMS-...>` 前缀(就像现在静默注入项目上下文/memory),让任何接手的子 Agent 都知道自己在为哪个需求干活。messageId 钉死原文(沿用 todos.js 已有的 mailMsgId 回填逻辑),三个月后还能 mail_get_full 打捞原始诉求。

## 1. 状态机(六阶段 + 闸口落点)

主状态:`INTAKE → STRUCTURED → IMPACTED → CLARIFYING → DESIGNED → IN_REVIEW → APPROVED`(+ 旁路 `BLOCKED`/`REJECTED`/`PARKED`)。状态机是单向推进 + 可回退:任何闸口"退回"都把 stage 拨回上一阶段并记 gate 决策。落地为账本里的 `stage` 字段 + 一个纯函数 `advance(reqId, fromStage, gateDecision)`(可单测,仿 orchestrator.js 的纯逻辑风格)。

四个**人审闸口**(只在这四处真正拍板,其余自动流转):
- **G0 立卡确认**(INTAKE 出口):机器从邮件/工单抽出标题+诉求,人确认"这确实是一条需求、归我处理、REQ-ID 成立"。轻闸口,一键通过居多。
- **G1 范围与影响确认**(IMPACTED 出口):人审"改动点清单 + 影响半径"。这是最贵的闸口——人勾掉误报落点、补漏报落点、确认风险分级。决定后续问题集和设计的边界。
- **G2 需求问题外发确认**(CLARIFYING 出口):机器生成要问业务/产品的问题集,**人审后才允许经发件箱(outbox)发出**。直接复用现有 outbox 15 秒延迟闸门 + mail_reply(挂回原邮件 thread)。这是"数据/语义不出网前的人控点"。
- **G3 设计评审签发**(IN_REVIEW 出口):评审人对详细设计文档逐条批注、必改/建议/可忽略分级,签发即 APPROVED,设计文档定版并回写工单/邮件。

闸口在代码上复用 orchestrate 的 `onBeforeBatch`(已有 approve/auto/abort 三态 + reqId 机制),把"批准下一批子任务"升级为"批准阶段产物+推进状态"。auto 模式允许熟练用户对 G0/低风险需求一路放行,但 G1/G2/G3 对"高风险标记"的需求强制不可 auto(见 creditSpecifics 的风险闸口)。

## 2. 逐阶段 输入/输出/承载形态

### 阶段一 · 需求录入 (INTAKE)
- 输入:① 邮件(mail_list/mail_get_full 原文 + 附件,mail_get_attachment_text 读 PDF/Word/Excel 需求说明书/变更通知)② 工单(粘贴文本或 HTTP 抓取)③ 本地文档(拖入卡片)。
- 处理:一个 analyst 子 Agent 抽 {标题, 一句话诉求, 提出方, 涉及业务域(从信贷域对象词表匹配), 紧急度, 疑似需求类型(新功能/缺陷/监管整改/参数调整/报表)}。生成 REQ-ID,写账本,登记 source(messageId 钉死)。
- 输出:**需求卡(Intake Card)** —— 新卡型,基于 card.html 骨架,顶部一条 REQ-ID 徽标 + 原文锚点(点开走 mailview.html 看原邮件)+ G0 一键"立卡/退回/这不是需求(归档)"。
- 状态:INTAKE →(G0 通过)→ STRUCTURED。

### 阶段二 · 需求理解与结构化 (STRUCTURED)
- 输入:阶段一的原文 + REQ-ID 上下文。
- 处理:analyst Agent 产出**结构化需求对象**(见 artifacts 的 StructuredReq),关键是把信贷需求拆成可勾的条目:涉及域对象、触发的业务流程节点(申请受理/审批/放款/还款/计息/分类)、是否触账(强一致敏感)、是否调外系统、是否影响存量数据/跑批、监管口径。这里调用一个新工具 `credit_domain_lens`(信贷领域知识库),把自然语言映射到标准域对象/流程枚举,避免每条需求让模型现编术语。
- 输出:结构化需求面板(承载在需求卡的可展开区,JSON+人读双视图)。此阶段**无独立闸口**(自动流转),但结构化结果会在 G1 一并被人审。
- 状态:STRUCTURED → IMPACTED(自动触发阶段三)。

### 阶段三 · 改动点 & 影响分析 (IMPACTED) —— 管线最重的一环
- 输入:结构化需求 + 双仓代码(projectDir 前端 / backendDir 后端,各自 ensureServe)。
- 处理:难度分诊后 fan-out。直接架在路线图旗舰①"跨双仓影响面雷达"上,把结构化需求里的域对象/字段/接口当种子,跑:
  - 前端落点:git grep 调用点、页面、表单字段;
  - 后端落点:方法引用、Mapper/SQL、存储过程(信贷系统存储过程多,需专门扫 `.sql`/proc 定义)、定时任务/跑批;
  - 账务影响:命中"触账"的需求,标红并强制走账务一致性 checklist;
  - 外系统影响:命中征信/核心/支付/反洗钱等适配层,列出契约面;
  - git blame 出"该问谁"(责任人)。
- 输出:**改动点清单(Change & Impact Card)** —— 复用 workflow.html 的 DAG/富结果形态,每个落点是可点 `文件:行` 跳 IDE(open-loc 已有),带风险分级(高/中/低)、置信度、blame 责任人。一张"影响半径矩阵":行=改动点,列=[触账? 调外系统? 影响存量? 影响跑批? 跨双仓?]。
- 闸口 **G1**:人勾误报/补漏报/确认分级 → 修正后的清单回写账本 artifacts.changeImpact。
- 状态:IMPACTED →(G1)→ CLARIFYING。

### 阶段四 · 需求问题生成 (CLARIFYING)
- 输入:结构化需求 + G1 确认后的改动点清单。
- 处理:analyst+architect 对抗生成"为了把设计做完备,还必须向业务/产品/上游问清楚的问题"。问题不是泛泛的"请明确需求",而是由 checklist 驱动的硬问题:每个标了"触账/调外系统/影响存量/监管口径"但信息缺失的条目,自动生成一条定向问题(见 creditSpecifics)。每条问题带:问谁、为什么问(挂到哪个改动点/风险)、缺了它会导致什么设计分支无法确定。
- 输出:**需求问题集(Clarification Card)** —— 可编辑清单,人增删改。
- 闸口 **G2**:人审后,选择"发出"→ 经 outbox 闸门,用 mail_reply 挂回原始邮件 thread(In-Reply-To 串起来),或生成工单评论草稿。回复到达后(下一轮 mail_list 命中同 thread),自动把答复回灌到该 REQ-ID,问题状态置"已澄清"。
- 状态:CLARIFYING →(问题全澄清 / 人判定信息足够)→ DESIGNED。允许"带已知风险继续"(把未澄清项标 ASSUMPTION 写进设计的假设章节)。

### 阶段五 · 详细设计文档 (DESIGNED)
- 输入:结构化需求 + 改动点清单 + 澄清答复/假设。
- 处理:fan-out(architect 主笔 + 各域 reviewer 校验),按**信贷系统详细设计模板**(见 artifacts 的 DesignDoc)生成文档,逐落点给到 `文件:行` 级方案、账务处理时序、外系统调用契约、数据迁移/跑批脚本影响、回滚方案、监管报送影响。文档每一节都反向链接到改动点 ID 和需求条目,保证"可追溯、无悬空设计"。
- 输出:**详细设计文档卡(Design Doc Card)** —— 富文本(复用 wf-final 的 markdown 渲染),可导出 .docx(已有 docx skill)/.md,落账本 artifacts.designDoc。
- 状态:DESIGNED → IN_REVIEW(提交评审)。

### 阶段六 · 评审确认 (IN_REVIEW → APPROVED)
- 输入:设计文档 + 全链路追溯(需求→改动点→设计→假设)。
- 处理:多 Agent 对抗评审(security/账务一致性/biz/arch 视角,复用旗舰评审能力)先做一遍机器自审,产出"评审材料包"喂给人。人(评审人/架构负责人)逐条批注。
- 输出:评审记录(necessity 分级)+ 定版设计文档。
- 闸口 **G3**:签发 → APPROVED;设计文档定版、回写工单/邮件(outbox 闸门),REQ-ID 进入下游编码阶段(衔接路线图旗舰④追溯台账,后续 session diff / 提交都继续挂这个 ID)。

## 3. 卡片/窗口承载总览
- 一条需求 = 一张**主需求卡**(常驻、可最小化进卡坞,沿用现有 history/dock),卡头永远显示 REQ-ID + 当前 stage 进度条(六段)。
- 每个阶段的重产出(改动点/问题集/设计)以**子窗口/可展开面板**呈现,但都从主需求卡进入,不散落。
- 闸口以现有 approve 控件形态(go/auto/no 三按钮 + 富文本待审区)内联在卡里。
- 跨阶段的子 Agent 工作 = 现有 workflow.html 的 DAG 卡,完成后把产物"收"回主需求卡对应阶段。

**信贷特有考量**
- 触账强一致是头号闸口:凡改动点命中'计息/计提/放款/还款/账务/总账'(从结构化需求的'是否触账'标记驱动),G1 不可 auto 放行,且阶段五设计必须含'账务处理时序+借贷方向+冲正/红冲方案+对账影响+跑批时点',阶段四自动生成硬问题'本次变更是否需要调整科目/记账规则/计提模型?口径以哪个监管文为准?'
- 外系统契约必须显式建模:把人行/百行征信、核心银行(账务/客户)、支付清算、反洗钱与制裁名单筛查、影像签章、决策/规则引擎、统一身份权限、监管报送(EAST/1104)做成适配层登记表;改动点命中某适配层 → 阶段四自动追问该外系统的'报文版本/字段/超时与重试/幂等/对账与冲正/灰度切换'(尤其征信查询的授权合规与留痕、反洗钱名单命中后的拦截分支)
- 存量数据与跑批是隐形大坑:信贷有台账/流水/对账大表、分库分表、历史数据。改动点命中表结构/枚举/状态机变更 → 影响半径矩阵标'影响存量',阶段五强制产出数据迁移脚本影响评估 + 跑批(批处理时点/重跑/补数)影响,阶段四追问'存量数据如何兼容?是否需要刷数?刷数窗口与回滚?'
- 审批工作流的并签/会签/退回是设计高频遗漏点:涉及审批流的需求,改动点要覆盖'多级审批矩阵/退回重提/并签会签/超时升级/代办授权',阶段四追问授权矩阵与权限点(对接统一身份权限系统),避免只改正向路径漏掉退回路径
- 五级/风险分类、贷后预警、担保抵押质押链:这类需求牵动分类规则引擎与担保物价值重估,改动点需关联决策/规则引擎适配层,设计含规则版本管理与回溯重算影响
- 合规三铁律内建到管线:数据不出网(所有子 Agent 走内网 serve,问题外发经 outbox 人控)、操作可审计(每个闸口决策 by/at/comment 落账本)、变更可回溯(REQ-ID 串需求↔改动点↔设计↔评审↔后续 diff/提交)

- G1 范围与影响确认(IMPACTED出口):审改动点清单+影响半径矩阵,勾误报/补漏报/定风险分级 —— 最贵闸口,触账/外系统/存量类需求强制不可 auto
- G2 需求问题外发确认(CLARIFYING出口):审要外发给业务/产品的问题集,经 outbox 延迟闸门 mail_reply 挂回原 thread —— 数据/语义出网前的人控点
- G3 设计评审签发(IN_REVIEW出口):评审人逐条批注(必改/建议/可忽略)、签发定版、回写工单/邮件 —— 进入下游编码的总闸

**风险**
- 领域大脑缺失会退化成泛泛分析:若 credit_domain_lens 知识库不实(枚举/触账判定/外系统模板),阶段二三四都靠模型现编术语,完备性崩塌 —— 必须先和业务沉淀域对象/流程/外系统清单,这是管线立得住的前提,不能省
- 改动点漏报比误报更危险:跨双仓+存储过程+跑批的隐藏依赖一旦漏扫(尤其账务和存量),下游设计带病推进;G1 闸口必须给人足够证据(blame责任人+置信度)去补漏,且对'低置信度/账务相关'落点默认展开不折叠
- 状态机回退与并发:同一需求多人/多卡操作、闸口退回与自动流转竞争,需账本层乐观锁(updatedAt 校验)+ 单向推进函数纯逻辑可单测,避免 stage 错乱
- auto 模式滥用绕过风险闸口:必须硬编码'高风险标记需求'禁用 auto,否则触账变更被一路放行就违背金融级管控
- 问题外发的合规边界:阶段四问题集可能含敏感业务语义,outbox 人审是底线;答复回灌要按 messageId/thread 精确匹配,错挂会污染 REQ-ID 追溯链
- REQ-ID 注入的上下文膨胀:每会话注入需求上下文,长需求可能撑大首条 prompt;需像 loadProjectContext 一样做长度截断与摘要,只注入结构化要点不灌原文全文

### 信贷领域知识底座 (Credit-Domain Knowledge Foundation)

**目标**:让 BocomHermes 在信贷系统「需求分析→改动点→需求问题生成→详细设计」时可信地使用业务知识:把术语表/产品参数/业务规则/监管口径建成可溯源知识库,对现有代码+设计文档+数据字典做索引/RAG,并构建信贷领域本体(域对象与关系)。铁律:任何业务断言必须可溯源(命中文档/代码/数据字典三类证据之一),不能溯源或证据冲突的一律降级为「需求问题」,绝不编造业务规则。整套能力以新 MCP 挂到现有底座,复用 evidence-bundle 的 ref 寻址、scan_impact 的 SUSPICIOUS 强制校验、memory.md 首条注入三套既有机制,数据全程 127.0.0.1 不出网。

分四层,每层都对齐底座既有机制,落点可在 /Users/zhangang/Desktop/个人智能体/Person-Agent/BocomHermes-shell grep 验证。

L0 语料采集与离线索引 (onboarding 建库 + 增量):信贷知识四类源,各建索引、各带固定证据类型。(1) 代码源:前后端双仓(复用 orchestrator.js 的双 serve + mcp/git-mcp.mjs)。重点抽取信贷里「真相常在 DB 而非应用层」的部分——存储过程/SQL(计息、计提、还款计划生成、五级分类跑批常在 PROC 内)、规则引擎调用点、外系统适配器(征信/核心/反洗钱 client)、枚举与常量(产品代码、币种、还款方式、担保类型)、字段校验注解。(2) 文档源:设计/需求/接口文档、监管报送口径(EAST/1104 指标定义),按标题切块,保留「文件:章节:行号/页码」锚点。(3) 数据字典源:表/字段注释、码值表(信贷海量「01=正常 02=关注」五级分类码、状态机码)、约束、分库分表规则、台账主键——最权威的字段级真相,单独建 dict-mcp。(4) 监管/制度源:内规、监管文件原文(只读、带条款号)。索引技术:本地向量库(sqlite-vss 或本地 embedding,模型走内网推理端,全程 127.0.0.1)+ 关键词/符号倒排(git grep 已验证可用)+ 结构化抽取产物(码值表、字段表、PROC 签名)。每个 chunk 落盘 userData/knowledge/<sourceId>/,分配稳定 ref——沿用 repro-mcp.mjs 的「ref#bundleId/name」寻址法,改为 kref#<sourceId>/<chunkId>。主上下文只放约 5KB 摘要,细节 agent 按 ref 拉(完全照搬 repro-mcp 顶部「摘要进上下文、细节按需 call」的省 token 设计)。

L1 信贷领域本体 (域对象图,知识骨架):建可机读本体 ontology.json + 反向索引。节点=域对象(客户/对手方、授信额度/敞口/限额、申请受理、尽职调查、审批流、合同、担保抵押质押、放款、还款计划、计息计提、账务/总账、逾期催收、五级/风险分类、贷后、预警、监管报送);边=关系(「一个授信下挂多笔合同」「合同关联还款计划」「放款写核心账务」「风险分类驱动计提」)。关键:本体每个节点/边都不是凭空写的——grounding 字段挂一组 evidence ref(哪段代码/哪张表/哪份文档支撑这条关系);无 grounding 的关系标 unverified=true,检索时强制连同 unverified 标记一起返回。本体在 onboarding 时半自动生成(LLM 读语料提议 + 人确认,见 humanGates),之后随代码变更增量校正。本体同时是「改动点定位」导航图:一个业务概念→关联域对象→关联表/PROC/接口/落点,直接喂给现有「跨双仓改动点定位/影响面雷达」。

L2 可信检索与「业务断言溯源」协议 (反编造核心):对 agent 只暴露一个高阶工具 credit_lookup(question, intent),返回结构强约束为「断言+证据+置信度+冲突」。(a) 每条 assertion 必带 evidence:[kref...](至少 1 条;规则类断言要求「代码 ref 与 文档/字典 ref 同时命中」才算 high 置信)。(b) confidence 取 high/medium/low/none:none/low 不允许作为设计结论使用,工具直接把它转成一条「待澄清需求问题」草稿。(c) conflict:当不同源给出矛盾口径(典型:文档写「逾期 90 天进次级」、代码 PROC 写 80 天),不做仲裁,原样并列两条证据并标 conflict=true,强制升级为需求问题。准则:问「贷后逾期口径」先查字典(状态机码)+代码(PROC)+文档,三者一致才 high。

L3 强制校验与底座接入:(a) 新设生产工具 design-mcp.assert_sourced,校验详设/需求文档里每条业务断言都挂了 kref,缺失就标 SUSPICIOUS(照搬 repro-mcp 的 scan_impact「改了没扫就标 SUSPICIOUS」验证闸口)。(b) 生成需求问题环节,把 low/none/conflict 自动熔成问题卡(携候选证据供人拍板)。(c) 领域本体/术语表摘要照 src/session.js 的 loadProjectContext 机制,以 <信贷领域知识> 包装注入首条消息。(d) orchestrator.js 的 ROLE_PROMPTS 增 domain-librarian 角色,专责 lookup 与冲突裁决草拟。

**信贷特有考量**
- 业务规则真相常藏在存储过程/跑批 SQL(计息、计提、还款计划生成、五级分类迁徙)而非应用层——必须专门建 PROC/SQL 索引并抽规则,否则核心规则成知识盲区

- 生成的需求问题卡由人选择「发给业务澄清」还是「已明确」,系统不自动用猜测答案堵上空缺

**风险**
- 表/PROC/文档体量大且变更频繁→索引陈旧误导;靠 git 增量重建 + chunk hash 失效检测缓解
- 存储过程逻辑抽取难(动态 SQL/临时表/游标),可能抽不准→宁可降 high、转需求问题也不猜
- 本地 embedding 对中文信贷术语召回差→补术语表同义词 + 符号倒排双路召回
- 人工确认的加载量决定知识质量,业务不配合则大量 unverified;需把确认嵌入日常改动闸口以降低负担
- 冲突检测漏报(同义不同表述、跨表口径)→借领域本体的同义节点辅助比对,并对账务/监管类强制双源核对

### 改动点与影响分析(跨代码)— 跨双仓「影响面雷达 Blast Radius」+ 信贷专用影响传播

**目标**:给定一条信贷需求(已结构化为 REQ 卡/改动意图),在前后端双仓自动定位「落点」(要改的文件:行)并算出「影响面」(谁会被牵动:调用链上下游、被引用方、配置、批处理/跑批、审批流引擎节点、账务入口),输出一份带 文件:行 可点跳、按风险分级、并标注「该问谁(blame 责任人)」的改动点清单,作为编码/评审/测试缺口/发布评估的统一输入。把「拉群问老人、漏一个就线上事故」变成一张可追溯、可复核的清单。

## 总体形态:一次「雷达扫描」= 一个 orchestrate 工作流(复用 src/orch.js 的 DAG + onBeforeBatch 人审闸口)

输入: REQ 卡(来自需求分析 facet,含改动意图 + 锚点种子:接口路径/DTO 字段/方法名/表名/枚举/审批环节名)。输出: 一份 BlastReport(见 artifacts)。全程跑在内嵌的双 serve 上,代码不出网。

### 阶段 0 · 锚点归一(Seed Resolution)
把自然语言改动意图熔成「机器可扫描的符号集」。种子有 6 类:① REST 路径/Controller 方法 ② DTO/VO/Entity 字段名 ③ Java/服务方法名 ④ 数据库表名/列名 ⑤ 枚举值/常量(状态码、产品码、五级分类码) ⑥ 审批流环节/账务分录科目(信贷特有,见 creditSpecifics)。由一个 analyst 角色子任务读 REQ 卡 + 抽样读两仓的命名风格(grep 几个已知接口),把模糊词("放款状态")解析成精确符号("LoanStatus 枚举 / t_loan.loan_status 列 / DisburseController.disburse")。这一步必出「锚点解析表」,人审第一道闸:确认锚点选对了再扫,避免在错符号上烧 token。

### 阶段 1 · 双仓并行符号扫描(核心,需新建 scan_impact MCP)
对每个锚点,在前端仓 + 后端仓**各起一个独立 serve**(opencode.js ensureServe,第二仓用 tryShare:false 强制独立 serve,因为 cwd 不同),并行 fan-out。扫描分层,每层用 git grep / ripgrep 跑出 文件:行:
- **后端层**: 方法定义→被调用点(直接 caller)→MyBatis/JPA Mapper XML 引用→该列在 SQL/存储过程里的读写点→Service 接口实现类→事件/MQ 发布订阅点→定时任务注解(@Scheduled/XXL-JOB/quartz)→配置项(application.yml/nacos/apollo key)。
- **前端层**: API 调用封装(axios/fetch 的 url 常量)→调用该 API 的组件/页面→表单字段绑定(v-model/Form.Item name)→枚举字典渲染点(下拉/标签/状态机渲染)→路由与权限点(菜单/按钮权限码)。
- **契约横切**: 同一字段在前端请求体 vs 后端 DTO 两侧的 文件:行 配对(复用契约对账 lens 思路),识别「只改了一侧」的漏改风险。
扫描结果是「锚点 → 命中点列表(文件:行 + 命中代码片段 + 所在层)」的原始矩阵。

### 阶段 2 · 传播闭包(Transitive Closure,有界 BFS)
仅靠一跳 grep 会漏掉「调用链深处的账务/跑批」。对**高危层**(账务入口、审批流引擎、跑批作业)做有界向上回溯:从命中的 Service 方法继续 grep 它的 caller,最多 N 跳(默认 3,可配),直到触达「边界标记」:① Controller(对外接口边界)② @Scheduled/Job(跑批边界)③ MQ 消费者(异步边界)④ 账务过账方法(资金边界)。每跳记录路径,产出「调用链路径串」。BFS 用 maxTasks/maxRounds 预算护栏防爆炸(orchestrate 已有)。对存储过程/SQL 大表:扫到列名命中存储过程时,标记为「人工核查项」而非自动展开(过程体逻辑机器读不可靠,见 risks)。

### 阶段 3 · 多 Agent 分诊与风险定级
复用 window.js 的难度分诊 + 角色 fan-out。每个命中聚合点交给角色化子 Agent 评估:
- biz(业务·逻辑): 这个命中点是否触达信贷关键不变量(额度不能超敞口、还款计划总额守恒、五级分类只进不出规则等)。
- security/合规: 是否触达征信查询授权、制裁名单筛查、数据权限(数据不出网/字段级脱敏)。
- arch: 是否跨越账务强一致边界、是否在事务/跑批临界区。
分诊器(沿用 window.js triage:输出 {difficulty,layers,strategy})决定单点直接定级 vs 上多 Agent 对抗。风险分级 P0/P1/P2/P3 由规则 + 模型共同给:命中账务过账/审批流状态机/跑批主表 = P0(必须人工 + 回归);命中契约一侧漏改 = P1;命中纯前端展示/文案 = P3。

### 阶段 4 · blame 责任归属 + 报告归并
对每个 P0/P1 命中点,调 git-mcp git_blame(文件 + from_line/to_line)拿「最后改这行的人 + 提交」,git_log 拿该文件近期 owner,产出「该问谁」。reduce 阶段把双仓所有命中、调用链、契约缺口、风险级、责任人归并成 BlastReport,挂到 REQ-ID 下(复用追溯台账),写入 session diff 可追溯。报告每条都是 文件:行 可点跳(复用富结果跳转)。

### 数据流
REQ卡 → [analyst 锚点解析(人审①)] → 双 serve 并行 scan_impact(前/后端 fan-out) → 命中矩阵 → 有界 BFS 闭包 → [多 Agent 分诊定级] → git_blame 责任 → reduce 归并 BlastReport → 挂 REQ-ID + 人审② 复核盲区 → 交付给编码/测试缺口/发布评估下游。

### 与现有底座的精确接驳点
- 编排: src/orch.js run-workflow / orchestrator.js orchestrate,新增 role: 'impact-scanner' / 'closure' / 'risk-grader' 到 ROLE_PROMPTS。
- 双仓: opencode.js ensureServe(backendDir, {tryShare:false}) 起第二独立 serve。
- blame: mcp/git-mcp.mjs 现成 git_blame/git_log/git_show。
- 人审: orch.js onBeforeBatch 已是「每批计划先发卡等批准」,锚点解析批 + 盲区复核批天然落在这上面。

**信贷特有考量**
- 审批流引擎是头号隐性影响源: 信贷审批多为「状态机 + 规则引擎(Drools/自研)+ 流程定义(BPMN/工作流表 t_wf_node/t_wf_route)」三层。改一个字段(如新增「风险敞口」)往往要同步: 流程变量定义、节点准入条件、退回/会签/并签的分支条件、审批表单字段权限矩阵。这些常配在数据库流程定义表或规则文件里,纯 grep 代码扫不到 → 必须把「流程定义表行 + 规则文件」也纳入锚点扫描层,作为 P0 命中。
- 账务入口是资金安全红线: 任何触达「过账/分录生成/计息计提/还款销账」的命中点一律 P0,且需识别它是否在数据库事务 + 与核心银行系统的账务接口(放款记账、还款扣款)耦合。账务讲强一致,改动若影响分录借贷平衡或台账流水生成,必须人工 + 全回归,机器只标记不自动判定安全。计息/计提改动还要算「历史数据重算」牵连(是否要跑批回算存量合同)。
- 跑批作业的牵连最易被漏: 信贷大量逻辑在夜间跑批(还款计划生成、计息、逾期认定、五级分类迁徙、计提、对账、监管报送 EAST/1104 抽数)。一个字段/列改动可能让某个 Job 的 SQL 抽数口径变化却无编译错误。scan_impact 必须专门扫: @Scheduled/XXL-JOB/quartz 注解、Job 调度配置表、批处理 SQL/存储过程里对该列的引用,并对「跑批主表大表」命中升级风险(改了跑批读写口径=可能影响全量存量数据)。
- 外系统调用边界要单独标注影响: 命中点若落在征信(人行/百行)、核心银行(账务/客户)、支付清算、反洗钱/制裁名单筛查、影像签章、决策/规则引擎、监管报送适配器上,要标「外部契约影响」——改字段可能要同步报文映射、字段长度/类型对齐、二次授权(征信查询需客户授权留痕)。这类命中机器只能定位调用点,报文规范对齐必须人工 + 标为跨系统协调项。
- 数据库特征放大影响面: 台账/流水/对账大表、强一致账务、存储过程多、分库分表、历史数据跑批。列名改动要扫: Mapper XML、存储过程体(标记人工核查,过程逻辑机器读不可靠)、分库分表路由键(若改的是 sharding key 则 P0 全局影响)、历史数据兼容(存量行的该列默认值/回刷)。监管报送口径(EAST/1104)对字段定义敏感,命中报送抽数 SQL 一律 P0 + 拉合规。
- 合规可审计要求贯穿: 数据不出网(扫描全在内网 serve,blame/grep 不外发)、操作可审计(每次雷达扫描挂 REQ-ID、人审决策留痕)、变更可回溯(BlastReport 入 session diff)。命中数据权限/字段级脱敏/敏感信息(身份证、征信、账户)读写点要单列「合规影响」分支。

**关键产物 / 数据结构**
- 锚点解析表 AnchorMap: {seedText, resolvedSymbols:[{kind:rest_path|dto_field|method|table_col|enum|wf_node|gl_subject, symbol, repo:frontend|backend, confidence}], 人审状态}
- 命中矩阵 HitMatrix: 每条 {anchor, repo, layer:backend_caller|mapper_xml|sql_proc|scheduled_job|config|fe_api|fe_component|fe_dict|contract_pair, file, lineStart, lineEnd, snippet}
- 调用链闭包 CallChains: {from:命中点, path:[file:line→file:line…], boundary:controller|job|mq|gl_posting, hops}
- 契约缺口表 ContractGaps: {field, frontend:file:line, backend:file:line, mismatch:missing_one_side|type|required|enum|auth_header}
- BlastReport(主交付): {reqId, anchors, hits:[{file:line(可点跳), layer, riskLevel:P0|P1|P2|P3, reason, creditTag:审批流|账务|跑批|外系统|合规|无, owner:{name,commit}(blame), needHuman:bool, needRegression:bool}], blindspots:[人工核查项: 存储过程/流程定义表/报文映射], summary:{p0Count,p1Count,跨系统协调项,存量回算项}}
- 盲区清单 Blindspots: 机器不可靠区域显式列出(存储过程体、规则引擎配置、外部报文规范),强制人审而非静默漏掉

**需要的工具能力**
- 新建 scan_impact MCP (扩展 mcp/ 下,参照 git-mcp.mjs 零依赖 stdio 风格): 输入 {symbol, kinds[], repoDir, layers[]},底层 git grep -n / ripgrep + 语言感知正则(Java 方法引用、MyBatis XML、Vue/TS API 常量、yaml/nacos key、@Scheduled/XXL-JOB 注解),输出 文件:行+片段+所在层。是路线图里点名的能力,当前仓内尚无,需补建。
- scan_impact 增信贷专用 lens: ① wf-lens 扫审批流(流程定义表 DDL/SQL、规则文件、节点条件)② gl-lens 扫账务入口(过账/分录/计息方法名模式 + 科目表)③ batch-lens 扫跑批(调度注解 + Job 配置表 + 批处理 SQL)④ ext-lens 扫外系统适配器(征信/核心/AML 客户端类与报文映射)。
- 扩展 ROLE_PROMPTS (orchestrator.js): 新增 impact-scanner / closure-bfs / risk-grader / credit-domain 角色引导语,让 fan-out 子 Agent 各司其职。
- 扩展 orch.js run-workflow: 增加「双 serve 并行」入口——同时 ensureServe(frontendDir) + ensureServe(backendDir,{tryShare:false}),把 repo 维度作为第一层 fan-out。
- 复用 git-mcp.mjs (git_blame/git_log/git_show) 做责任归属,无需改造;复用富结果 文件:行 跳转渲染 BlastReport;复用 REQ-ID 追溯台账把报告钉死到需求。
- risk-rules 配置文件: 把「账务/审批流/跑批主表/sharding key/报送 SQL = P0」这类信贷定级规则做成可维护规则表(不写死在代码),供 risk-grader 与模型共同裁定。
- (可选)符号索引缓存: 大仓 grep 慢,可对方法定义/引用做一次性索引落本地缓存,增量更新,提速重复扫描。

**人审闸口**
- 闸口① 锚点解析确认: 扫描前,人审 AnchorMap——确认「放款状态」解析成的符号集正确、没扫错列/方法,避免在错锚点上扩散。落在 orch.js onBeforeBatch 第一批。
- 闸口② P0 命中复核 + 盲区签字: 报告生成后,所有 P0(账务/审批流/跑批主表/sharding/报送)命中与 Blindspots(存储过程、规则配置、外部报文)必须人工逐条复核确认,机器只定位不拍板资金/合规安全。
- 闸口③ 跨系统协调项指派: 命中外系统契约的项,人工确认是否需联动征信/核心/AML/报送团队,并决定是否升级为跨团队协调(可触发邮件 MCP 起草协调信,经发件箱闸门发出)。
- 闸口④ 存量数据回算决策: 命中计息/计提/历史列改动时,人工拍板是否需要跑批回刷存量合同,机器给出受影响台账规模估算供决策。

**风险**
- 漏扫(假阴性)是最致命风险: 动态调用(反射/SpEL/规则引擎里的字段名字符串)、存储过程内引用、配置/流程定义表里的字段,grep 都可能扫不到。缓解: 显式 Blindspots 清单 + 把流程定义表/规则文件/proc 纳入扫描源 + 强制人审,绝不让「扫不到」等于「没影响」。
- 过扫(假阳性)淹没信号: 同名方法/通用字段名(status、amount)会命中海量无关点。缓解: kind 限定 + 包路径/模块过滤 + 风险分级把噪音压到 P3 + 多 Agent 二次判定相关性。
- 调用链 BFS 爆炸或漏边界: 深链回溯可能超预算或在错误边界停止。缓解: orchestrate 的 maxRounds/maxTasks/超时护栏 + 显式边界标记(Controller/Job/MQ/过账),触界即停。
- 存储过程/规则引擎逻辑机器读不可靠: 强行让模型解析过程体或 Drools 规则易给错结论。缓解: 命中即标「人工核查项」,只定位不判定,不进自动定级。
- 大仓性能: 双仓全量 grep + 多跳 BFS 在百万行台账系统上慢。缓解: 符号索引缓存 + 增量 + 并发上限,首扫慢但可接受,重复扫快。
- 锚点解析错误会让全链失真: 种子解析错则后续全错。缓解: 闸口①强制人审 + confidence 低的锚点高亮 + 抽样验证(grep 一个已知正确点对照)。
- 信贷不变量未被识别: 模型未必知道「额度不能超敞口、还款计划总额守恒」这类领域硬约束。缓解: credit-domain 角色注入领域不变量清单作为定级依据,关键不变量命中一律 P0。

### 数据库维度影响分析(信贷管理系统)——DB 变更影响引擎

**目标**:在 BocomHermes 现有底座(scan_impact 符号引用扫描 + git-mcp blame/log + 难度分诊多 Agent fan-out + 证据包/断言/便签 + memory.md 跨会话注入 + 每批人审闸口 + 数据不出网)之上,新增一个"DB 影响引擎":把信贷库的数据字典/DDL/存储过程/跑批脚本摄取成可检索的离线索引,接到上游需求改动点后,自动从代码符号一路血缘下钻到 表/字段/索引/外键/视图/存储过程,产出一份带"文件:行 + 表:字段"双向证据锚点、风险分级、可回滚的【DB 变更清单 + 迁移与回滚草案】;并对账务平衡、台账/流水大表在线 DDL、分库分表、跑批对账这四类信贷高危场景给出专项评估与人工闸口。核心目标:把"漏改一个存储过程 / 算错一张台账 / 在线 DDL 锁住放款流水大表"这类只躺在 DBA 脑子里的隐性风险,变成有证据、可追溯、可回滚的清单。

分四层,全部本地化、数据不出网,复用现有 stdio MCP + scan_impact + 多 Agent + 人审闸口。

【第一层 · 摄取与离线索引(新建 db-catalog-mcp)】
信贷库不能连生产、DDL 几千张表、存储过程上万行,所以摄取走"离线快照"而非连库:
- 输入源(均可内网拿到、不出网):①数据字典 Excel/CSV(中文表名/字段名/业务含义/值域/是否台账流水);②建表 DDL / DBML / Liquibase / Flyway 脚本;③存储过程/函数/触发器/Package 源码(.sql/.prc/.pck);④跑批作业清单(调度 XML / SQL 脚本 / sqlldr 控制文件);⑤ORM 映射(MyBatis Mapper XML、JPA @Entity、Hibernate hbm)。
- 解析:轻量 SQL 方言分词器,覆盖信贷常见 Oracle/DB2/MySQL/达梦/GaussDB,抽出 表/列/类型/主键/外键/索引/分区键/视图依赖;对存储过程静态解析 INSERT/UPDATE/DELETE/SELECT/MERGE/CURSOR,得出每个 proc 读哪些表、写哪些表、调哪些 proc;对 MyBatis 从 namespace+resultMap+SQL 文本抽"Java 方法→SQL→表/列"。解析不了的(动态拼 SQL、EXECUTE IMMEDIATE、字符串拼列名)降级为"文本片段 + TODO 人工确认",绝不丢弃。
- 落盘成可检索索引(完全对齐 scan_impact 落 userData/scans 的做法,落 userData/db_index):schema 图(表/列/PK/FK/索引/分区,JSON)、对象→定义文件:行反查表、存储过程读写矩阵(proc→{reads,writes,calls})、代码↔表映射(列名/表名→Mapper/Entity 文件:行)、分库分表路由规则(shardingKey + 分片算法)。每份索引带快照版本号 + 来源 DDL 的 git ref,使任何一次影响分析都可被审计、可重放。

【第二层 · 双向血缘定位(新建 db_impact 工具,反向延伸 scan_impact)】
入口是上游 facet 给的【改动点】(改了哪些代码符号/接口/字段)。db_impact 做双向血缘:
- 正向(代码→DB):对每个改动的 ORM 方法/SQL,查代码↔表映射→命中表/列→顺 schema 图 BFS 扩散:该列被哪些 索引/外键/视图/存储过程/触发器/跑批 引用(标深度);跨对象用读写矩阵传播(改了 T 表结构→写 T 的存储过程→调它的跑批→它产出的对账中间表→监管报送视图)。
- 反向(DB→代码,漏改重灾区):若改动点本身是改字段/加表,直接复用 scan_impact 引擎对物理列名/表名/Mapper id 做 git grep,把后端 Mapper、动态 SQL 拼接、报表 SQL、存储过程里硬编码的列名全找出来。
- 每条影响输出:对象类型 + 名字 + 定义文件:行(可点跳) + 命中路径(从改动符号到该对象的血缘链) + 风险标签。无法静态判定的(动态 SQL / EXECUTE IMMEDIATE / 拼列名)单列"灰区清单"强制人工确认,绝不静默判"无影响"——信贷漏改的根因 90% 在这里。

【第三层 · 信贷四类高危专项评估(难度分诊→多 Agent fan-out)】
沿用现有启发式分诊:命中台账/流水大表、改了金额/利率/余额/计提字段、触及存储过程或对账逻辑→difficulty 拉到 4-5→触发多 Agent 对抗,各 lens 持一个假设并交叉反驳,结论写入证据包便签(confirmed/excluded/maybe,省 token 防重复):
- account-balance lens(账务平衡):识别改动是否触及"借贷必相等"的台账/分录/总账表,检查计息计提口径、精度/舍入(金额建议 DECIMAL 不可改 FLOAT)、对账平衡式是否被破坏;新增/改金额字段必须追问"历史回填取什么值、回填期间余额是否短暂不平"。
- big-table-ddl lens(大表在线 DDL):对放款/还款/流水/计息台账等亿级大表,判定 ADD COLUMN(可空默认值是否触发全表重写)、加索引(是否需 ONLINE/pt-osc/gh-ost/Oracle DBMS_REDEFINITION)、改类型(是否锁表),给出锁时长量级估计与避锁改写方案。
- sharding lens(分库分表):判断改动是否触及 shardingKey;改 shardingKey=灾难(全量重路由),要明确标红;新查询是否带分片键(否则全分片扫描);跨分片事务/全局唯一 ID/广播表一致性。
- batch-reconcile lens(跑批对账):改了表结构后,夜间跑批 / 日终对账 / 监管报送(EAST、1104)是否需同步改;字段含义变更是否破坏 T+1 对账平衡;存储过程改动是否影响计提批量与总账过账。

【第四层 · 产出物 + 人工闸口】
汇总成两份草案,经现有"每批人审闸口"放行:
- DB 变更清单:逐条 {对象, 变更类型(ADD/MODIFY/DROP/INDEX/PROC), 风险级(高/中/低), 证据锚点(文件:行 + 表:字段), 命中血缘链, 灰区待确认}。
- 迁移与回滚草案:在线 DDL 选型(直改 vs 影子表 vs 在线工具)、历史数据回填脚本草案(分批 + 限速 + 进度断点)、回滚脚本(每个正向 DDL 配逆向)、账务一致性校验 SQL(回填前后跑平衡式)、跑批/对账联动改清单。每条标"谁拍板":DBA 确认锁影响、账务负责人确认平衡、监管口确认报送。

**信贷特有考量**
- 台账/流水/计息大表(亿级)是核心约束:ADD COLUMN 带默认值在 Oracle 11g 会全表重写、MySQL 加索引需 gh-ost/pt-osc,引擎必须按表量级给锁时长量级,而不是泛泛说'可能慢'——放款流水大表锁 30 秒就是生产事故
- 账务强一致/借贷平衡:任何触及分录、总账、计提、计息台账的字段改动,必须自动生成'回填前后平衡式校验 SQL'(SUM(借)=SUM(贷)),金额字段类型从 DECIMAL 误改 FLOAT 要直接标红——这是信贷会计铁律
- 存储过程是信贷账务主战场且最易漏改:计息计提、日终过账、对账多藏在 PL/SQL 里,EXECUTE IMMEDIATE 和动态拼列名静态分析覆盖不到,必须显式产出'灰区清单'逼人工确认,不能假装查全了
- 分库分表路由:信贷按客户号/机构/合同号分片,改 shardingKey 等于全量数据重路由=灾难级,新增查询不带分片键会全分片扫描拖垮库,必须单独判定并标红
- 外系统账务边界:核心银行系统(账务/客户主数据)、支付清算、人行/百行征信、反洗钱制裁名单、监管报送(EAST/1104)。本系统字段改动若是这些外系统的镜像/落地字段,改了会导致对账不平或报送字段缺失——引擎要标注'此字段来源/去向外系统,改动需同步对方接口与对账'
- 监管报送与历史可回溯:EAST/1104 报送视图、五级/风险分类、贷后台账依赖稳定字段口径;字段语义变更要联动报送映射,且历史数据回填不能改变已报送口径,合规要求变更可回溯、操作可审计
- 跑批与对账时序:日终跑批、T+1 对账、计提批量有严格时序依赖,表结构改动若未同步跑批 SQL/sqlldr 控制文件,会在夜间批量静默失败或对账不平,白天才暴露——必须把跑批作业纳入血缘扩散

**关键产物 / 数据结构**
- db_index/ 离线索引(落 userData/db_index,对齐 scan_impact 落盘模式):schema 图(表/列/PK/FK/索引/分区 JSON)+ 对象→定义文件:行反查表 + 存储过程读写矩阵 proc→{reads,writes,calls} + 代码↔表映射(列/表→Mapper/Entity 文件:行)+ 分库分表路由规则;每份带快照版本号 + 来源 DDL 的 git ref
- DBImpactReport(变更影响清单):每条 {对象类型, 名字, 变更类型 ADD/MODIFY/DROP/INDEX/PROC, 风险级 高/中/低, 双向证据锚点(代码 文件:行 + 库 表:字段), 命中血缘链(从改动符号到该对象), 是否灰区待确认}
- MigrationDraft(迁移与回滚草案):在线 DDL 选型 + 分批限速回填脚本草案(带进度断点)+ 逐条逆向回滚脚本 + 账务平衡校验 SQL(回填前后)+ 跑批/对账/报送联动改清单 + 每条的人工拍板归属(DBA/账务/监管)
- GrayList(灰区清单):动态 SQL / EXECUTE IMMEDIATE / 拼列名等静态分析判不准的点,带文本片段定位,强制人工确认
- DB 影响指纹(可选,接现有事故记忆库):本次表/字段/proc 影响图蒸馏成指纹,下次同表改动命中即弹'上次改这张台账漏了对账批 X',修一次永久免疫

**需要的工具能力**
- db-catalog-mcp(新建 stdio MCP,与 git-mcp/repro-mcp 同构):catalog_ingest(摄取字典/DDL/proc/Mapper 落 db_index)、catalog_schema(查表/列/FK/索引/分区)、catalog_proc_io(查存储过程读写矩阵)、catalog_code_to_table(代码↔表反查)、catalog_sharding(查分片路由规则)、catalog_status(索引版本/覆盖率/解析失败统计)
- db_impact 工具(逻辑上扩 scan_impact 或并列):入参=改动点(符号/字段/接口)+ cwd,做正向(代码→DB BFS 扩散)与反向(列名 git grep 复用 scan_impact 引擎)双向血缘,输出 DBImpactReport + GrayList,落 userData/db_scans 备查并被验证流程检查(改了 DB 相关代码却没跑 db_impact→标 SUSPICIOUS,完全对齐现有 scan_impact 的 SUSPICIOUS 机制)
- 四个评估 lens 接入现有难度分诊/fan-out:account-balance、big-table-ddl、sharding、batch-reconcile,各持一假设并行查证,结论写入证据包便签(confirmed/excluded/maybe)
- memory.md 注入:把'本库台账/流水大表清单、分片键约定、账务平衡式、外系统镜像字段映射、上次漏改教训'作为项目事实跨会话注入,避免每次重学
- draft 渲染:DBImpactReport + MigrationDraft 在工作流卡内渲染为可点跳清单(对象 文件:行 / 表:字段),经现有 onBeforeBatch 人审闸口放行;迁移/回滚 SQL 仅生成草案不自动执行(信贷数据不出网、不直连生产铁律)

**人审闸口**
- DBA 闸口:在线 DDL 选型与锁影响估计(尤其放款/还款/计息台账等大表的 ADD COLUMN / 加索引 / 改类型)必须 DBA 确认锁时长量级与执行窗口,引擎只给草案不自动执行
- 账务负责人闸口:任何触及分录/总账/计提/利率/余额/金额精度的改动,需账务负责人确认平衡式不被破坏、历史回填取值与回填期间是否短暂不平
- 分片闸口:涉及 shardingKey 或新查询不带分片键的改动,标红并要求架构/DBA 显式拍板(改分片键=全量重路由,默认拦截)
- 灰区确认闸口:动态 SQL/EXECUTE IMMEDIATE/拼列名清单逐条人工确认是否真受影响,确认前不得判'无影响'
- 监管报送闸口:字段语义变更触及 EAST/1104/五级分类报送映射时,监管/报送口确认报送字段与历史口径不被破坏
- 外系统对账闸口:改动字段是核心系统/支付清算/征信/反洗钱的镜像或落地字段时,需对方接口负责人确认同步并跑跨系统对账
- 回滚草案签核:每条正向 DDL 必须配通过评审的逆向回滚脚本,缺回滚脚本的变更不放行

**风险**
- 静态分析天然不全:动态拼 SQL、EXECUTE IMMEDIATE、ORM 二级缓存、存储过程里字符串拼列名会漏血缘——缓解:强制灰区清单 + 默认保守判'可能受影响'而非'无影响',并用 memory.md 积累漏改教训
- 离线索引会过期:DDL/proc 快照与生产实际漂移会导致误判——缓解:索引绑快照版本号 + 来源 git ref,catalog_status 暴露覆盖率与解析失败率,过旧索引在报告顶部告警
- 锁时长/数据量只能估不能准:不连生产就拿不到真实行数与统计信息,锁影响是量级估计而非精确值——缓解:明确标'量级估计,需 DBA 用生产统计信息复核',不让 Agent 给出确定性承诺
- 回填脚本风险高:大表回填可能拖垮库或破坏账务平衡——缓解:只产草案、分批限速、带回填前后平衡校验 SQL,执行权始终在 DBA,Agent 不直连生产(数据不出网铁律)
- 信贷领域知识依赖配置:台账/流水大表清单、分片键、平衡式、外系统镜像字段需要团队预先在 memory.md/字典里标注,冷启动覆盖不足——缓解:首次摄取产出'待人工标注清单',随使用迭代沉淀
- SQL 方言差异:Oracle/DB2/达梦/GaussDB 语法差异导致解析器覆盖不全——缓解:解析失败降级为文本片段+TODO,按方言增量补解析规则,不阻断主流程

### 外系统调用维度 — 发现、契约影响分析与时序设计（信贷管理系统）

**目标**:在『需求→改动点→需求问题→详细设计』第一道大关里，把『这次改动会碰哪些外系统、契约怎么变、幂等/重试/超时/降级/对账/时序怎么设计、怎么挡板联调』做成有代码证据、可追溯、可评审的产物。让 Agent 在不实际外呼的前提下（代码不出网），从前后端双仓代码+配置+接口清单识别出所有外系统调用点（征信/核心/支付/反洗钱/影像签章/决策引擎等），交叉验证契约（报文/字段/鉴权）的变更影响，产出『外系统交互清单 + 契约变更影响 + 时序图』，并把最易被漏掉的『账务强一致/时序窗口/对账口径/降级回退』显性化为需求问题清单交人拍板。

分四层，全部建立在现有底座上（scan_impact 的 git-grep 引擎、httpcap 抓包、git-mcp blame/log/show、orchestrator 角色 fan-out、bundle_note 共享便签、mail-mcp 发件箱闸门、memory.md、代码不出网约束）。

【第一层·识别：从双仓代码+配置静态发现外系统调用点】
不靠单一关键字，靠多信号联合定位。新增工具 ext_scan_integrations，复用 scan_impact 同款 git-grep 引擎做『外系统调用点指纹库』匹配，信号叠加打分：(a) 客户端类型/Bean 名：*Client/*Adapter/*Gateway/*Facade/*Feign/*RpcService/*ApiInvoker；(b) 配置键：application.yml/properties/nacos/apollo 里的 url/endpoint/host/wsdl/appId/secret/cert/keystore/queue/topic，@FeignClient(name/url)、Dubbo @Reference、WebServiceTemplate、RestTemplate/WebClient 的 baseUrl 注入点；(c) 报文/契约物料：*.wsdl/*.xsd/*.proto/*.json-schema、ESB 服务总线服务码（信贷常见 8 位服务号/交易码）、定长报文模板、ISO8583/ISO20022 域定义、银联/网联报文域；(d) 中间件出向：MQ（IBM MQ/Kafka/RocketMQ）producer topic、定时跑批 job（quartz/xxl-job）里发起的外呼（批量征信、批量代扣、日终对账文件 SFTP）；(e) 数据库出向：存储过程里的 dblink/异库同义词/接口表写入——很多信贷『外系统调用』其实是写接口表+核心跑批捞，这是最容易漏的一类，Agent 必须把『接口表/中间表写入』也当外系统边界；(f) 文件/影像：SFTP/FTP 路径、影像平台上传 SDK、电子签章 API。每个命中归一化成一条『外系统调用点』记录，用内置映射表+域线索自动归类到信贷外系统目录（征信-人行/百行、核心-账务/客户、支付清算-银联网联/超网、反洗钱-名单筛查/可疑交易、影像、电子签章、决策/规则引擎、统一身份权限、消息通知、监管报送 EAST/1104）。标注调用方向与触发语义：同步 RPC / 异步 MQ / 文件批量 / 回调入向（外系统反向通知，如签约结果回调、代扣结果异步通知、征信异步报文）。入向回调点（@PostMapping 的 notify/callback/return/result 控制器）同样要扫——契约变更对回调的影响最易被忽略。最后用 scan_impact 收口：对每个调用点跑『本次改动符号→调用点』的可达性，只保留『本次改动真正触达的外系统』，输出触达子集+每个的触达路径（文件:行链），避免把整库外系统都列出来淹没人。

【第二层·契约变更影响：报文/字段/鉴权三维交叉对账】
复用并扩展旗舰②契约对账引擎，对象从『前后端两仓』换成『我方调用端 vs 外系统契约物料』。对每个触达的外系统建三张差异表：(a) 报文结构差异：解析 wsdl/xsd/proto/json-schema/定长模板，用 git_show 取改动前版本 vs 工作区对比字段集合——新增/删除/类型变更/必填可选翻转/长度精度变更/枚举码值增减/嵌套层级变化。金额字段专项标记（精度/单位：分还是元、有无符号、定长右对齐补零）、日期格式（YYYYMMDD vs 带时分秒）、字符集（GBK/UTF-8，信贷与核心常 GBK，跨系统乱码高发）。(b) 字段语义/映射差异：本次新增的业务字段是否需要透传给外系统（如新增授信品种码、担保方式码，核心账务/征信报文是否要带、码值映射表是否已登记），以及外系统返回字段我方是否新消费、落库口径是否对齐。(c) 鉴权/通道差异：报文头里的渠道号/机构号/操作员/MAC/验签字段、证书/密钥、加签验签算法、ESB 路由头、双向 TLS。任何鉴权字段动了都标红——这是联调一调就废且最难自查的。每条差异落『契约变更影响』记录：变更类型、兼容性判定（向后兼容/破坏性/需对端同步改造）、需不需要外系统配合改、需不需要走对端发版窗口。破坏性变更（删字段、改类型、必填收紧、码值含义变）一律升级为人工闸口必答的需求问题。

【第三层·非功能契约：幂等/重试/超时/降级/对账/时序的设计要点抽取与缺口检查】
这是信贷外呼最容易在需求阶段漏、上线后炸的部分。对每个触达外系统，Agent 从代码读现状并对照清单逐项体检，缺什么就生成需求问题：(1) 幂等：写类外呼（放款、代扣、记账、额度占用/释放）是否带幂等键（业务流水号/请求流水），外系统是否按幂等键去重，重试会不会重复记账/重复扣款——账务强一致下这是头号雷。(2) 重试与退避：超时/网络抖动时是否重试、重试次数与退避、是否区分『可重试（超时、5xx）』与『不可重试（业务拒绝、参数错）』，避免对不可重试错误盲目重试放大故障。(3) 超时：连接/读超时配置、是否短于上游（如审批工作流）整体超时、是否会把同步调用挂死在审批事务里。(4) 降级与回退：外系统不可用时是降级（如征信不可用走人工/缓存）还是阻断，降级开关在哪、降级后业务状态机如何流转、恢复后如何补偿。(5) 对账：异步/文件类外呼是否有 T+1 对账（代扣结果、放款结果、核心记账），本端流水表与外系统对账文件的勾对口径、长短款处理、对不平的差错处理流程；本次改动是否动了对账关键字段（金额、流水号、状态码映射）导致对账口径变。(6) 时序与最终一致：跨外系统的调用顺序（先核心占额度还是先放款、先记账还是先通知），失败时的补偿/冲正（红冲、反向交易）、TCC/SAGA/本地消息表/事务消息的现状，半成功（我方成功外系统超时但实际成功）如何收敛。每项输出『现状证据（文件:行）+ 缺口 + 风险等级 + 建议设计』。

【第四层·产物生成与挡板联调】
用 orchestrator 角色 fan-out 并行产出，bundle_note 共享已确认/已排除假设避免重复扫：(1) 外系统交互清单（结构化表）：外系统名/调用点 文件:行/方向（同步·异步·文件·回调）/触发场景/报文契约物料路径/鉴权方式/幂等重试超时降级现状/对账方式/本次是否触达。(2) 契约变更影响（结构化表）：外系统/字段/变更类型/兼容性/是否需对端配合/风险等级/对应需求问题 ID。(3) 时序图：用 mermaid sequenceDiagram 生成，泳道含本系统+各外系统，标注同步/异步、幂等键、超时、补偿/冲正分支、对账回路；改动前后两张图叠放，红色高亮本次新增/变更的交互。挡板联调：复用 httpcap——既能在内网真实联调时抓真实外系统报文做契约交叉验证（左侧代码期望 vs 右侧真实报文），也能据扫描出的契约物料自动生成 mock 挡板配置（基于 wsdl/schema 造桩响应），让前后端在外系统未就绪时先联调；mock 桩同时支持注入超时/5xx/慢响应/重复回调以验证重试幂等降级路径。最后把清单+影响+时序图熔成一份详设附录，挂到需求卡 REQ-ID 下（复用旗舰④台账），破坏性契约变更若需通知外系统团队，起草通知邮件走 mail-mcp 发件箱闸门人审后发出。

**信贷特有考量**
- 账务强一致是头号约束：写类外呼（放款/代扣/记账/额度占用释放）必须有幂等键且外系统按键去重，否则重试导致重复记账/重复扣款；半成功（我方超时但核心实际已记账）必须有冲正/红冲与对账兜底，不能简单重试。
- 信贷『外系统调用』常以接口表/中间表+核心跑批捞数实现（非实时 RPC）：扫描必须把对接口表的写入、异库同义词、dblink、存储过程也识别为外系统边界，否则整类核心交互被漏；这类改动的影响要看跑批时序（日终/批量窗口）。
- 对账口径是隐性契约：代扣结果、放款结果、核心记账普遍 T+1 文件对账，本次若动了金额精度/流水号/状态码映射会让对账对不平；需求阶段必须显性问『本次改动是否影响对账文件字段与勾对口径、长短款/差错处理是否要改』。
- 报文细节是联调杀手：金额单位（分/元）、定长右对齐补零、字符集 GBK vs UTF-8、日期格式、码值映射（授信品种/担保方式/五级分类码）跨系统不一致；金额与码值字段任何变更都要专项标红。
- 鉴权与通道：渠道号/机构号/操作员/MAC 验签、证书密钥、ESB 路由头、双向 TLS——动一个就联调全废且难自查，必须单列鉴权差异并升级为人工闸口。
- 外系统差异化语义：征信（人行/百行）有查询授权与留痕、查询频次与用途合规、异步报文回盘；反洗钱名单筛查命中后是阻断还是转人工的业务流转；决策/规则引擎入参变更会静默改变审批结论，需关注规则版本与灰度。
- 监管报送（EAST/1104）是特殊外系统：本次字段改动若涉及报送取数口径，要识别报送影响并提示报送验证，避免监管数据错报。
- 时序与补偿：跨核心+支付的调用顺序（占额度→放款→记账→通知）与失败补偿（冲正/反向交易）、TCC/SAGA/本地消息表/事务消息现状必须摸清，半成功收敛策略要在需求阶段定。

**关键产物 / 数据结构**
- 外系统交互清单 ExternalCallInventory：字段 callPointId / system（征信/核心/支付/反洗钱/影像/签章/决策引擎/监管…）/ direction（sync·async·file·callback）/ trigger（业务场景）/ location（文件:行）/ contractArtifact（wsdl/xsd/proto/定长模板路径）/ auth（鉴权方式）/ idempotency·retry·timeout·degrade（现状）/ reconcile（对账方式）/ touchedByThisChange（bool）
- 契约变更影响表 ContractDelta：system / field / changeType（add·remove·typeChange·requiredFlip·lenPrecision·enumChange·authChange）/ compatibility（backwardCompatible·breaking·needsCounterparty）/ counterpartyAction / riskLevel / linkedQuestionId
- 非功能契约缺口表 NFRGapList：callPointId / dimension（idempotency·retry·timeout·degrade·reconcile·sequencing）/ currentEvidence（文件:行）/ gap / riskLevel / suggestedDesign
- 需求问题清单 OpenQuestions：每条带 questionId / 触发证据（文件:行 或 契约差异）/ 为什么必须回答 / 默认建议 / 是否阻断（账务一致·破坏性契约·对账口径·降级回退 类强制人工闸口）
- 时序图 SequenceDiagram：mermaid sequenceDiagram，改动前/后两张，泳道=本系统+各外系统，标注幂等键/超时/补偿冲正/对账回路，红色高亮本次新增或变更交互
- REQ-ID 详设附录 ExtIntegrationDesignDoc：以上四表+两张时序图熔成 markdown，挂到需求卡 REQ-ID 下，落 memory.md 供跨会话复用

**需要的工具能力**
- 新增 MCP 工具 ext_scan_integrations：复用 scan_impact 的 git-grep 引擎+内置外系统调用点指纹库（客户端类名/配置键/契约物料/MQ topic/接口表写入/回调控制器），多信号打分，输出归类到信贷外系统目录的调用点清单（带文件:行）；与 scan_impact 联动做『本次改动→调用点』可达性收口
- 新增 ext_contract_diff：解析 wsdl/xsd/proto/json-schema/定长模板，git_show 取改动前 vs 工作区，产出报文/字段/鉴权三维差异，金额·日期·字符集·码值·鉴权字段专项标红，输出兼容性判定
- 新增 ext_nfr_audit：对每个触达外系统按幂等/重试/超时/降级/对账/时序六维逐项体检，从代码取现状证据并标缺口与风险
- 新增 ext_sequence_diagram：从调用点+方向+补偿逻辑生成 mermaid sequenceDiagram，输出改动前后对比图，可用内嵌浏览器渲染预览
- 扩展 httpcap：联调时按外系统主机/服务码标签归集真实报文，与 ext_contract_diff 的契约期望左右交叉验证；并据契约物料生成 mock 挡板配置（造桩响应+可注入超时/5xx/慢响应/重复回调）供外系统未就绪时联调
- 扩展 orchestrator：注册新角色 integration-analyst（外系统识别）/ contract-auditor（契约对账）/ resilience-reviewer（幂等重试对账时序），fan-out 并行产出，bundle_note 共享已确认/已排除假设
- 复用 mail-mcp 发件箱闸门：破坏性契约变更起草『对端通知/联调约档』邮件，人审后发出；复用旗舰④ REQ-ID 台账挂载产物

**人审闸口**
- 破坏性契约变更（删字段/改类型/必填收紧/码值含义变/鉴权字段变）必须人工确认是否需对端同步改造与发版窗口，禁止 Agent 自动判定兼容
- 账务一致相关设计（幂等键选取、半成功冲正/红冲策略、重试是否会重复记账）必须人工拍板
- 降级/回退策略（外系统不可用时阻断还是降级、降级后业务状态机与恢复补偿）必须人工确认，涉及风险敞口不能默认
- 对账口径变更（金额/流水号/状态码映射影响对账文件与勾对）必须人工确认并通知对账与运营
- 对外系统团队的通知邮件经发件箱闸门人审后发出，不自动外发
- 反洗钱命中处置（阻断 vs 转人工）与征信查询授权/用途合规等监管语义，必须业务/合规人工确认
- 监管报送（EAST/1104）取数口径若受影响，必须人工确认是否需同步调整报送逻辑

**风险**
- 静态扫描漏报/误报：动态代理、反射、配置中心运行时注入、ESB 服务码间接路由会让 git-grep 找不到调用点；需指纹库持续沉淀到 memory.md，并允许人工补录调用点，关键路径辅以 httpcap 真实联调抓包兜底
- 契约物料缺失或过期：很多外系统契约只有线下 Word/PDF 接口文档、无 wsdl/schema 入库，或仓库里的版本与对端线上不一致，导致差异分析失真；需在产物中显式标注『契约来源与可信度』，并把『契约物料是否权威』列为需求问题
- 代码不出网约束下无法真实外呼验证：契约影响在改码阶段只能静态推断，真伪需联调期 httpcap 抓真实报文确认；Agent 须把静态结论标为『待联调确认』，避免给出过度确定的兼容性判断
- 半成功与账务一致最难自动判定：幂等/冲正逻辑常跨多服务+存储过程+跑批，Agent 易给出看似合理但实际破坏一致性的方案，必须强制人工闸口，Agent 只做证据汇总与问题提出，不替业务定账务策略
- 外系统耦合面大易淹没：不做 scan_impact 可达性收口会把整库外系统全列出来，降低信噪比；必须只呈现本次改动真正触达的子集，其余折叠备查
- 敏感信息泄露：扫描会命中 secret/cert/keystore/密钥配置，产物与日志须脱敏，遵守数据不出网与可审计要求

### 需求问题生成(Requirement Clarification Question Generation)——从"需求文本 ⟷ 代码/数据现状"的差距中,自动产出分类、带优先级、每条挂代码/数据依据的"需求澄清问题清单",回给业务/分析师,作为研发第一道关(需求分析→改动点→需求问题→详细设计)里"问题生成"这一环。

**目标**:在进入详细设计前,把会卡死设计的"未知/歧义/冲突"一次性挖干净并问清楚:输入=需求原文(邮件/工单,带 REQ-ID + messageId)+ 改动点facet产出的落点清单(文件:行);输出=一份按"是否阻塞设计"排序、分类(歧义/缺口/冲突/边界/异常分支/数据口径/合规/性能/兼容)的问题清单,每条挂"因为代码/数据/外系统里看到 X,所以要问 Y"的硬依据,可一键经发件箱闸门回给分析师并闭环登记。核心指标:阻塞类问题召回率高、琐碎问题率低(信噪比),且每条都可追溯到证据。

## 总体管线(建在现有 orchestrator 动态编排 + scan_impact + bundle_note 之上)

把"问题生成"实现为 orchestrator 里一条新工作流 `req-clarify`,五阶段:

**阶段0 · 需求结构化(reqcard 解析)**:输入是 REQ-ID 卡(旗舰④的产物,钉死 messageId 原文)。用 analyst 角色把自然语言需求拆成**可校验的"需求断言"列表**(req-claims):每条 = {动作, 域对象, 字段/属性, 约束/规则, 触发条件, 前后置}。例如"逾期超过 30 天自动降级为关注类"→ claim{动作:状态变更, 对象:五级分类, 规则:逾期天数>30→关注, 触发:?}。**故意把缺失项标 `__MISSING__`**(触发条件没写、口径没写),这些缺口就是问题的种子。

**阶段1 · 现状采集(读代码/读DB/读外系统契约)**:对每条 req-claim,fan-out 多个只读 worker 并行:
- 代码侧:复用改动点facet的落点清单 + `scan_impact`(git grep 符号引用)拿到对应实体类/Mapper/Service/枚举/状态机/校验注解;
- 数据侧:读 DDL/建表语句/`*Mapper.xml`/存储过程,抽出**字段口径**(类型/精度/可空/默认值/枚举码值/唯一约束/分库分表键);
- 外系统侧:grep 出对人行征信/核心系统/反洗钱/决策引擎的调用点,抽出**接口契约**(必填字段、超时、重试、幂等键、返回码语义)。
每个 worker 把结论写进 `bundle_note`(confirmed/excluded/maybe + 证据"已读 LoanMapper.xml:120"),后续 lens 不重查、不互相打架——直接复用现有去重底座。

**阶段2 · 差距对齐(claim ⟷ 现状的九类检测器 lens)**:对每条 req-claim × 现状证据,跑九个**专项检测器**(每个是一个带固定 system prompt 的 lens,产出候选问题,而非自由发挥):
1. **歧义 lens**:需求用词在代码里有 ≥2 种映射(如"客户"在库里有 cust/party/counterparty 三张表)→ 问"指哪个口径"。
2. **缺口 lens**:req-claim 有 `__MISSING__` 字段,或落点代码里有需求没提到的必经分支(如审批流的退回/会签节点)→ 问补全。
3. **冲突 lens**:需求规则与代码现有规则/DB约束/已有需求(memory/历史REQ)矛盾(如需求说额度可为负,DB 有 `CHECK(amount>=0)`)。
4. **边界 lens**:数值/日期/金额的临界(>30 还是 ≥30 天;跨日跑批时点;币种精度;首末期还款)。
5. **异常分支 lens**:外系统失败/超时/降级、并发、空数据、撤销/冲正的处理需求没说。
6. **数据口径 lens**:同名指标不同算法(敞口=授信-用信?含/不含表外?计息基准 ACT/360 vs 365;时点 vs 时段余额)。
7. **合规 lens**:涉及征信/反洗钱/客户信息/留痕,需求没说**授权依据/脱敏/审计字段/报送口径(EAST/1104)**。
8. **性能 lens**:落点命中大表/跑批/存储过程,需求量级(并发/数据量/时延)没给。
9. **兼容 lens**:改动点facet标了下游受影响接口/历史数据,需求没说**老数据迁移/灰度/前后端契约同步**。

每个 lens 产出候选问题对象(见 artifacts 的 ClarifyQuestion),**必须带 evidence ref**(文件:行 / DDL / 接口签名 / 历史 REQ-ID),无证据的问题直接丢弃——这是"避免琐碎"的硬闸。

**阶段3 · 评分、去重、排序**:
- **去重/聚类**:同一证据或同一字段引发的问题合并;用 `bundle_note` 的 key(如 `ambiguity:customer_id`)做幂等。
- **阻塞度评分 blockingScore(0-3)**:0=不影响设计(可备注), 1=影响实现细节, 2=影响接口/数据模型设计, 3=不答就无法动笔(如核心口径未定、合规授权未明、与DB约束直接冲突)。评分规则化(不是纯模型打分):落点是数据模型/外系统契约/状态机/账务一致性 → 至少 2;合规/冲突类 → 至少 2,直接矛盾 → 3。
- **价值过滤**:剔除"代码里已能确定答案"的伪问题(若 confirmed 证据已给出唯一答案,降级为"现状告知"而非提问);剔除可由默认约定回答的琐碎项。
- 排序键:blockingScore desc → 类别权重(合规/冲突/数据口径靠前)→ 受影响落点数 desc。

**阶段4 · 成稿 + 回问 + 闭环**:
- 渲染成**分类分级问题清单**(Markdown + 结构化 JSON 双份),每条形如:`[P0·数据口径] 敞口口径含不含表外?——因 ExposureCalc.java:88 现用"授信-用信"未含表外承诺,而需求第3条提"全口径敞口"(REQ-2026-117 邮件原文第2段)`。
- 经 **mail-mcp `mail_reply`** 钉着原 messageId 回给分析师(走发件箱 15s 延迟闸门,人可改可撤);只把 **P2/P3 阻塞项**默认放进正文必答区,P0/P1 备注区,避免一次轰炸 40 个琐碎问题。
- 回执登记进 REQ-ID 台账(旗舰④)+ `todo_add`(关联 mailMsgId)形成"待澄清"闭环;分析师回信后,工作流增量重跑只对"已回答"的 claim 收敛、对新引入的矛盾再追问。

## 关键策略点
- **检测器化而非自由提问**:九类各有独立 prompt + 触发条件 + 证据要求,模型只在"已定位证据"上生成问题,从源头压琐碎。
- **规则 + 模型混合评分**:阻塞度由可解释规则托底(落点类型/类别),模型只做语义判重和措辞,避免"全是 P0"或"全是 P3"。
- **增量与收敛**:支持多轮——分析师答一轮,只对变化面重算,问题清单单调收敛,不重复问已答项(memory.md + bundle_note 记住已澄清结论)。

**信贷特有考量**
- 数据口径是信贷澄清的头号雷区,必须内建口径词典:敞口(含不含表外承诺/或有)、余额(时点 vs 平均 vs 日终)、利率(名义/实际/年化/APR;ACT360 vs ACT365 vs 30/360)、逾期天数(自然日 vs 工作日;宽限期算不算)、五级/十级分类的迁移规则、计提基数、本金/利息/费用的还款冲销顺序。同名指标在不同子系统算法不同→必问'按哪个口径'。
- 审批工作流的隐藏分支:需求常只写'通过'路径,代码里却有退回/补件/并签/会签/转授权/超时自动流转/撤回。检测器要专门对状态机(Activiti/自研流程引擎的节点定义、状态枚举表)做缺口扫描,问'退回到哪一步''会签几人通过''审批中能否撤回'。
- 账务强一致与冲正:放款/还款/计息一旦涉及总账,必问'失败如何冲正''是否幂等''跨日跑批时点''红蓝字处理''与核心系统对账不平怎么办'——这些是异常分支 lens 的信贷特化,代码里看到调用核心记账接口就触发。
- 外系统调用的合规与失败语义:人行/百行征信要问'查询授权书是否在库/有效期''留痕字段';反洗钱/制裁名单筛查命中要问'是阻断放款还是转人工';决策引擎/规则引擎要问'规则版本/生效时点/灰度';这些调用点 grep 到就触发合规 lens + 异常分支 lens。
- 监管报送口径:涉及客户/合同/五级分类/逾期字段的改动,合规 lens 自动追问'是否影响 EAST/1104 报送字段映射、报送时点、历史回溯',因为报送口径错是监管事件。
- 数据库特征带来的兼容问题:分库分表键变更、大台账表加字段/改类型的历史数据回刷、存储过程里的口径(很多计息逻辑藏在存储过程而非Java)——兼容 lens 要专门读存储过程,问'老数据怎么迁''跑批窗口够不够''分片键能否改'。
- 担保/抵押/质押与额度的联动:授信额度(集团/单一客户/敞口/品种限额)的占用与释放规则极易需求漏写,边界 lens 针对额度表的多级限额结构追问占用时点(受理占用 vs 放款占用)、释放时点(还款 vs 结清)、超限处理。

**关键产物 / 数据结构**
- ReqClaim:需求断言。{claimId, REQ-ID, 动作, 域对象, 字段, 规则/约束, 触发条件, 前后置, sourceQuote(原文片段+messageId+段落定位), missingSlots[](标 __MISSING__ 的槽位)}——把自然语言需求结构化、可逐条对账的中间产物。
- EvidenceNote:现状证据(复用 bundle_note 落盘)。{noteKey, claimId, side(code/db/external), ref(文件:行 / DDL / 存储过程名 / 接口签名), status(confirmed/excluded/maybe), digest(口径/约束/枚举/契约摘要)}——问题的'依据'就指向它,保证每条问题可追溯。
- ClarifyQuestion:澄清问题。{qId, category(歧义/缺口/冲突/边界/异常分支/数据口径/合规/性能/兼容), blockingScore(0-3/P0-P3), question(给分析师的问法), why(因为看到X), evidenceRefs[](指向 EvidenceNote/落点 文件:行), affectedHits(受影响落点数), suggestedDefault(若有合理默认,降低提问成本), dedupKey, status(open/answered/auto-resolved)}——facet 的核心产出对象。
- ClarifyReport:成稿。{REQ-ID, 按类别+优先级分组的 ClarifyQuestion[], 必答区/备注区切分, 渲染后的 Markdown, 回信 messageId, 台账锚点}——发给分析师并登记闭环的最终件。
- 口径词典 GlossaryRules:信贷指标口径/术语映射规则表(敞口/余额/利率/逾期/分类...),驱动数据口径 lens 与歧义 lens,可随项目积累写入 memory.md 复用。

**需要的工具能力**
- 新增 MCP 工具 req_parse_claims:输入 REQ-ID 卡原文,输出结构化 ReqClaim[](含 __MISSING__ 槽位)。底层=analyst 子会话 + 容错JSON解析(复用 orchestrator.extractJson),落盘 userData/reqclaims/<REQ-ID>.json。
- 扩展 scan_impact → scan_impact_for_claim:在现有 git grep 符号引用基础上,额外按 ReqClaim 的域对象/字段反查 实体类/Mapper.xml/枚举/状态机/存储过程引用,产出按 side(code/db/external) 归类的落点。
- 新增 db_lens MCP(只读):解析 DDL/建表语句/Mapper.xml/存储过程,抽字段口径(类型/精度/可空/默认/枚举/唯一/分片键),供数据口径/边界/兼容 lens 用。内网只读、数据不出网。
- 新增 contract_lens 复用(契约对账旗舰②的引擎):对外系统调用点抽接口契约(必填/超时/重试/幂等/返回码),供合规/异常分支 lens 用。
- 新增 question_score 规则引擎:把 blockingScore 评分规则化(落点类型→分值、类别权重),模型只补语义,输出可解释评分。
- 复用 bundle_note/read_notes 做证据去重与多 lens 共享假设;复用 mail_reply(+发件箱闸门)回问;复用 todo_add(关联 mailMsgId)做澄清闭环;复用 memory.md 写入已澄清结论与口径词典,跨会话不重复问。
- orchestrator 注册 role:'clarifier' 角色 + 九类 lens 子提示,fan-out 并行检测;onBeforeBatch 作为人审闸口。

**人审闸口**
- 分析师(业务方):问题清单经发件箱 15s 延迟闸门回给他/她——开发者可在发出前编辑、删减、撤回(复用 outbox),避免把半成品问题轰炸业务。
- 开发者在 onBeforeBatch 检查点审阅'本轮要问哪些 claim/lens',可勾掉误报、补充人工已知答案(标 confirmed 写入 bundle_note,后续不再问)。
- P0/P1(非阻塞)问题默认进'备注区'而非'必答区',是否提问由人决定;只有 P2/P3 阻塞项默认强制进正文必答。
- 数据口径与合规类问题(高风险)在成稿前强制人审一眼:这两类既是召回重点也是误报代价最高处,需人确认口径词典命中是否成立。
- 分析师回信后,工作流不自动改设计,只把'已澄清结论'登记台账并提示开发者确认是否进入详细设计阶段——人在'澄清完成→开工'之间拍板。

**风险**
- 过度提问(信噪比塌)风险最大:若 lens 在无证据/弱证据上生成问题,会淹没真正阻塞项。缓解=证据强制 + 规则化阻塞度 + 必答/备注分区 + 单调收敛去重。需用真实历史需求做召回/精度回归基线。
- 代码现状≠真实业务现状:很多信贷口径藏在存储过程、跑批脚本、配置表甚至运维约定里,git grep 抓不全→漏问或问错。缓解=db_lens 专扫存储过程 + bundle_note 标 maybe 提示'证据不充分,建议人工确认',不把 maybe 当 confirmed。
- 口径词典覆盖不足会漏掉数据口径类(信贷最贵的一类)问题:词典是冷启动短板。缓解=随项目用 memory.md 持续沉淀,首版内置敞口/余额/利率/逾期/分类等高频口径。
- 合规误报/漏报代价不对称:漏问监管报送/征信授权是严重问题,过度问合规则惹烦业务。缓解=合规 lens 走'命中外系统调用点/敏感字段才触发 + 强制人审'的高阈值规则。
- 内网/数据不出网约束:所有解析(DDL/存储过程/邮件原文)必须本地完成,模型只看摘要不外传库结构。缓解=沿用现有'工具一律本地、127.0.0.1、发件箱闸门'底座,db_lens 设计为只读本地解析。
- 需求多版本/变更:分析师边答边改需求,问题清单可能抖动。缓解=按 REQ-ID 版本化 ReqClaim,增量重算只动差异面,问题状态机 open/answered/auto-resolved 保证单调收敛、不回头重问已答项。
- 对编排稳定性的依赖:九 lens fan-out 若某些超时/解析失败,可能丢问题。缓解=复用 orchestrator 的每任务超时/重试/失败兜底,失败的 lens 标记为'未覆盖'明示而非静默丢。

### 详细设计文档生成（信贷详设文档模板结构 + Agent 从前序产物自动组装链路）

**目标**:把前序四阶段（需求分析→改动点定位→需求问题澄清→设计）的离散产物，自动组装成一份信贷领域专用、可追溯（带 REQ-ID 与 文件:行 证据）、可导出（Word/Confluence）、每节事实/建议/待确认分层的详设文档。让评审人 30 分钟看懂改动来由与影响面，让监管/审计可回溯。把完备做实：完备≠篇幅，而是每个待确认项有归属、每个事实有出处、空 section 被自检拦下、每条 REQ-ID 都能正反向追到代码落点与测试用例。

一、模板=结构化 Section schema（不是标题清单）。每节是一个对象，声明：sectionId、源产物（哪个前序阶段/哪个 MCP）、必填字段、证据要求、分层规则、信贷专用 checklist。Agent 逐节按 schema 填充并自检空洞。落盘分两份：ddd.json（机器可读中间稿，每个断言带 claim 对象 text/layer/evidence[]/reqIds[]）与渲染出的 Markdown/Word/Confluence（人读）。机器稿与渲染稿分离，是可导出多目标 / 可增量重生成 / 差异重渲染的根。

二、三层标签贯穿全文（ddd.json 里 claim.layer 枚举 fact|suggest|todo），打在每个段落 / 表格行 / 图节点上：事实=有 文件:行、git_show 提交 hash、httpcap 实抓 payload、邮件 messageId、DB schema 反查等硬证据支撑，渲染成可点跳锚点；建议=architect/coder 角色的方案主张，无外部事实约束，需人工采纳，浅色标注；待确认=缺证据或存歧义，必须挂一个前序需求问题生成阶段的 Q-id，文末自动汇成未决清单。未决项不清零则文档头部水印强制标"草稿/待定 N 项"，不允许标"已定稿"。这是把完备做实的强制机制：完备≠篇幅，而是每个待确认有归属、每个事实有出处、空 section 被自检拦下。

三、16 节模板（每节标源 + 信贷要点）：(1) 背景与目标 — 源需求卡（mail-mcp 原文 + messageId）；区分"业务诉求"与"合规硬约束"（如满足《商业银行授信工作尽职指引》、口径对齐 1104/EAST）。(2) 需求与验收点 — 源 REQ 卡 json_schema 抽取的验收点；每条验收点=一个 REQ-ID，后续所有节用 REQ-ID 回链，文末生成 REQ-ID↔章节↔代码落点追溯矩阵。(3) 术语与缩略语 — 源记忆库术语词典 + 本次新增；信贷强制消歧：授信/敞口/额度/承诺额/或有、五级分类、计提/拨备、本金/利息/罚息/复利、用信/支用、展期/借新还旧；尤其"额度"在集团/客户/产品/单笔四层语义不同，必须分别定义。(4) 业务流程 As-Is/To-Be — mermaid flowchart；把审批工作流的多级审批/退回/并签会签/转授权画全，每节点标权限角色与流转触发条件。(5) 领域模型 — mermaid classDiagram/ER；源后端模型反查（grep 实体 + DB schema）；画清 客户-授信-合同-担保-放款-还款计划-账务 的聚合边界与基数关系（一笔授信 N 合同、一合同 N 放款、一放款一还款计划、担保多对多）。(6) 数据库设计 — 字段级表格 + DDL 草案；信贷最难块（见 creditSpecifics）；区分台账/流水/对账/参数表，标注是否触碰存储过程/跑批/历史表/分库分表键。(7) 接口设计 — 内部 API；源契约对账产物（前端实发 + 后端实收，httpcap 实抓）；逐字段名/类型/必填/枚举/金额精度与币种/敏感级别。(8) 外系统交互 — 信贷第二难块（见 creditSpecifics）；每外系统一张交互卡：协议、报文样例（脱敏）、超时/重试/幂等、失败降级、对账机制。(9) 时序与状态机 — mermaid sequenceDiagram + stateDiagram；放款时序的分布式事务边界（额度占用→核心记账→影像归档→签章）；授信申请/合同/放款/贷后五级分类四张状态机，每条边标触发事件 + 守卫条件 + 反向边（退回/撤销/冲正）。(10) 异常与对账 — 信贷强一致核心；账务两阶段失败补偿、与核心系统日终/日间对账规则、长款短款挂账、冲正/红冲蓝补、跑批失败重跑的幂等与断点。(11) 权限与审计 — 源统一身份角色映射；数据权限（管户/机构/条线隔离）、不相容职责分离（经办/复核/审批）、审计日志字段（谁/何时/改哪条台账/改前改后值）、敏感操作留痕（手工调账/利率特批/风险分类调整）。(12) 非功能 — 跑批时间窗/日终切日点/大表分页分区/账务一致性级别/数据保留与归档/国密合规。(13) 测试要点 — 源 tester 角色 + 验收点；每条验收点映射 正常/边界/异常 用例，关联 repro 断言；信贷专用：跨日担保、闰年计息、提前/逾期还款、部分冲正边界数据。(14) 影响面与回滚 — 源跨双仓影响面雷达（前序阶段产物）+ session diff；受影响 文件:行 清单、数据迁移脚本、回滚草案（含跑批重跑/账务冲正顺序）。(15) 遗留问题 — 汇总全文 [待确认] 生成未决清单（Q-id/描述/阻塞严重度/建议问谁）。(16) 附录 — 证据清单（所有 文件:行 / messageId / commit hash / httpcap 抓包号）、文档版本与 session diff 指针。

四、Agent 自动组装链路（复用 orchestrator.js 的 fan-out + 人闸 onBeforeBatch）：第 1 步 入口=一个 assemble_ddd 工具，拉齐前序四阶段产物（REQ 卡/改动点清单/问题清单/设计纲要）与记忆库术语词典，产出 ddd.skeleton.json（空 section 骨架 + 源指针）。第 2 步 planner 按 section 难度分诊：背景/术语/验收点 走 writer 单 Agent；领域模型/数据库/接口/外系统/状态机/异常对账 走 architect+coder fan-out 并取证据（调 git_blame/git_show/scan_impact/httpcap/grep DB schema）。第 3 步 每 section 产出 section.json（claim[] 带 layer+evidence+reqIds），reducer 并回一份 ddd.json。第 4 步 校验闸（纯 JS 自检，不走 LLM）：事实类 claim 必须有 evidence 否则降为待确认并报 SUSPICIOUS；每个 REQ-ID 至少被一个 section 引用，未覆盖 REQ 起 TODO；每张 mermaid 过语法（本地渲染器 dry-run）；每个 文件:行 反查仍存在（git 变动会让行号漂移，锁 commit hash 而非浮动行号）。第 5 步 人闸：onBeforeBatch 把 [待确认] 清单与争议性 [建议] 推给人拍板，人可编辑/补证/退回每 section。第 6 步 渲染：ddd.json→Markdown（源）→Word（用 docx 技能，三层标签转批注/高亮/水印）与 Confluence storage XHTML（mermaid 转 PlantUML 宏或预渲染 PNG，追溯锚转 Confluence 页内锚）。

五、增量重生成：需求变更时，只重跑受影响 section（按 REQ-ID 反查哪些 section 引用了该 REQ），生成详设 vN vs vN-1 的节级 diff，避免全重写，同时刷新 traceability.matrix 的验收状态。

**信贷特有考量**
- 数据库难点一：信贷表分三类要分开设计。(a) 台账/余额表（授信额度台账、放款本金余额、应收利息）双写/余额重算务必幂等，设计时标出乐观锁版本号字段；(b) 流水表（交易流水/账务流水）只追加不可改，要有冲正标识与原流水号；(c) 参数/码表要标生效日期、是否参与跑批。逐字段强制给：金额用 DECIMAL(18,2) 禁 FLOAT、币种、记账借贷方向、余额快照、交易码、冲正标志、账务日期（≠系统时间，受日终影响）。凡改台账类字段一律标 [事实] 需反查现有 DDL/存储过程，不准拍脑袋。
- 数据库难点二：分库分表与存储过程。信贷台账常按客户号/机构号分片，详设要明确分片键与跨片查询（报表/监管报送需跨片汇总）走广播表还是数仓；计息/计提/逾期置换大量在存储过程/跑批脚本里，Agent 要 grep 出被改逻辑是 Java 还是 SP，[事实] 标明 文件:行 或 存储过程名:行；跑批改动必须出断点重跑/补跑设计，否则上线出错无法恢复。
- 外系统难点：每个外系统调用都有信贷特有强约束，详设逐个说透。人行/百行征信：查询授权与留痕、查询码、二次查询限制、授权书影像、哪些字段不落库；核心银行：账务记账两阶段（预记账/确认）、联动号/流水号幂等、日终后冲正走红冲蓝补、账务日与交易日区分；反洗钱/制裁名单：同步/异步、命中后冻结与人工复核门、误报释放；决策/规则引擎：评分入参出参快照、版本号、试算 vs 正式；电子签章/影像：原件哈希、签署状态回调。每张交互卡强制填：超时多少、重试几次、是否幂等（重复放款/重复记账是事故）、外系统宕机是阻断还是降级放行、如何对账。
- 业务知识难点：额度模型与状态机最易出错。额度要区分 集团统一授信/集团客户/单一客户/产品线/单笔 五层，占用与释放画清额度占用链与重复占用口径；状态机要画出退回、冻结、失效、赎回这些反向/异常边，不能只画 happy path；还款要区分 正常/提前/逾期/展期/借新还旧 对 本金利息罚息复利 的不同冲减顺序；五级分类调整要留痕且可能触发减值准备。这些一律以 [事实] 引现有实现或 [待确认] 挂业务，不准 Agent 自编账务规则。
- 合规与不出网：报文样例、证件号、账号、额度数据在文档里必须脱敏（渲染层加脱敏规则库，如证件号/卡号/金额脱敏）；文档生成全程走内网 opencode serve，证据取自本地 git/DB 只读/httpcap，不调云端；Word/Confluence 导出产物本地生成后才由人走内网上传，不由 Agent 直发。

**关键产物 / 数据结构**
- ddd.skeleton.json：空骨架。字段 docId/reqIds[]/sections[]，每 section={sectionId,title,sourceStage,sourceRefs[],status:empty,creditChecklist[]}。由 assemble_ddd 从 REQ 卡 + 改动点 + 设计纲要拉齐生成。
- ddd.json：中间稿主体（唯一真相源）。每 section.claims[]，claim={id,text,layer:fact|suggest|todo,evidence[],reqIds[],qId?}；evidence[]={kind:fileline|commit|httpcap|mail|dbschema,ref,locator}；section 带 diagrams[]={type:mermaid,src,renderedRef}。一切渲染与校验都从它出。
- traceability.matrix：REQ-ID×章节×代码落点×测试用例×验收状态 的二维表，从 ddd.json 的 reqIds 反建，未覆盖 REQ 高亮为缺口。
- open-issues.list：未决清单。从所有 layer=todo 的 claim 聚合，字段 qId/desc/blockingSeverity/suggestedOwner/relatedReqIds，可一键转邮件问业务。
- evidence.manifest：证据清单附录。集中登记所有 fileline/commit-hash/messageId/httpcap-id，供审计复查与后续反查过期检测。
- render.config：导出配置。target:word|confluence|md、脱敏规则集、三层标签样式映射、mermaid 渲染后端。

**需要的工具能力**
- 新增 assemble_ddd MCP 工具：输入 reqId + 前序四阶段产物路径，输出 ddd.skeleton.json；内部复用 mail-mcp(原文)、session diff、记忆库术语词典。
- 新增 ddd_validate MCP 工具（纯 JS 不走 LLM）：校 事实类 claim 缺证、REQ 未覆盖、mermaid 语法、fileline 是否失效（调 git_show 验证 commit hash 与行），返回 SUSPICIOUS 清单阻断定稿。
- 新增 render_ddd MCP 工具：ddd.json→Markdown/Word(复用 anthropic docx 技能)/Confluence storage XHTML；内置三层标签样式与脱敏规则；mermaid 本地渲染 PNG 或转 PlantUML 宏。
- 扩展 git-mcp：增 git_grep（供 DB schema/存储过程/实体溯源取 文件:行 证据，现只有 blame/diff/show/log/status/branch）；增按 commit hash 锁行的稳定锚点生成。
- 复用 orchestrator.js：planner 按 section 难度分诊 writer 单 Agent vs architect+coder fan-out，收尾 reducer 并 ddd.json；onBeforeBatch 做人闸。
- 扩展 anthropic docx 技能 + 新增内网 Confluence REST 连接器（可注册到 opencode.json）：三层标签→Word 批注/高亮/草稿水印，追溯锚→书签/脚注/页内锚。

**人审闸口**
- 闸1（骨架拍板）：assemble_ddd 出骨架后，人确认 16 节范围与 REQ-ID 映射是否齐全，再启动 fan-out 填充。
- 闸2（未决清单）：onBeforeBatch 把 [待确认] 清单推给人，人逐条判定 转问业务/自行拍板/降为建议；未清零不能出定稿。
- 闸3（争议性设计）：数据库变更/状态机/账务冲正/外系统降级策略这几节的 [建议] 必须人工采纳后才能转 [事实]，不准 Agent 自升级。
- 闸4（合规发出）：Word/Confluence 导出产物走发件箱闸门或人工上传，脱敏检查通过后才允许出网。

**风险**
- 文件:行 证据随 git 提交漂移：必须锁 commit hash 而非浮动行号，ddd_validate 定期反查，否则跳转到错行反误导评审。
- LLM 在账务/计息/状态机上幻觉造规则：强制账务与额度规则只能 [事实] 引现有实现或 [待确认] 挂业务，validate 抓无证据的 fact。
- 未决项被强行闭合以凑完整：未决不清零强制水印草稿 + 阻断定稿，防止为看起来完整而隐藏风险。
- 外系统报文/客户敏感数据泄露：渲染层未脱敏就导出；用脱敏规则库 + 闸4 发出门双保险。
- 增量重生成漏更受影响 section：需求变更时按 REQ-ID 反查引用点，漏某 section 会造成详设与代码不一致；用 traceability.matrix 全量校对兼验收状态。
- 分库分表/存储过程漏分析：计息跑批逻辑多在 SP，Agent 只扫 Java 会漏；强制 grep DDL 与 SP，未扫 SP 的数据库节标 SUSPICIOUS。

### 完备性保证与防幻觉(Completeness Assurance & Anti-Hallucination)——覆盖「需求分析→改动点→需求问题生成→详细设计」全程的可信度底座

**目标**:让信贷管理系统的需求分析与详设产物做到「无源不立、覆盖可证、不确定外显」:每条业务/影响/外系统/账务断言强制挂可机读来源(代码 file:line@sha、邮件 messageId、DB对象、接口契约、监管口径),无源即降级为 UNKNOWN 并自动转成需求澄清问题;系统化证明是否覆盖了全部受影响模块/库表/接口/审批分支/外系统/跑批;对每条断言打置信度;叠加信贷监管合规检查单(审计留痕/数据安全/报送);并在固定闸口由人拍板。最终产出两件可落地资产:完备性 checklist(Coverage Matrix)+ 可信度标注规范(Claim Ledger schema)。严格建在现有底座之上(orchestrator 的 onBeforeBatch 人审闸口、git-mcp 的 blame/log/grep、repro-mcp 的 bundle_note confirmed/excluded/maybe 加强制 evidence 字段先例、mail-mcp messageId 溯源、memory.md 注入、session diff),只做组合与扩展,不另起炉灶。

核心思想:把「断言」做成一等公民——需求分析与详设里所有结论必须以结构化 Claim 落账(Claim Ledger),每条 Claim 必带 source 与 confidence;任何无源断言在汇总(reduce)成文档前被拦截、自动降级为 UNKNOWN 并转成澄清问题。这是把 repro-mcp 现有的 bundle_note(status:confirmed/excluded/maybe,evidence 必填)与 repro_assert(why 必填)模式,从「调试期」通用化上移到「需求/详设期」。

【A. 证据强制 Evidence-or-UNKNOWN】
1) Claim Ledger 是会话级 append-only JSON 台账,挂在 REQ-ID 下(复用旗舰④的 REQ-ID 锚)。每个 worker 子 Agent(analyst/architect 等)产出的每条业务/影响/外系统断言都必须 emit 一条 claim。
2) source 必须是「可机读且可重取」的 ref,而非自然语言转述。允许的 source 类型(直接对应底座工具、可被校验器原样回放复查):code 类=repo:file:行区间@gitsha(经 git-mcp 的 git_grep/git_blame 命中,校验器重跑 grep 比对该行内容哈希,行漂移标 STALE);mail 类=messageId 加引文片段(经 mail-mcp 取原文,引用业务来由即需求出处,禁止脑补需求);db 类=schema.object 加列名(指向真实 DDL/存储过程/视图,见 C 节,需新增 db-catalog-mcp);api 类=openapi operationId 或前端调用点 file:line 对后端 handler file:line(复用旗舰②契约对账);reg 类=EAST/1104/制度条款号(监管口径来源);memory 类=memory.md 节点(跨会话沉淀事实,置信度有天花板)。
3) 无源拦截器(reduce 前置 gate):在 orchestrator 的 buildReducePrompt 之前插入 evidenceGate——扫 ledger,凡 source 缺失/不可解析/回放失败的 claim,一律不得进入设计文档正文,自动改写为 UNKNOWN 并生成一条需求澄清问题(进 Question Queue)。即「宁可问、不可编」。这是把 orchestrator 现有 sanitizePlan(计划净化)思路平移到「结论净化」。
4) 防转述幻觉:校验器对 code/mail 类 source 做「引文核对」——把 claim.text 与 source 实际内容做一致性比对(LLM judge 加关键 token 命中),不一致标 MISCITED 并降级。复用 repro-mcp 的 list_evidence/get_evidence 原样回放证据的离线模式。

【B. 覆盖度清单 Coverage Matrix(完备性的分母)】
完备性本质是「分母是否齐全」。先用工具枚举受影响实体全集(分母),再逐一标是否已分析(分子),空缺即缺口。分母来自机器枚举而非人脑回忆:(a) 代码面=复用旗舰①影响面雷达,对触发符号/字段/接口跨双仓 git_grep 出全部引用点;(b) 库表面=从 db-catalog-mcp 拿到关联台账/流水/账务表、视图、存储过程、触发器、跑批作业全集;(c) 接口面=openapi/proto 加前后端契约对账给出的对内外接口全集;(d) 审批流面=工作流引擎配置里该业务全部审批分支(多级/退回/并签会签)枚举;(e) 外系统面=对该域对象的全部外联调用(征信/核心/支付/反洗钱/影像签章/决策引擎/报送)枚举;(f) 监管面=该改动触及的报送项(EAST/1104)与留痕要求。Coverage Matrix 每行=一个受影响实体,列=实体ref/维度/是否已分析/关联claim_id/置信度/状态(covered/partial/UNCOVERED/NA)。UNCOVERED 行强制变成需求问题或人审项。完备性判据不是「LLM 觉得讲全了」,而是「Coverage Matrix 无 UNCOVERED 行,或每个 UNCOVERED 都已转问题/人审签字豁免」。

【C. 置信度分级 Confidence Tiers】
五级,机器按 source 类型自动定级,人可下调不可随意上调:C5 已验证(有可回放证据,如契约对账通过/录制回放断言通过/grep 命中且哈希一致);C4 直接证据(单一权威 source 直接命中,如 DDL/handler/openapi);C3 推断(多源交叉但无单一铁证,需标推断链);C2 弱信号(仅 memory/历史经验/命名约定);C1 假设(无源,等同 UNKNOWN,必转问题)。规则:涉及账务/计息/监管报送/审批权限的 claim,置信度未达 C4 一律强制进人审闸口;memory 类 source 置信度封顶 C2(防止上次会话的猜测被当事实复用,呼应 memory.md 注入需标记可信度)。

【D. 监管合规检查单 Compliance Checklist】
详设产出前自动跑信贷专用合规闸:①审计留痕——凡改动涉及客户敏感字段/审批动作/额度变更,详设必须含操作日志埋点与变更可回溯方案;②数据安全/数据不出网——校验未引入外网依赖、敏感字段(证件号/账号/征信)有脱敏与权限控制说明;③账务一致性——凡触及账务/总账/计提,必须声明事务边界、对账影响、冲正/红冲路径;④监管报送——若触及 EAST/1104,必须列出受影响报送指标加取数逻辑变更加报送时点影响;⑤变更可回溯——改动挂 REQ-ID 全链路。每项产出 pass/缺口/NA 加证据 ref,缺口全部进人审。

【E. 人审闸口 Human Gates】
复用 orchestrator 的 onBeforeBatch 检查点(已支持 approve/edit/abort),三处硬闸:G1 需求问题闸(澄清问题清单发出/答复回填前人确认,复用邮件发件箱延迟闸门);G2 设计定稿闸(Coverage Matrix 仍有 UNCOVERED、或存在 C1/C2 关键 claim、或合规检查单有未豁免缺口时,阻断出文档,逐条拍板);G3 高危改动闸(账务/计息/审批权限/监管报送类改动无条件进人审,即便机器置信度 C5)。闸口产物全部落痕(谁/何时/批了什么/豁免理由),满足审计。

【F. 与现有编排接线】
每个 worker 的 system prompt 追加「证据强制契约」;新增 verifier 角色(reviewer 之后)专职跑 evidenceGate 加 Coverage 闭合检查加合规闸,产出 completeness report;buildReducePrompt 改为消费 ledger 而非自由文本,只把 verified/direct 的 claim 写进正文,UNKNOWN/缺口写进「待澄清」与「未覆盖」专章。整条链路离线、可审计、可回溯。

**信贷特有考量**
- 业务知识难点:信贷审批流是图不是线——多级审批/退回重提/并签会签/转授权,Coverage Matrix 必须把每条审批分支枚举为分母;漏一条退回分支等于线上审批卡死。改动点分析要识别状态机全部跃迁而非只看主干。
- 账务强一致难点:凡触及计息/计提/放款/还款计划/总账的 claim,必须挂事务边界与冲正(红冲/蓝补)路径证据,不能只说改了金额字段而不交代对账与日终跑批影响;这类 claim 强制 C4 起步且无条件人审(G3)。
- 数据库特征难点:信贷库有台账/流水/对账大表加大量存储过程加分库分表加历史跑批。Coverage 分母必须穿透到存储过程/触发器/跑批作业,而不只是应用层代码——单读代码会漏掉逻辑藏在 PL/SQL 里的影响面;需 db-catalog-mcp 把 DDL/存储过程/作业元数据作为只读证据源。
- 外系统调用难点:征信(人行/百行)、核心账务、支付清算、反洗钱/制裁名单、影像签章、决策引擎、报送(EAST/1104)都是强契约外联。每个外调点是一个 claim 分母行,必须挂接口契约证据并标注同步/异步、超时/补偿、幂等、对账机制;外系统报文字段变更要走契约对账(旗舰②)取真实期望字段而非文档臆测。
- 监管报送难点:EAST/1104 报送指标取数逻辑与底层字段强绑定,任一台账字段语义变更都可能污染报送;合规检查单第④项把受影响报送项做成强制分母,缺口直接进人审,避免改了字段没人想到报送会错。
- 风险分类/贷后:五级分类、风险预警、催收规则常以规则引擎/配置承载,Coverage 必须把规则引擎配置纳入分母,否则代码没改但规则口径变了会被漏掉。
- 合规硬约束:数据不出网→所有证据回放/校验离线进行(复用 repro-mcp 全本机 evidence 目录模式);操作可审计→人审闸口全程落痕;变更可回溯→claim 与缺口全挂 REQ-ID。

**关键产物 / 数据结构**
- Claim Ledger(claim.jsonl,append-only):字段 claim_id、req_id、author_agent、type(business/impact/external/account/regulatory)、text、source(类型加ref加gitsha或messageId)、confidence(C1-C5)、verify_status(verified/direct/inferred/weak/UNKNOWN/STALE/MISCITED)、gate(none/human_required)、ts。
- Coverage Matrix(coverage.json 可渲染表):每行=受影响实体,列=entity_ref/dimension(code/table/api/approval/external/regulatory)/analyzed/claim_ids/confidence/status(covered/partial/UNCOVERED/NA);附完备性判据:无 UNCOVERED 或全部已转问题或豁免。
- Question Queue(questions.json):由无源/UNKNOWN/UNCOVERED 自动熔铸的需求澄清问题,每条挂触发 claim_id或coverage 行加建议收件人(git_blame 出的责任人),经发件箱闸门发出。
- Completeness Report(completeness.md):覆盖率统计、置信度分布、未覆盖清单、合规检查单结果、人审待办——作为详设文档的可信度封面。
- 可信度标注规范文档(confidence-spec.md)加完备性 checklist 模板:两件交付资产,规定 source 语法、五级定级规则、强制人审条件、合规检查项。

**需要的工具能力**
- claim-ledger-mcp(新增):emit_claim/list_claims/verify_claim(回放 source 比对哈希,出 verified/STALE/MISCITED)/to_question(无源转问题)。直接复刻 repro-mcp 的 bundle_note与get_evidence 本机离线、强制 evidence 字段范式。
- db-catalog-mcp(新增,信贷关键):只读拉取 DDL/存储过程/触发器/视图/跑批作业元数据,作为 db 类 source 与 Coverage 库表分母来源;离线连内网元数据库,数据不出网。
- coverage-mcp(新增):enumerate_impact(复用旗舰①影响面雷达 git_grep 引擎产代码分母)加 build_matrix 加 check_closure(出 UNCOVERED 清单)。
- git-mcp(扩展):显式化 git_grep 子命令,供 code 类 source 回放校验与代码分母枚举。
- orchestrator(扩展):worker prompt 注入证据强制契约;新增 verifier 角色;reduce 前置 evidenceGate;buildReducePrompt 改为消费 ledger。
- compliance-checklist-mcp(新增):跑信贷合规五项闸(审计留痕/数据安全/账务一致/监管报送/可回溯),出 pass/缺口/NA 加证据 ref。
- 复用:mail-mcp(messageId 溯源加发件箱闸门发澄清问题)、repro-mcp(离线证据回放范式)、memory.md 注入(标记置信度天花板 C2)、session diff(改动点取证)。

**人审闸口**
- G1 需求问题闸:澄清问题清单发出或答复回填前人确认(复用 orchestrator onBeforeBatch 加邮件发件箱延迟闸门)。
- G2 设计定稿闸:Coverage Matrix 仍有 UNCOVERED、或存在 C1/C2 关键 claim、或合规检查单有未豁免缺口时,阻断出文档,逐条拍板(补证据或签字豁免)。
- G3 高危改动闸:账务/计息/还款计划/审批权限/监管报送类改动无条件进人审,即便机器置信度 C5。
- 所有闸口决策落痕(谁/何时/批了什么/豁免理由),挂 REQ-ID,满足操作可审计与变更可回溯。

**风险**
- 证据回放成本:大改动 claim 数量多,逐条 grep或回放校验可能慢——需批量校验加缓存 gitsha 命中,并对低风险维度抽样。
- db-catalog-mcp 落地依赖内网元数据库可达性与权限;存储过程/跑批逻辑藏得深,枚举分母可能不全(分母本身不完备会假装完备)——需把元数据覆盖率也作为一项已知风险显式标注而非隐藏。
- LLM judge 做引文核对本身可能误判(MISCITED 误报或漏报);关键账务/监管 claim 不能只靠 judge,必须落到人审(G3)。
- 证据强制可能让 Agent 倾向只说有源的、回避难判断的,造成表面完备实则避重就轻——需用 Coverage Matrix 机器枚举分母兜底,强制对每个分母行表态(covered/UNCOVERED/NA),不允许沉默。
- memory 类 source 若被误当高置信复用会传播旧猜测——已用 C2 封顶加人不可随意上调缓解,但需在规范里写死。
- 审批流/规则引擎配置常不在代码仓(在 DB 或独立配置中心),若不接入对应只读源,这部分分母会系统性缺失——属于最易漏的完备性盲区,需优先打通。
