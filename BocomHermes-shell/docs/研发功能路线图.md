# BocomHermes 研发流程功能路线图

> 目标:把这个桌面 AI Agent 放进软件研发团队的 SDLC 流水线里,补足「现在靠老人记忆、口头协调、手工搬运」撑着的流程环节。
> 关注**功能丰富性 / 解决研发流程痛点**,不是安全 / CI / 可观测这类隐性机制(那部分另有清单)。
> 适用场景:有前后端**双仓库**的研发团队(金融 / 信贷内网,代码不出网)。
>
> 状态:**待逐个详聊**。本文件是构思备忘,后续每项展开成详细设计再实现。

## 产品定位(一句话)

一个能**同时读写前后端两个仓库、会用浏览器自验、还会主动巡场**的「全栈数字队友」:它把研发团队里那些靠隐性流程(需求来由、影响面、契约对账、回归验证、事故复盘、变更管控)撑着的环节,变成**有代码证据、可追溯、能闭环**的资产——人只在关键节点拍板。

## 立足点(已有能力,新功能在其上做组合 / 升级)

- 跨前后端双仓库读写代码 + context-matched diff 安全应用
- 角色化多 Agent fan-out + 难度分诊(单 Agent vs 多 Agent 对抗分析)
- 内嵌浏览器(CDP:控制台 / 网络面板 / 元素拾取 / REPL / source-map 还原)
- 复现 →「发给 Agent」→ debug bundle → 多 Agent 分析 → 改码 → 浏览器验证 的调试工作流
- 个人记忆库(跨会话记住偏好 / 项目事实)
- 邮件:晨间摘要、收发 / 回复、搜索 + 跨文件夹、待办闭环、发件箱延迟闸门
- MCP:浏览器自动化、HTTP 抓包、邮件、待办;可一键注册到 opencode.json
- 内网部署(连内网 opencode serve)、待办面板、卡坞 / 历史会话、设置面板

## 🚀 五个旗舰功能(杀手锏)

> 都建立在它独占的能力组合上(双仓并行 serve + 多 Agent + 内嵌浏览器 + 邮件/MCP),单仓工具和普通 Copilot 做不到。

### ① 跨双仓影响面雷达(Blast Radius) · 成本 M · ⭐先做
- **电梯陈述**:选中一个字段 / 方法 / 接口,机器替你读两个仓库,30 秒列出「改这里会牵动谁」——受影响 `文件:行` 可点跳、风险分级、外加 blame 出来的「该问谁」。
- **杀掉的痛点**:这是整张地图复用度最高、最贵的人工动作(需求落地 / 编码 / 评审 / 测试缺口 / 发布影响评估 / 跨仓变更预警,6 个环节都在重造同一个轮子)。核心——同时 grep 前端调用点 + 后端方法引用 + git blame 历史责任人——只有能并行驾驭两个独立 serve 的 Agent 才做得到,单仓工具和人脑都覆盖不全。把「拉群问老人、漏一个就线上事故」变成一张可点清单。
- **基于**:双仓独立 serve(frontend/backend 各 ensureServe)+ git-mcp(blame/log)+ scan_impact(git grep 符号引用)+ 角色化多 Agent fan-out + 富结果 `文件:行` 跳转。

### ② 前后端契约对账单(Contract Reconciliation) · 成本 L
- **电梯陈述**:输入一个接口或页面,机器在前端读出实际发的请求、在后端读出实际期望的字段(或用内嵌浏览器真实复现一次抓 payload),左右逐条对账:字段缺失 / 类型不符 / 必填可选不匹配 / 枚举对不上 / 鉴权头差异,直接定位两端 `文件:行`。
- **杀掉的痛点**:双仓团队最高频也最累的人工核对,日常扯皮的源头(「我按文档传了」「我没收到这个字段」拉群截图对半小时)。成立的唯一前提是同时读两个仓 + 用浏览器抓真实请求做交叉验证——单仓 Linter / 普通 Copilot 永远做不到。把「开会对字段」变成「一份自动对账单」,还能从「出 bug 才查」前移成「联调前主动对账」。
- **基于**:双仓独立 serve + contract lens(参数格式 / 字段缺失 / 类型 / 鉴权头)+ 内嵌浏览器 CDP network 面板 + httpcap-mcp 抓包 + repro debug bundle。

### ③ 录制即用例 → 改完自验闭环(Record→Replay→Assert) · 成本 M
- **电梯陈述**:把一次手动复现录成具名回归用例(带 `no_console` / `has_element` / `no_net` 断言),改完代码自动回放、挂 V8 覆盖率查「改的函数到底跑没跑过」,不过就把报错 + diff 回灌 Agent 自修,直到标「已验证 ✓」。
- **杀掉的痛点**:一条主干同时供养测试欠债 / 生产热修 / 发布冒烟三个环节。录制带内网登录态恢复(cookies/localStorage),全程**数据不出网**、不引入任何外部测试框架,完全跑在内嵌浏览器里(金融内网硬门槛)。V8 precise coverage 对本次 diff 做「盲改告警」——「改了但没被任何用例执行到」的硬证据,没有第二个工具能把改码 / 浏览器执行 / 覆盖率三者绑一起回灌自修。
- **基于**:repro-mcp recording(状态快照恢复)+ replayRec/diffReport + repro_assert 断言 + startCoverage/coverageHits(V8 Profiler)+ context-matched diff 触发验证。

### ④ 需求 ↔ 代码追溯台账(REQ-ID 全链路) · 成本 M
- **电梯陈述**:一封邮件 / 一段文本一键熔成带唯一 REQ-ID 的需求卡(原文 messageId 钉死),此后所有定位、改码会话、session diff、对照表、回归项都挂在这个 ID 下,谁接手点开就懂,不用二次打捞。
- **杀掉的痛点**:最贵的隐性成本——「同一个需求的上下文每个接手的人都要重新打捞一遍」,以及三个月后 git blame 只剩 fix、对不上业务来由。成立是因为这个 Agent 同时握着邮件原文(mail-mcp)、真实改动(session diff / git-mcp)、跨会话记忆(memory.md 注入),用 REQ-ID 串成可检索、可继续、可审计的链路——纯 IDE 工具 / 工单系统各缺一块拼不出。
- **基于**:mail-mcp(原文 + messageId)+ json_schema 抽验收点 + `GET /session/:id/diff` + 卡坞历史(recordHistory/touchHistory)+ memory.md 注入 + todos 闭环。

### ⑤ 事故指纹库 + 防复发哨兵(Incident Memory) · 成本 L
- **电梯陈述**:每次走完验证 PASS 的事故自动蒸馏成指纹(报错栈 / 接口 / 根因 / 修法 diff / 断言);新事故立案时用栈 + 接口签名匹配,命中就在卡顶弹「疑似复发:3 个月前同根因,上次修法见此 diff」,并把当时断言注册成定时静默回放的哨兵,坏了先于用户报障亮卡。
- **杀掉的痛点**:「同一故障换人值班又踩一遍,根因只躺在某人脑子里」和「修完没人保证它不再犯」。靠已沉淀的证据包 / 断言 / 修复 diff 做指纹检索 + 定时无人值守回放——记忆资产 + 浏览器回放 + 调度三者缺一不可,而它们正好都在底座里。把事故知识从「随人蒸发」变成「修一次永久免疫」。
- **基于**:memory.md + repro 证据包/get_evidence + verifyFix(PASS 判定)+ replayRec/repro_assert 回放断言 + trigger.js 定时调度 + 桌面通知。

## 🗺 完整能力地图(7 大主题)

### 主题 1 · 影响面与契约:跨双仓的「改这里会牵动谁」
把散落在 6 个环节里重复造的「血缘 / 影响 / 对账」轮子收成一个跨双仓引擎——选一个符号或接口,机器读两个仓给你带 `文件:行` 的影响清单与契约差异。
- 跨双仓改动点定位:一条需求 → 前后端落点清单(带 `文件:行`)
- 影响面雷达:改这里会牵动谁(跨前后端 Blast Radius)〔旗舰 ①〕
- 改动影响半径 + 浏览器实证:这次改了会不会踩别处
- 前后端契约对齐校验:跨仓查这次改动两端有没有对上
- 前后端联调对账 · 契约一致性体检(Contract Reconciliation)〔旗舰 ②〕
- 改动影响 → 测试缺口提案:每次改完自动算该补哪些测、谁该回归
- 跨仓影响评估:改了接口 → 谁会受影响,自动列下游
- 跨仓变更影响雷达:谁动了我的接口

### 主题 2 · 追溯与考古:每行代码都对得上来由
用 REQ-ID + git blame/log 把「需求↔会话↔diff↔提交↔回归」串成可检索台账,并能回答任意一段遗留代码「当年为什么这么写、能不能安全删」。
- 需求立卡:从邮件 / 工单一键熔铸需求卡(可追溯锚点)
- 改码即留痕:提交前自动生成需求 - 代码对照并回写工单 / 邮件
- 需求追溯台账:按 REQ-ID 串起需求↔会话↔diff↔提交↔回归全链路〔旗舰 ④〕
- 这段代码当年为什么这么写(代码考古)
- 需求变更影响面雷达:旧需求 vs 新需求 diff → 受冲击代码 + 漏改清单

### 主题 3 · 录→放→断言闭环:改完自验 / 回归 / 冒烟 一套底座
一条「录制带登录态 → 回放 → 断言 → V8 覆盖率 → 不过自动回修」的浏览器闭环,同时供养改码自验、夜间回归、热修体检、发布冒烟,数据不出网。
- 改完先验:浏览器跑一遍,确认没改坏
- 录制即用例:把一次手动复现固化成可回归的活体测试〔旗舰 ③〕
- 覆盖率盲区雷达:用回放真覆盖,圈出这次改了但没人测到的代码
- 线上 bug 复现包 → 可复现失败用例:跨双仓根因 + 回归用例
- 测试数据工厂:Agent 读后端模型造一组合规测试数据脚本
- 夜间回归守夜人:定时重放全部回归用例,只在变红时给你一张卡
- 热修闸门:改完先过自动回放,再让你发版
- 发布后冒烟验证:内嵌浏览器跑关键路径 + 断言回放

### 主题 4 · 多 Agent 评审与编排:难的活分角色辩论,带人审闸口落地
难度分诊决定单 Agent 还是 fan-out;评审分 security/perf/biz/arch 多视角对抗,需求落地分 analyst→coder×2→tester→reviewer 带闸口编排,人只在关键节点拍板。
- 字段贯穿改:一句话改一个业务字段,前后端一次性改齐
- 照着这个建一个:从范例蒸馏样板,一键起新模块
- 评审包(Review Pack):一键把 diff 变成可评审的「为什么」
- 团队规范守门人:把靠人盯规范变成自动逐条比对
- 争议改动多 Agent 对抗评审:难的改动不靠一个人拍板
- 评审闭环回执:改完自动核对评审意见是否真落实
- 变更评审材料包:多 Agent 对抗评审,审批前自查
- 目标 → 团队 · 带闸口的需求落地编排

### 主题 5 · 记忆与知识沉淀:tribal knowledge 不再随人走
把答案 / 讲解 / 事故根因 / 规范蒸馏成带 `文件:行` 出处的知识资产,新人自己点着看、老人少答、事故修一次永久免疫,文档跟着真实改动自动更新。
- 上手向导 · 跨仓系统讲解员(Onboarding Codebase Tour)
- 改完即更新 · 文档/Wiki 漂移哨兵(Doc Drift Sentinel)
- 答案沉淀 · 自进化团队知识卡(Self-Distilling Team Q&A)
- 跨团队问答信使 · 把代码答案直接发回提问人
- 事故记忆库:修一次,下次秒认这是老毛病〔旗舰 ⑤〕
- 防复发哨兵:把本次断言变成常驻回归监视〔旗舰 ⑤〕

### 主题 6 · 变更与事故管控:从报障 / 上线到复盘的金融级闭环
双仓 diff 自动起草变更单 / 回滚草案 / 分版发布说明,告警邮件直接落事故卡,闭环结束自动生成复盘时间线——人来回沟通 + 手工搬运 + 凭记忆全部资产化。
- 变更单一键起草(双仓 diff → 变更单 / 影响评估 / 回滚草案)
- 发布说明 + 变更广播(按受众分版,经发件箱闸门发出)
- 出事即查:线上异常 → 关联最近变更 + 一键回滚草案
- 一键报障落卡:从告警邮件直接生成证据包
- 复盘报告自动起草:闭环一结束就生成事故时间线

### 主题 7 · 主动巡场:Agent 当团队成员,主动盯卡点
靠 trigger.js 定时调度 + 邮件/git 双信号,每早一张巡场卡(昨夜谁动了我的目录、谁 @我、待办剩啥),并持续追踪「谁卡我 / 我卡谁」,超时主动亮卡起草催办信。
- 晨间研发巡场卡(My Standup)
- 谁卡我 · 我卡谁(评审 / 依赖闭环追踪)
- 技术债登记官(随手记债 → 自动跟踪 → 定期复盘)

## 先做哪个:① 跨双仓影响面雷达(Blast Radius)

1. **第一天就有体感**:选个字段点一下、30 秒拿到影响清单 + 该问谁,当场省掉「拉群问老人」,不用先攒数据 / 改流程习惯。
2. **复用度最高、风险最低**:双仓 serve、git blame/log、符号引用扫描、富结果跳转**都已部分存在**,组合即用,M 量级。
3. **它是地图的发动机**:后续 5 个旗舰里有一半(需求落点、评审影响半径、测试缺口、发布影响评估、跨仓变更预警)直接复用这一个引擎,先打磨硬,半张地图顺势点亮。

> 第二步候选:契约对账(②,价值高但工程量 L)、录制闭环(③,需先把内网登录态调通)。
