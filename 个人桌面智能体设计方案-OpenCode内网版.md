# 个人桌面智能体设计方案（OpenCode 接入内网大模型）

> 用途：在**内网、代码不出网**的前提下，把 OpenCode + 私有化大模型组装成一个常驻的"个人桌面研发智能体"，服务于信贷系统等内网软件开发工作。
> 关系：本方案是《信贷系统Wiki生成手册-OpenCode执行版.md》的上层延伸——前者解决"让 Agent 看懂这套代码"，本方案解决"把 Agent 变成天天能用的桌面助手"。
> 红线：所有数据（代码/SQL/客户信息）全程留在内网；模型私有化部署；一切结论可追溯。

---

## 0. 关键假设与术语

| 项 | 取值（按你的实际改） |
|---|---|
| 桌面环境 | Windows 11 + 内网 |
| 大模型部署 | 内网 GPU 服务器，暴露 **OpenAI 兼容 API**（`/v1/chat/completions`） |
| 推理框架 | vLLM / Xinference / Ollama / 内部模型网关（四选一） |
| 模型 | 代码类优先（如 Qwen2.5-Coder / DeepSeek-Coder / GLM-Coder 等内网已有的） |
| 智能体内核 | **OpenCode**（开源终端 AI 编码 Agent，模型无关） |
| 使用人数 | 单人（个人桌面），但方案保留团队化扩展点 |

**为什么是 OpenCode 当内核**：模型无关（任意 OpenAI 兼容端点）、本地优先、支持 `AGENTS.md` 上下文（你已经在用）、支持自定义 Agent/命令/MCP、有 TUI 也有 headless/server 两种驱动方式——既能人机对话，也能脚本批跑。

## 1. 总体架构（四层）

```
┌─────────────────────────────────────────────────────────────┐
│  L4 交互层  桌面入口：TUI / 托盘GUI / 全局热键 / 右键菜单        │
├─────────────────────────────────────────────────────────────┤
│  L3 能力层  自定义 Agent（评审/重构/测试/SQL分析/需求定位）      │
│            + 自定义命令(/命令) + Skills + RAG 知识检索          │
├─────────────────────────────────────────────────────────────┤
│  L2 内核层  OpenCode（会话编排 / 工具调用 / 文件读写 / 权限）    │
│            ── opencode.json 把 provider 指向内网端点 ──         │
├─────────────────────────────────────────────────────────────┤
│  L1 模型层  内网私有化大模型（vLLM/Xinference/Ollama/网关）      │
│            OpenAI 兼容 API，全程内网，数据不出网                 │
└─────────────────────────────────────────────────────────────┘
       横切：安全合规（权限/审计/脱敏/沙箱） · 离线运维（内网源/镜像）
```

核心思想：**模型层私有化、内核层标准化、能力层定制化、交互层轻量化**。L1/L2 用现成件，你的工程投入主要在 L3（贴合信贷研发的能力）和 L4（顺手的桌面入口）。

## 2. 形态设计方案（一系列方案 + 推荐）

你要"一系列设计方案"，这里给 4 种由轻到重的形态，可按阶段演进。

### 方案 A —— 纯 OpenCode TUI（最轻，今天就能用）
- 做法：装好 OpenCode，写好 `opencode.json` 指向内网模型，配齐 `AGENTS.md`，直接用终端 TUI 对话。
- 优点：零开发、即刻可用、最稳。
- 缺点：只有终端入口；批量/定时任务要手动敲。
- 适用：**第一阶段先落地，验证模型与权限。**

### 方案 B —— OpenCode 内核 + 轻桌面壳（推荐主线）★
- 做法：`opencode serve` 起本地 HTTP 服务，外面套一个**轻量桌面壳**（系统托盘程序 / 全局热键唤起的小窗 / 资源管理器右键"问问 AI"）。壳只做三件事：选目录、起会话、贴结果。
- 优点：保留 OpenCode 全部能力，又有顺手的桌面入口；壳很薄、好维护。
- 缺点：要写一点点壳代码（托盘 + 调本地 API）。
- 适用：**日常主力形态。**

### 方案 C —— OpenCode + 自动化编排（批处理/流水线）
- 做法：用 `opencode run "..."`（headless）把重复任务脚本化：生成 Wiki、批量补单测、批量写 Mapper 注释、PR 前自检。配 Windows 计划任务或手动触发。
- 优点：把"重复脑力活"变成一键/定时；与方案 B 并存。
- 适用：**Wiki 生成、文档刷新、夜间批跑这类活，是方案 B 的补充。**

### 方案 D —— 完全自研 Agent（不推荐）
- 做法：直接用模型 SDK 从零写 Agent，OpenCode 只当其中一个工具。
- 缺点：重复造轮子，工具调用/权限/上下文管理都要自己扛，维护成本高。
- 仅当：OpenCode 满足不了某个强定制诉求时，才局部自研。

> **推荐路线**：A（验证）→ B（主力）+ C（批处理补充）。D 仅按需。

## 3. OpenCode 接入内网大模型（落地细节）

> 注：OpenCode 迭代较快，以下配置为**结构示例**，字段以你本机 `opencode` 版本的实际 schema 为准（先 `opencode --version`，对照官方/内置 schema 校验）。

### 3.1 自定义 Provider 指向内网端点
在项目根或 `~/.config/opencode/opencode.json`：

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "intranet": {
      "npm": "@ai-sdk/openai-compatible",   // OpenAI 兼容适配器
      "name": "内网大模型",
      "options": {
        "baseURL": "http://10.x.x.x:8000/v1", // vLLM/Xinference/网关地址
        "apiKey": "INTRANET_TOKEN"            // 没有就给占位串，按网关要求
      },
      "models": {
        "qwen2.5-coder-32b": { "name": "Qwen2.5-Coder-32B" },
        "deepseek-coder-v2": { "name": "DeepSeek-Coder-V2" }
      }
    }
  },
  "model": "intranet/qwen2.5-coder-32b"
}
```

要点：
- **关键是 `baseURL` 指向内网 + 模型名对齐部署时的 `served-model-name`**。
- 多模型可并列：日常用代码模型、长上下文任务切大窗口模型。
- API Key 走环境变量更安全：`"apiKey": "{env:INTRANET_TOKEN}"`，避免明文进仓库。

### 3.2 不同推理框架的对接差异
| 框架 | baseURL 形态 | 备注 |
|---|---|---|
| vLLM | `http://host:8000/v1` | 启动时 `--served-model-name` 要和 config 里模型名一致 |
| Xinference | `http://host:9997/v1` | 兼容 OpenAI；注意模型 UID |
| Ollama | `http://host:11434/v1` | 自带 OpenAI 兼容层；适合小模型本机跑 |
| 内部模型网关 | 网关给的地址 | 多带鉴权头，按网关文档配 `headers` |

### 3.3 内网离线安装（最容易踩坑的一步）
内网装不了公网 npm，必须提前准备：
1. **Node.js 运行时**：离线安装包（OpenCode 依赖 Node）。
2. **OpenCode 本体**：优先用预编译二进制；或在能上网的机器 `npm pack` 后拷进内网装。
3. **Provider 适配包** `@ai-sdk/openai-compatible`：在内网 npm 私服（Nexus/Verdaccio）做镜像，或预先打包随 OpenCode 一起拷入。
4. 配一个**内网 npm registry** 一劳永逸：`npm config set registry http://nexus.intranet/repository/npm/`。

> 建议：把"Node + OpenCode + 适配包 + opencode.json 模板"打成一个**内网安装包/一键脚本**，新机器开箱即用。

### 3.4 连通性自检清单
- [ ] 内网机能 `curl http://10.x.x.x:8000/v1/models` 拿到模型列表。
- [ ] `opencode run "用一句话自我介绍"` 能正常返回（验证 provider 通）。
- [ ] 长上下文任务不报超长（确认模型上下文窗口够读你的大文件）。
- [ ] 中文输出正常、代码块不乱码。

## 4. 桌面智能体能力描述（这台"助手"能干什么）

按"日常研发动作"组织，每条标注**默认权限**（读 = 只读分析；写 = 会改文件，需批准；执行 = 会跑命令，需批准）。

### 4.1 代码理解与问答（读）
- 整库/模块问答："授信审批的入口在哪""改这个枚举会牵连哪些表"。
- 调用链/依赖梳理：谁调它、它调谁、影响面评估。
- 直接复用你已建的 `glossary.md` / `capability-map.md`，把业务黑话翻译成代码符号再定位。

### 4.2 代码生成与补全（写）
- 按需求/接口契约生成 Service、Controller、DTO、Mapper 骨架。
- 遵循仓库既有风格（从 `AGENTS.md`、邻近代码学约定）。

### 4.3 修改与重构（写）
- 跨文件重构、改名、抽公共方法、统一异常处理。
- 状态机/枚举调整时，联动提醒同步 `capability-map`、`glossary`。

### 4.4 Bug 定位与修复（读 → 写）
- 给报错栈/现象，定位可疑代码段，给出修复 diff 与原因。
- 区分"症状/根因"，不确定的标【存疑】而非硬猜。

### 4.5 单元测试生成（写 + 执行）
- 为指定类/方法补 JUnit + Mock；可跑测试看结果。
- 针对 MyBatis 动态 SQL 分支补边界用例。

### 4.6 代码评审（读）
- PR/diff 评审：正确性、空指针、SQL 注入（`${}` 拼接）、事务边界、金额单位（你们"金额以分计"这类坑）。
- 输出"必改/建议/可忽略"三档。

### 4.7 SQL 与 Mapper 分析（读）
- 把 `<if>/<choose>/<foreach>` 动态分支翻译成业务含义（延续 sql-logic 手册）。
- 报表口径、批处理顺序依赖、SQL 注入点排查。

### 4.8 需求 → 改动点定位（读）★ 信贷研发高频
- 输入一条需求，按 `glossary → capability-map → modules` 链路产出：改哪些文件/类/方法 + 受影响表/接口/报表 + 需回归项。
- 这是你 Wiki 体系的"消费端"，把前期文档投入变现。

### 4.9 文档与注释（写 + 批处理）
- 生成/刷新模块文档、`AGENTS.md`、接口清单、ER 图（Mermaid）。
- 批量补关键方法注释；Wiki 全量再生（走方案 C）。

### 4.10 工作流/审批流分析（读）
- 逆向自研工作流引擎、抽流程模板、回答"怎么改一条审批流"（延续 workflow-templates 手册）。

### 4.11 终端与 Git 辅助（执行）
- 生成规范 commit message、解释 diff、协助分支操作。
- 跑构建/测试命令并解读输出。**默认需批准，禁跑破坏性命令。**

### 4.12 知识检索 / RAG（读）
- 对内网文档（设计文档、规范、历史工单）做语义检索，回答"我们的规范是怎么定的"。
- 可作为 MCP/工具挂入（见 §5.3）。

> **能力边界声明（务必写进 AGENTS.md）**：默认只读分析；写文件、跑命令必须经你确认；绝不联网外发；拿不准就标【存疑】不编造。

## 5. 能力层定制（把通用 Agent 调成"信贷研发专用"）

### 5.1 自定义 Agent（子智能体）
在 `.opencode/agent/` 放角色化 Agent（markdown 定义 + 各自系统提示 + 工具白名单）：
- `reviewer` —— 代码评审专用，只读，按你们规约清单输出。
- `sql-analyst` —— 专啃 Mapper / 报表口径。
- `req-locator` —— 需求落点定位，强制走 glossary→capability-map 链路。
- `test-writer` —— 补单测，可执行测试。
不同 Agent 给不同权限，**最小授权**：评审/分析类一律只读。

### 5.2 自定义命令（/命令，把套路固化）
在 `.opencode/command/` 放常用指令模板，敲 `/` 即用：
- `/review-pr` —— 评审当前 diff。
- `/locate <需求>` —— 需求改动点定位。
- `/gen-test <类名>` —— 补单测。
- `/wiki-refresh <模块>` —— 刷新模块文档。
- `/sql-explain <Mapper>` —— 翻译动态 SQL 分支。

### 5.3 MCP 接入内网系统（让 Agent 够得着工具）
通过 MCP 把 Agent 连到内网工具（全程内网）：
- 内网 GitLab / SVN —— 取 diff、提交、读历史。
- 内网 Jira / 禅道 —— 读工单作为需求输入。
- 数据库（只读账号）—— 查表结构、字典、样例数据辅助理解。
- Confluence / 内部 Wiki —— RAG 检索的来源。
> 数据库 MCP **强烈建议只读账号**，且不查真实客户敏感数据。

### 5.4 RAG 知识库（可选增强）
内网向量库（如内网部署的 Milvus/PGVector）+ 内网 Embedding 模型，把规范/设计文档/历史工单向量化，作为检索工具挂给 Agent。起步阶段可先不做，用 `AGENTS.md` + docs 体系顶着。

## 6. 安全与合规（内网研发红线）

这是银行内网场景的重中之重，建议逐条落实并写入团队规范。

| 维度 | 措施 |
|---|---|
| 数据不出网 | 模型私有化 + provider 仅指内网地址；**禁用任何公网 provider/联网工具**；机器层面限制出网。 |
| 最小授权 | 默认只读；写文件/跑命令走 OpenCode 权限确认；危险命令（rm/格式化/drop）黑名单。 |
| 敏感信息 | 不让 Agent 处理真实客户数据；样例数据脱敏；密钥走环境变量不进仓库/不进对话。 |
| 沙箱执行 | 命令在受限目录执行；构建/测试用专用账号；避免直连生产。 |
| 审计 | 开启会话/操作日志留存，可追溯"谁让 Agent 改了什么"。 |
| 结论可信 | 强制"代码即事实来源 + 带出处 + 不确定标【存疑】"（沿用你现有原则）。 |
| 供应链 | OpenCode 与依赖走内网私服，版本固定，定期人工审更新。 |

## 7. 与现有信贷 Wiki 体系的协同

你已经投入做了 `glossary / capability-map / data-model / sql-logic / workflow-templates / modules`，桌面智能体是它们的**消费端与维护端**：

```
   需求/问题  ──►  桌面智能体(L3 Agent)
                      │ 读
                      ▼
        glossary ─► capability-map ─► modules ─► 代码
        (翻译黑话)   (定位入口)      (看流程)    (精确改动)
                      │ 写（批处理）
                      ▼
              刷新 Wiki / 补注释 / 出影响面清单
```

- **消费**：每次"需求定位/评审/答疑"都让 Agent 先查这三份索引，再下代码 grep。
- **维护**：代码变更后用方案 C 批跑，保持 Wiki 与代码同步，避免文档腐化。
- **闭环**：把"§4.8 需求定位"固化成 `/locate` 命令，让你的文档投入持续变现。

## 8. 落地路线图（分阶段、可中断）

| 阶段 | 目标 | 关键动作 | 产出 |
|---|---|---|---|
| **P0 通路** | 模型连通 | 内网离线装 OpenCode；配 provider 指向内网模型；过 §3.4 自检 | 能对话的 TUI（方案 A） |
| **P1 上下文** | 看懂这套代码 | 落地 `AGENTS.md` + docs（已有手册）；验证需求定位准确率 | 可用的 L3 知识底座 |
| **P2 定制** | 调成专用助手 | 写自定义 Agent + `/命令`；接内网 GitLab/Jira MCP | reviewer/req-locator 等可用 |
| **P3 桌面壳** | 顺手入口 | `opencode serve` + 托盘/热键轻壳（方案 B） | 日常主力桌面形态 |
| **P4 批处理** | 解放重复活 | 把 Wiki 生成/补测/刷文档脚本化（方案 C） | 一键/定时流水线 |
| **P5 增强** | 锦上添花 | 内网 RAG 知识库；评审/定位准确率度量与调优 | 知识检索 + 质量看板 |

> 每阶段独立可用、可停。**P0–P1 是地基，先把"连通 + 看懂"做扎实，后面才稳。**

## 9. 选型决策点（需要你拍板的几处）

| 决策点 | 选项 | 建议 |
|---|---|---|
| 桌面形态 | A纯TUI / B轻壳 / C批处理 / D自研 | **A 起步 → B 主力 + C 补充** |
| 内网模型 | Qwen-Coder / DeepSeek-Coder / GLM / 其它内网已有 | 选**代码能力强 + 上下文窗口大**的；可多挂并存 |
| 推理框架 | vLLM / Xinference / Ollama / 网关 | 已有就复用；新建优先 vLLM（吞吐好） |
| 壳技术 | 系统托盘脚本 / Tauri / Electron | 个人用先**最薄**（托盘+热键调本地API） |
| RAG | 现在做 / 后做 | **后做**，先用 AGENTS.md+docs 顶 |
| 数据库接入 | 只读 / 不接 | 接就**只读账号 + 脱敏**，否则不接 |

## 10. 风险与对策

| 风险 | 对策 |
|---|---|
| 内网装不上（npm/依赖缺失） | 提前做内网私服 + 一键离线安装包（§3.3） |
| 模型上下文窗口不够读大文件 | 分模块/分阶段喂；挑大窗口模型；用 docs 索引压缩上下文 |
| 模型代码能力弱、易编造 | 强制带出处 + 标【存疑】；只读为主；人工把关写操作 |
| 文档与代码不同步 | P4 定时批跑刷新；变更后触发 `/wiki-refresh` |
| 误删/误改/跑坏命令 | 默认只读 + 写/执行需确认 + 危险命令黑名单 + 专用账号沙箱 |
| 敏感数据泄露 | 不接真实客户数据；密钥走环境变量；全程内网无出网 |
| OpenCode 升级破坏配置 | 版本固定，升级前在测试机验证 config schema |

## 附：一句话总结

> 用 **OpenCode 当内核**、**内网私有化模型当大脑**、**你已有的 Wiki 体系当记忆**、**轻桌面壳当入口**，拼出一个数据不出网、专懂信贷代码的个人研发智能体。先打通"连通 + 看懂"（P0–P1），再做定制与入口（P2–P3），最后用批处理解放重复活（P4）。
